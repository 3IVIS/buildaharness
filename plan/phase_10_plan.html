What Phase 10 delivers
Phases 0–9 built all of the harness logic in the adapter layer. Phase 10 brings that logic to the surface: every major harness state structure gets a dedicated canvas node type, making it possible to design, inspect, and debug harness flows visually. This is a pure UI/schema layer — no new harness logic is introduced. All data displayed comes from live reads of the HarnessRunState via the GET /runs/{id}/harness-state endpoint established in P0.6.

Foundation nodes (P10.1, P10.7) — The world model node and evidence store node are the most information-dense displays. The world model node shows generation_id, belief and observation counts, staleness ratio, and expands to show individual beliefs with confidence and derived_from[] chains. The evidence store node shows reliability class counts (H/M/L), per-tool envelopes, and tool availability status.

Reasoning and control nodes (P10.2, P10.3) — The hypothesis set node colour-codes diversity_score red when below 0.7 (triggering the diversity threshold from P1.7). The control state node shows the NORMAL/CAUTIOUS/BLOCKED risk state with colour coding and displays a staleness warning when the node's generation_id is behind the world model's.

Planning, verification, and recovery nodes (P10.4, P10.5, P10.6) — Task graph shows the 6-state badge per task with depends_on arrows and a completion progress bar. The verification gate lets operators select which of the 9 layers to enable; unavailable layers are shown as SKIPPED with an explanation sourced from the tool_availability_manifest. The recovery node shows the current strategy, next in order, and the matched pattern confidence labelled as advisory.

Experience store and diagnostic dashboard (P10.8, P10.9) — The optional experience store node shows warm_start status, run count, and a (strategy, failure_class) weights heatmap; it shows "unavailable" gracefully when the DB is unreachable. The diagnostic health dashboard is a canvas side panel (not a node) showing all 10 sub-dimensions as [0,1] bar charts. The dep_class_gap_annotation is rendered as a text note, not a chart bar — as required by INV-07.

Reviewer pass node (P10.10) — Displays the three-lens status (PASS/FAIL/PENDING), adversarial prior beliefs with their causal proximity scores, and the list of tasks reopened by drain_propagation_queue() with the reason for each re-open.

Key constraints to respect in Phase 10
Canvas nodes are read-only consumers of harness state: No canvas node may write to world_model, control_state, or any other harness state structure. The canvas UI reads state via the GET /runs/{id}/harness-state endpoint and displays it. Mutations happen only through the adapter harness layer. Violating this would break INV-06 (control_state as sole control input).
dep_class_gap_annotation must render as a text note, not a chart bar: The diagnostic health dashboard (P10.9) shows 10 sub-dimensions as bar charts. The dep_class_gap_annotation is explicitly not one of those bars — it must be shown as a text annotation beneath the charts. Adding it as a bar would imply it is a normalised [0,1] numeric input, violating INV-07.
Experience store node must handle unavailable state gracefully: The experience store node (P10.8) must render without error when experience_store.available=False. It shows an "unavailable" indicator and empty heatmap rather than failing. This is a UI enforcement of INV-10.
Pattern confidence on recovery node is advisory — label it clearly: The recovery node (P10.6) displays the failure_mode_library matched pattern confidence. This value must be visually labelled "advisory" and must not be rendered alongside recovery strategy controls in a way that implies it drives the selection. INV-08 applies in the UI layer as much as in the logic layer.

Architecture invariants engaged in Phase 10
Phase 10 introduces no new invariants. It enforces four existing invariants in the UI layer — ensuring that the visual representation of harness state respects the same constraints as the underlying logic.
INV-06 — control_state as sole control input (canvas read-only) P10.3 scope The control state canvas node must never write to control_state. It reads the current value via the harness-state API and displays it. Any action the operator takes in the UI (e.g. the strategy override in P10.6) modifies harness_config in the FlowSpec, not the live control_state structure. The adapter applies those config changes on the next run start. No UI action may mutate a live control_state during a run.
INV-07 — dep_class_gap advisory only (dashboard display) P10.9 scope The diagnostic health dashboard renders 10 sub-dimension bar charts. The dep_class_gap_annotation must be displayed as a text annotation in a clearly labelled "Advisory Annotations" section — never as a bar chart, never as a numeric axis value, and never in the same visual group as the 10 normalised sub-dimensions. Rendering it as a bar would visually imply it is a normalised [0,1] numeric input, which would contradict its advisory-only role.
INV-08 — Failure mode library as advisory (recovery node display) P10.6 scope The recovery node shows the failure_mode_library matched_pattern confidence. This value must be visually labelled "advisory" and must be positioned and styled separately from the strategy selection controls. It must not appear in a way that implies it drives or overrides the strategy order. The operator can read it as context but cannot configure the recovery node to "use pattern confidence as the strategy selector" — that option must not exist in the UI.
INV-10 — experience_store is a no-op when absent (experience store node) P10.8 scope The experience store node must render cleanly and without error when experience_store.available=False. It shows an "UNAVAILABLE" state with an informational message. No missing-data error states, no broken heatmap, no exceptions propagating to the canvas. The UI behaviour mirrors the logic: absent experience store → clean no-op behaviour, not a degraded or broken state.
