Phase 8 — Experience Store & Learning

Adds cross-run learning to the harness. The experience store captures successful decompositions, tool workflows, verification plans, and recovery sequences from prior runs, then seeds the next run via warm_start(). When unavailable the entire system is a no-op — the harness runs correctly with flat priors. When available, strategy selection and task decomposition improve empirically.

What Phase 8 delivers
Phase 8 adds optional cross-run learning to the harness. All prior phases operate with flat, stateless priors — every run starts from scratch with the same fixed recovery strategy order, the same default dep_graph_budget decay rates, and no structural hints about how to decompose the task. Phase 8 changes this by introducing an experience store that persists successful structural patterns and empirical strategy outcomes across runs.

Experience store data model (P8.1) defines the Postgres-backed store and its available property. The available property is the single guard used by every other P8 function — when DB connectivity is absent, all operations are silent no-ops, preserving INV-10. warm_start() (P8.2) is called once at the very start of each harness run. It loads prior strategy weights, failure-class base rates, successful decomposition patterns, tool workflow templates, verification plan templates, and recovery sequence templates from the experience store. Each loaded artefact seeds the corresponding harness structure rather than replacing it — the run can still override anything warm_start provides. update_experience_store() (P8.3) is the write side: called after each successful task completion, it captures the completed tool_workflow, verification_plan, and — if a recovery was needed — the recovery_sequence, then re-normalises the empirical strategy weights. Adaptive strategy upgrade (P8.4) replaces the fixed recovery_strategy_order enum with a softmax policy over (strategy_type, failure_class) pairs when the experience store is available. The fixed enum remains the correct default for new deployments with no experience data.

The critical architectural constraint throughout Phase 8 is INV-10: every code path that touches the experience store must guard with experience_store.available. A run without the experience store must be structurally identical to one with it — no missing keys, no exceptions, no different control paths.

Key constraints to respect in Phase 8
INV-10 must hold unconditionally: Every function that reads from or writes to the experience store — warm_start(), update_experience_store(), softmax_strategy_policy(), and any loader — must guard with if not experience_store.available: return as its first statement. This is not optional safety; it is a hard architectural invariant. Tests must verify that a run with experience_store.available=False produces a structurally identical output to one where the store is present.
warm_start() seeds, it does not replace: Structural patterns loaded from the experience store initialise harness structures with a head start — they do not replace the harness's ability to override them. A decomposition loaded via warm_start() is a starting point for the task_graph, not a locked-in plan. The harness must be able to replan away from any warm-started structure during the run.
Fixed enum remains the correct cold-start default: The adaptive softmax policy (P8.4) applies only when experience data exists for the specific (strategy_type, failure_class) pair. When the store is empty or when the queried pair has no entries, the fixed DIRECT_EDIT → TRACE_EXEC → BROADER_SEARCH → REIMPLEMENT → MINIMAL_FIX → ESCALATE order must be used unchanged. Document this explicitly so new deployments are not misconfigured.

P8.1 Experience store data model Week 1 · ~3 days
Implement the Postgres-backed experience store, its data types, and the available property that guards all downstream usage. The store holds five structural entry types and one indexed table of empirical strategy weights. The available property must return False — never raise — when DB connectivity is absent, satisfying INV-10.
Implementation steps
1
Create Alembic migration in adapter/migrations/versions/. Two tables: experience_entries (id UUID PK, entry_type VARCHAR, failure_class VARCHAR, task_class VARCHAR, payload JSONB, run_id UUID, created_at TIMESTAMP DEFAULT NOW()) and experience_strategy_weights (id UUID PK, strategy_type VARCHAR, failure_class VARCHAR, success_count INT DEFAULT 0, attempt_count INT DEFAULT 0, rate FLOAT DEFAULT 0.5, updated_at TIMESTAMP DEFAULT NOW(), UNIQUE(strategy_type, failure_class)). All columns non-nullable with documented defaults.
2
Create adapter/harness/experience_store.py. Define ExperienceType enum: DECOMPOSITION, TOOL_WORKFLOW, VERIFICATION_PLAN, RECOVERY_SEQUENCE, FAILURE_PATTERN. Define ExperienceEntry dataclass: id: str, entry_type: ExperienceType, failure_class: str | None, task_class: str | None, payload: dict, run_id: str, created_at: datetime. Define StrategyWeightKey as a named tuple: (strategy_type: str, failure_class: str).
3
Define ExperienceStore dataclass with fields: db_session_factory: Callable | None, _cache: dict = field(default_factory=dict). Implement available property: attempts a lightweight SELECT 1 ping via db_session_factory inside a try/except; returns False on any exception, True on success. The ping must complete in under 50ms — use a short connect timeout.
4
Implement query methods. append(entry: ExperienceEntry) → None: inserts into experience_entries; no-op if not available. query_by_type(entry_type: ExperienceType, task_class: str | None = None, limit: int = 10) → list[ExperienceEntry]: returns most recent matching entries ordered by created_at DESC; returns empty list if not available. get_strategy_weights() → dict[StrategyWeightKey, float]: returns all rows from experience_strategy_weights as a dict keyed by StrategyWeightKey; returns empty dict if not available.
5
Wire into HarnessRunState in state_store.py. Replace the existing experience_store_ref: dict field with experience_store: ExperienceStore | None = None. Update load() to construct an ExperienceStore using the same db session factory when the experience_store_ref JSONB column is non-empty, or leave it as None when empty. Update save() to persist a marker (not the store contents — those are in their own tables) so the presence flag survives pause/resume.
Acceptance tests
experience_store.available returns False when the DB session factory raises a connection error — no exception propagates to the caller
Calling append(entry) followed by query_by_type(entry.entry_type) returns the stored entry with identical payload and metadata — full round-trip without data loss
get_strategy_weights() returns a dict keyed by StrategyWeightKey(strategy_type, failure_class) tuples — not plain strings — and values are floats in [0,1]
P8.2 warm_start() — structural reuse Week 2 · ~3 days
Implement warm_start() — the function called once at the very start of each harness run that loads prior empirical data and structural patterns into the current run's state structures. When the experience store is unavailable, warm_start() returns immediately without touching any state. When available, it seeds — but does not lock — the structures that benefit from prior knowledge.
Implementation steps
1
Define WarmStartResult dataclass in experience_store.py: loaded: bool (True if experience store was available), decompositions_seeded: int, tool_workflows_seeded: int, verification_plans_seeded: int, recovery_sequences_seeded: int, strategy_weights_loaded: bool, class_priors_loaded: bool. This is a log record — it does not control execution, but surfaces to Langfuse tracing in P11.
2
Implement load_strategy_priors(experience_store, strategy_state) → None. Calls experience_store.get_strategy_weights(). If non-empty: write the returned dict into strategy_state.prior_strategy_weights. Load failure_mode_library.class_priors from the most recent FAILURE_PATTERN entries (aggregate by failure_class, compute base rates). Both are no-ops if the store returns empty results.
3
Implement load_structural_decompositions(experience_store, task_graph, task_class) → int. Queries query_by_type(DECOMPOSITION, task_class=task_class, limit=3). For the highest-confidence entry: deserialise the payload into Task dataclasses and append as PENDING tasks to the task_graph. Returns the count of tasks seeded (0 if no entries found). The seeded tasks are subject to full task_graph validation — they may be removed or modified during decomposition_gate.
4
Implement load_tool_workflow_seeds(experience_store) and load_verification_plan_seeds(experience_store). Both query their respective entry types and return a list of payload dicts. These are made available to plan_tool_workflow() (P5) and verification_adequacy_critic() (P5) via a seeds parameter — callers use them as starting points rather than building from scratch. Both return empty lists if the store is unavailable.
5
Implement the warm_start(experience_store, strategy_state, failure_diagnostics, task_graph, task_class, dep_graph_budget) → WarmStartResult entry point. First check: if not experience_store.available: return WarmStartResult(loaded=False, ...all zeros...). Then call each loader in sequence: load_strategy_priors(), load_structural_decompositions(), load_tool_workflow_seeds(), load_verification_plan_seeds(). Also update dep_graph_budget.confidence_decay_rate from the median decay rate seen in successful runs. Populate and return WarmStartResult.
Acceptance tests
warm_start() with experience_store.available=False returns a WarmStartResult(loaded=False) immediately — no fields in strategy_state, task_graph, or dep_graph_budget are mutated
After storing a successful decomposition via append(), calling warm_start() on a second run seeds the task_graph with PENDING tasks matching the stored payload — task count and descriptions match the prior decomposition
strategy_state.prior_strategy_weights after warm_start() contains non-uniform rates that differ from the flat 0.5 default — confirming empirical data was loaded rather than defaults returned
P8.3 update_experience_store() on successful run Week 3 · ~2.5 days
Implement the write side of the experience store. After each successful task completion, the harness captures the concrete artefacts that made the run succeed — the tool workflow, the verification plan, and (if a recovery was executed) the recovery sequence — and stores them as reusable experience entries. Strategy weights are re-normalised with each new data point so the adaptive policy (P8.4) reflects the most current empirical rates.
Implementation steps
1
Add a success hook in loop.py. After a task transitions to COMPLETE status: call update_experience_store(completed_task, strategy_state, execution_context, experience_store). The hook fires for every task completion — not just the final task. If experience_store.available is False, the call returns immediately with no side effects.
2
Implement update_experience_store(completed_task, strategy_state, execution_context, experience_store) → None in experience_store.py. First guard: if not experience_store.available: return. Then build an ExperienceEntry for the completed task's decomposition contribution (subset of tasks derived from this task). Use execution_context.task_class as the task_class tag for future retrieval.
3
Capture and store the tool workflow artefact. From execution_context.completed_tool_workflow: serialise the ordered list of (tool_name, call_params_schema, result_schema) tuples to a payload dict. Append as an ExperienceEntry(entry_type=TOOL_WORKFLOW, task_class=..., payload=...). Also store the verification plan from execution_context.verification_plan as a VERIFICATION_PLAN entry.
4
Store the recovery sequence if one was used. Check strategy_state.recovery_was_used. If True: append a RECOVERY_SEQUENCE entry with payload containing the strategy name, failure_class, and the sequence of actions taken. This entry is used by load_verification_plan_seeds() in future warm_starts to prime the strategy ordering for the same failure class.
5
Re-normalise strategy_weights. After storing all artefacts: call upsert_strategy_weight(strategy_type, failure_class, success: bool, db_session). This function: increments the relevant attempt_count; increments success_count if success=True; updates rate = success_count / attempt_count. Use a database UPSERT (INSERT ... ON CONFLICT DO UPDATE) to avoid race conditions on concurrent runs.
Acceptance tests
After a successful run using strategy DIRECT_EDIT on failure_class "type_error": get_strategy_weights()[StrategyWeightKey("DIRECT_EDIT","type_error")] returns a rate closer to 1.0 than the prior 0.5 default
After a successful run that required a TRACE_EXEC recovery: query_by_type(RECOVERY_SEQUENCE) returns an entry with payload containing "TRACE_EXEC" and the correct failure_class — the recovery sequence is stored and retrievable
update_experience_store() with experience_store.available=False returns immediately with no database calls, no mutations to strategy_state or execution_context, and no exceptions raised
P8.4 Adaptive strategy upgrade Week 4 · ~2.5 days
Replace the fixed recovery_strategy_order enum with a softmax-weighted policy when empirical data is available. The policy selects strategy ordering based on historical success rates for the current (strategy_type, failure_class) pair. When no empirical data exists for a pair, or when the experience store is unavailable, the fixed enum order is used unchanged.
Implementation steps
1
Implement softmax_strategy_policy(strategy_weights, failure_class, temperature=1.0) → list[str] in experience_store.py. Filters strategy_weights to entries matching the given failure_class. Applies softmax over the empirical rates: score_i = exp(rate_i / temperature), then normalise to probabilities. Returns strategies sorted by descending softmax score. If no entries match failure_class: returns the fixed default order ["DIRECT_EDIT","TRACE_EXEC","BROADER_SEARCH","REIMPLEMENT","MINIMAL_FIX","ESCALATE"]. Temperature of 1.0 is the default; higher values increase exploration of lower-probability strategies.
2
Implement build_strategy_ordering(failure_class, experience_store) → list[str]. This is the single call site used by the recovery system. Logic: if not experience_store.available → return fixed default order. Else: call experience_store.get_strategy_weights(), pass result to softmax_strategy_policy(weights, failure_class), return the result. Document that callers must use this function — they must not read strategy_state.recovery_strategy_order directly when the experience store is available.
3
Wire into switch_strategy() in recovery.py. In switch_strategy(strategy_state, failure_class, experience_store): replace the direct index advancement into the fixed enum with a call to build_strategy_ordering(failure_class, experience_store). The returned list becomes the effective ordering for this switch. Update strategy_state.current_strategy to the next entry in the returned list after the current strategy.
4
Document temperature parameter and cold-start behaviour. Add a module-level docstring to experience_store.py explaining: (a) temperature default of 1.0 — lower values concentrate ordering around the highest-rate strategy; higher values are more exploratory; (b) cold-start behaviour — when attempt_count == 0 for all strategies in a failure class, the fixed default order is returned (not a random shuffle); (c) new deployments are expected to use the fixed default order for their first N runs. Expose temperature as a configurable field on HarnessRunState defaulting to 1.0.
Acceptance tests
After 5 stored runs where DIRECT_EDIT failed (rate 0.0) and TRACE_EXEC succeeded (rate 1.0) for failure_class "import_error": softmax_strategy_policy(weights, "import_error") returns a list with "TRACE_EXEC" ranked first — ahead of "DIRECT_EDIT"
build_strategy_ordering("any_class", experience_store_with_available=False) returns the fixed default order ["DIRECT_EDIT","TRACE_EXEC","BROADER_SEARCH","REIMPLEMENT","MINIMAL_FIX","ESCALATE"] without database access
Calling build_strategy_ordering(failure_class, experience_store) on run 1 (empty store) and run 10 (10 stored outcomes) returns different orderings for the same failure_class — confirming that empirical data changes the strategy ranking over time

Architecture invariants engaged in Phase 8
INV-10 — experience_store is a no-op when absent P8.1–P8.4 full enforcement Every function introduced in Phase 8 — warm_start(), update_experience_store(), build_strategy_ordering(), and all loader helpers — must begin with if not experience_store.available: return (or equivalent for functions returning a value). The run with experience_store.available=False must produce a structurally identical harness output to a run where the store is present: no missing keys in HarnessRunState, no exceptions, no empty list where a non-empty list is normally expected. This invariant is not advisory — it determines whether Phase 8 is safe to ship.
INV-06 — control_state as sole control input (strategy selection) P8.4 scope The softmax policy (P8.4) produces a reordered list of recovery strategies — it does not write to control_state, block_mask, or escalation_reason directly. Strategy ordering is an advisory input to switch_strategy(); the decision to switch strategy is still driven by cannot_make_progress() and control_state.risk_state. The experience store shapes how strategies are ordered, not whether they are used.
INV-08 — Failure mode library scope (class_priors loading) P8.2 scope warm_start() loads failure-class base rates into failure_mode_library.class_priors. These priors are an advisory input to Tier 4 of resolve_control_state() — they do not write to block_mask or escalation_reason. The same INV-08 constraint that governs the failure mode library's role in Tier 4 applies to the class priors loaded here: advisory normalised input only.
