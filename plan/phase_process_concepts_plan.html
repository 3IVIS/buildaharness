Process Concepts Phase — Static Step Templates for Harness Runs

Introduces process concepts: human-authored, domain-specific task graph templates that seed the harness task graph at run initialisation. A concept is a static JSON file that defines an ordered sequence of typed steps, their dependencies, expected tools, per-step success criteria, and optional strategy hints. The harness uses it as a strong prior on decomposition — the model can still adapt, add, or remove tasks during execution via the existing replanning machinery.

What this phase delivers
Phases 0–10 gave the harness a complete reasoning and control architecture plus a full canvas UI. Every harness run currently starts from a blank task graph — the model decomposes the objective from scratch each time, guided by its world model and hypothesis set.

Process concepts change the starting conditions. A concept is a hand-authored JSON file that describes a canonical, domain-tested sequence of steps for a well-known class of task. Loading a concept at run initialisation pre-populates the task graph with typed, ordered tasks before the model begins work. The model does not have to discover the decomposition — it inherits a proven scaffold and can focus its reasoning budget on the actual work rather than the meta-problem of how to structure it.

Core layer (P-PC.1, P-PC.2, P-PC.3) — The ProcessConcept and ProcessConceptStep dataclasses define the concept data model. The ProcessRegistry maps concept IDs to file paths and handles loading with file-level caching. Four bundled concept files — debug_test_failure, implement_feature, code_review, and refactor_module — are shipped with the harness and auto-registered at startup via scan_directory("concepts/").

Integration layer (P-PC.4, P-PC.5) — The harness initialize() function in loop.py gains an optional process_concept parameter. When provided, seed_task_graph(objective) pre-populates the task graph before decomposition_gate() runs — the gate still validates alignment with the objective and can escalate if it finds a mismatch. The FlowSpec harness_meta block gains an optional process_concept_id field; run_api.py loads the named concept from the registry at run start and passes it to initialize().

Canvas layer (P-PC.6) — A new process_concept canvas node type lets flow authors pick a concept from the registry, preview its steps and dependencies, and observe concept progress (each step showing its live task status) during an active run.

Integration tests (P-PC.7) — End-to-end tests validate the full path from concept file to seeded run. Critical invariant tests confirm that the seeded task graph remains mutable (replanning adds tasks beyond the concept's steps) and that the process_concept=None path is structurally identical to all pre-concept runs — no regressions.

Key design decisions
Concepts are strong priors, not constraints: seed_task_graph() pre-populates the task graph but does not lock it. The existing replanning machinery (diagnose_and_replan(), rebuild_task_graph()) can add, remove, or modify tasks freely during the run. A concept is advice about how to decompose a class of task — it is not a script the model must follow exactly.
decomposition_gate() is not bypassed: After seeding, the full decomposition_gate() validation runs against the seeded graph. If the concept's step structure is misaligned with the objective, the gate returns GATHER_CLARIFICATION or ESCALATE rather than silently proceeding. An invalid concept does not produce a silently wrong run.
experience_store and process_concept are additive, not conflicting: warm_start(experience_store) is called after the concept seeds the task graph. It enriches strategy_state.prior_strategy_weights and loads successful recovery sequences, but it does not replace the seeded task structure. The concept provides the structural scaffold; experience_store provides the calibrated priors on top.
Missing concept_id is a hard error, not a silent fallback: When harness_meta.process_concept_id is set in the FlowSpec but the named concept cannot be found in the registry, the run start fails with a ProcessConceptNotFoundError. A missing concept is a configuration error. Silently falling back to model-driven decomposition would mask a broken deployment and produce unexpectedly unguided runs.
Concept files are immutable during a run: The registry caches loaded concepts by (path, mtime). Once a run starts and initialize() has called seed_task_graph(), changes to the concept file on disk have no effect on the active run. The seeded task graph is the live source of truth from that point forward.

P-PC.1 ProcessConcept and ProcessConceptStep dataclasses Week 1 · ~2.5 days
Creates adapter/harness/process_concept.py with the core data model. ProcessConceptStep maps to a single Task in the task graph. ProcessConcept is the container holding all steps for a given task class. The seed_task_graph() method is the primary integration point — it converts the concept into a fully constructed TaskGraph ready for validate_task_graph().
Implementation steps
1
Create adapter/harness/process_concept.py and define ProcessConceptStep. Fields: id: str (slug format — alphanumeric and underscores only, enforced by regex on construction), description: str, depends_on: list[str] (step IDs within the same concept), risk_level: RiskLevel (imported from risk.py), abstraction_level: str (must be one of "leaf" or "subgoal"), expected_tools: list[str] (advisory; cross-checked against tool_availability_manifest at init but not a run blocker), success_criteria: str, strategy_hint: str | None = None (must be a valid strategy name from the 6-strategy order in recovery.py if set). Implement to_dict() and from_dict(d) on the step.
2
Define ProcessConcept dataclass. Fields: id: str, name: str, description: str, success_criteria: str (top-level; individual steps carry their own per-step criteria), schema_version: str (must equal "1.0" — strict check; any other value raises ProcessConceptValidationError immediately), steps: list[ProcessConceptStep]. Define ProcessConceptValidationError(ValueError) as the typed exception raised for all structural violations.
3
Implement ProcessConcept.validate(). Three checks in order: (1) step ID uniqueness within the concept — duplicate IDs raise immediately; (2) depends_on reference validity — every ID in any step's depends_on[] must exist as a step ID in this concept (forward references are allowed — validation is over the complete step list, not incremental); (3) cycle detection — run a topological sort; any cycle raises ProcessConceptValidationError with the cycle path in the message. validate() is called by both from_dict() and from_file() before returning.
4
Implement ProcessConcept.from_dict(d: dict). Parses the raw JSON dict: checks schema_version first, constructs ProcessConceptStep objects from d["steps"], then calls validate(). If any field is missing or has the wrong type, raise ProcessConceptValidationError with the field path in the message. Never return a partially-constructed concept. Implement a symmetric to_dict() for serialisation to the journal.
5
Implement ProcessConcept.seed_task_graph(objective: str) → TaskGraph. Converts each ProcessConceptStep to a Task (imported from task_graph.py). Task IDs are namespaced as {concept_id}:{step_id} — raw step IDs never appear in the output task graph. depends_on[] entries are translated to the same namespaced format. risk_level and abstraction_level are copied directly. If strategy_hint is set, it is written to task.assigned_strategy. All tasks are initialised with status PENDING. Returns a fully constructed TaskGraph that can be passed directly to validate_task_graph().
Acceptance tests
from_dict() raises ProcessConceptValidationError when a step's depends_on references a step ID that does not exist in the concept's step list — error message includes the missing ID
validate() detects a two-step cycle (step A depends on step B, step B depends on step A) and raises ProcessConceptValidationError with the cycle path in the message
seed_task_graph() produces a TaskGraph where all task IDs are namespaced as {concept_id}:{step_id} — no raw step IDs appear; confirmed by asserting no task ID equals any step ID from the source concept
seed_task_graph() output passes validate_task_graph() without error for a well-formed 4-step concept with a linear dependency chain
P-PC.2 ProcessRegistry — file-backed concept registry Week 1 · ~1.5 days
Creates adapter/harness/process_registry.py. The registry maps concept IDs to file paths and handles loading with file-level caching keyed by (path, mtime). A module-level default registry instance is auto-populated from the concepts/ directory at startup. Thread-safe for concurrent load() calls from parallel run starts.
Implementation steps
1
Create adapter/harness/process_registry.py and define ProcessRegistry. Internal state: _registry: dict[str, Path] mapping concept IDs to absolute file paths, _cache: dict[tuple[Path, float], ProcessConcept] keyed by (path, mtime). Define ProcessConceptNotFoundError(KeyError) as the typed exception for missing concept lookups. Use a threading.Lock to guard concurrent load() calls — parallel run starts may request the same concept simultaneously.
2
Implement register(concept_id: str, path: str | Path). Resolves the path to absolute. Raises ProcessConceptNotFoundError immediately if the file does not exist at registration time — fail fast at startup, not at first use. Stores the mapping in _registry. Does not load the concept eagerly; loading is deferred to the first load() call so that startup is fast even with many registered concepts.
3
Implement load(concept_id: str) → ProcessConcept. Looks up the path in _registry; raises ProcessConceptNotFoundError with the missing ID in the message if not registered. Reads the file's current mtime. If (path, mtime) is in _cache: return the cached object. Otherwise: call ProcessConcept.from_file(path), store in cache, and return. The mtime key means a concept file edited between runs is automatically reloaded without restarting the adapter.
4
Implement list_available() → list[str]. Returns a sorted list of all registered concept IDs. Used by the canvas process_concept node to populate the concept picker dropdown. Also useful for debugging and the adapter's GET /concepts endpoint (if added in P-PC.5).
5
Implement scan_directory(dir_path: str | Path). Walks the directory for all *.json files. For each file: attempts to read its id field from the JSON without fully parsing (fast peek), then calls register(concept_id, path). Files that fail JSON parsing or are missing the id field are logged as a warning and skipped — the scan never aborts due to a single malformed file. Create a module-level DEFAULT_REGISTRY = ProcessRegistry() and call DEFAULT_REGISTRY.scan_directory("concepts/") at module import time if the directory exists.
Acceptance tests
register() raises ProcessConceptNotFoundError when called with a path that does not exist on disk — the error message includes the path
load() on an unregistered concept_id raises ProcessConceptNotFoundError with the missing ID in the message; a second call with the same ID raises the same error without side effects
scan_directory() registers all valid JSON files in a temp directory and logs exactly one warning (without raising) for a file with invalid JSON structure — the other valid files are still registered
list_available() returns concept IDs in sorted alphabetical order; after scan_directory() on a directory with 3 valid concept files it returns a list of length 3
P-PC.3 Bundled concept JSON files and JSON schema Week 2 · ~1.5 days
Creates the concepts/ directory containing a formal JSON schema (concept_schema.json) and four domain concept files. Each concept file is validated against the schema on load. The four concepts cover the most common AI coding agent task classes: debugging a failing test, implementing a new feature, reviewing code, and refactoring a module.
Implementation steps
1
Create concepts/concept_schema.json. A JSON Schema (draft-07) document covering all fields of ProcessConcept and ProcessConceptStep. Top-level required fields: id, name, description, success_criteria, schema_version, steps. Per-step required fields: id, description, depends_on, risk_level, abstraction_level, expected_tools, success_criteria. strategy_hint is optional and its enum must match the 6 strategy names. risk_level enum: "LOW", "MEDIUM", "HIGH". abstraction_level enum: "leaf", "subgoal". Add schema validation via jsonschema in ProcessConcept.from_dict() — validation runs before field parsing.
2
Create concepts/debug_test_failure.json. 6 steps: read_test (read the failing test, LOW) → run_test (capture live failure output, LOW, expected_tools: ["bash"]) → locate_source (trace stack to source, LOW, expected_tools: ["read_file","grep"]) → form_hypothesis (state root cause and predicted fix, LOW) → apply_fix (apply minimal change, MEDIUM, strategy_hint: "MINIMAL_FIX", expected_tools: ["edit_file"]) → verify_fix (re-run suite, check regressions, LOW, expected_tools: ["bash"]). Linear dependency chain.
3
Create concepts/implement_feature.json. 7 steps: read_requirements (LOW) is the entry point. survey_codebase (LOW, expected_tools: ["grep","read_file"]) and design_interface (LOW, abstraction_level: "subgoal") both depend on read_requirements — this is the only concept with a parallel fan-out. implement (MEDIUM, expected_tools: ["edit_file"]) depends on both survey_codebase and design_interface. write_tests (LOW, expected_tools: ["edit_file"]) depends on implement. run_tests (LOW, expected_tools: ["bash"]) depends on write_tests. refine (LOW) depends on run_tests. This concept tests the parallel fan-out and re-join in seed_task_graph().
4
Create concepts/code_review.json. 5 steps, strictly sequential: read_diff (LOW, expected_tools: ["read_file","bash"]) → check_logic_correctness (LOW) → check_test_coverage (LOW) → check_code_style (LOW) → summarise_findings (LOW). No parallel steps, no strategy hints. The simplest concept — used as the baseline fixture in integration tests where minimal complexity is needed.
5
Create concepts/refactor_module.json. 7 steps: read_current (LOW, expected_tools: ["read_file"]) → run_tests_baseline (LOW, expected_tools: ["bash"]) → identify_hotspots (LOW, strategy_hint: "BROADER_SEARCH", expected_tools: ["grep","read_file"]) → design_target (LOW, abstraction_level: "subgoal") → apply_refactor (HIGH, expected_tools: ["edit_file"]) → run_tests_post (LOW, expected_tools: ["bash"]) → verify_no_regression (LOW). The apply_refactor step with risk_level HIGH verifies that the HIGH-risk path in the execution and verification layers is exercised when this concept seeds the task graph.
Acceptance tests
All 4 bundled concept files parse via ProcessConcept.from_file() without error and each passes validate() — confirmed by asserting no exception is raised and returned objects have the correct step counts (6, 7, 5, 7)
implement_feature.json encodes the parallel fan-out correctly: survey_codebase and design_interface both list read_requirements in their depends_on[]; after seed_task_graph() the implement task depends on both namespaced IDs
refactor_module.json produces an apply_refactor task with risk_level=HIGH after seed_task_graph() — confirmed by iterating the returned task graph and asserting the task with the expected namespaced ID has the HIGH risk level
JSON schema validation (concept_schema.json) correctly rejects a concept file where one step has "risk_level": "CRITICAL" (not in the enum) — ProcessConceptValidationError is raised and its message references the invalid field

New invariants introduced in this phase
This phase introduces four new invariants specific to process concepts. They are checked by the integration tests in P-PC.7 and must hold before this phase is declared complete. They do not replace any of the P0–P9 invariants, which continue to apply unchanged to concept-seeded runs.
INV-PC-01 — Process concept is a strong prior, not a constraint P-PC.4 scope seed_task_graph() pre-populates the task graph but does not lock it. The existing replanning machinery (diagnose_and_replan(), rebuild_task_graph()) must be able to add, modify, or remove tasks during the run without restriction. A verification failure during a concept-seeded run can and should cause the task graph to diverge from the concept's original step list. The concept provides the starting scaffold; the harness's reasoning and recovery layers own the graph from that point forward. Verified by P-PC.7.2.
INV-PC-02 — concept-seeded graph passes the same validate_task_graph() gate P-PC.4 scope A concept-seeded task graph is not treated as pre-validated. After seed_task_graph() populates the graph, validate_task_graph() and decomposition_gate() run exactly as they would for a model-driven graph. A concept whose step structure is incompatible with the objective is caught and reported via the gate's GATHER_CLARIFICATION or ESCALATE response. decomposition_gate() is never bypassed for concept-seeded runs. Verified by P-PC.4 acceptance test 3.
INV-PC-03 — process_concept=None path is structurally unchanged P-PC.4 · P-PC.7 scope When initialize() is called without a process_concept (or with process_concept=None), the resulting HarnessRunState must be structurally identical to a state produced by the pre-concept code. No new required fields, no changed defaults, no altered task graph initialization. The optional process_concept_id field on HarnessRunState is None and does not affect any downstream logic. This ensures that all 208 P0–P9 tests continue to pass without modification. Verified by P-PC.7.3 and P-PC.7.5.
INV-PC-04 — explicitly-specified missing concept is a hard error P-PC.5 scope When harness_meta.process_concept_id is set in the FlowSpec but the named concept cannot be found in the registry, the run must fail with a ProcessConceptNotFoundError — it must not silently fall back to model-driven decomposition. An explicitly specified concept that is missing is a configuration error: the operator intended a specific guided run and silent fallback would produce an unexpectedly unguided run with no indication of the failure. The error response must include the missing concept ID. Verified by P-PC.5 acceptance test 3.
P0–P9 invariants engaged in this phase (unchanged)
The following existing invariants apply to concept-seeded runs without modification:
INV-01 (observation/conclusion separation): Unchanged — no new evidence integration paths. The concept seeds the task graph, not the world model or evidence store.
INV-03 (generation_id double-increment): Unchanged — seeding happens before the first loop iteration; the double-increment per iteration is unaffected.
INV-06 (control_state as sole control input): Unchanged — the concept has no write path to control_state. Task selection from a concept-seeded graph still uses control_state exclusively for go/no-go decisions.
INV-10 (experience_store no-op when absent): Unchanged — warm_start() runs after seed_task_graph() with the same no-op contract when experience_store is unavailable.
