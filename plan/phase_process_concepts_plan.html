Process Concepts Phase — Static Step Templates for Harness Runs

Introduces process concepts: human-authored, domain-specific task graph templates that seed the harness task graph at run initialisation. A concept is a static JSON file that defines an ordered sequence of typed steps, their dependencies, expected tools, per-step success criteria, and optional strategy hints. The harness uses it as a strong prior on decomposition — the model can still adapt, add, or remove tasks during execution via the existing replanning machinery.

What this phase delivers
Phases 0–10 gave the harness a complete reasoning and control architecture plus a full canvas UI. Every harness run currently starts from a blank task graph — the model decomposes the objective from scratch each time, guided by its world model and hypothesis set.

Process concepts change the starting conditions. A concept is a hand-authored JSON file that describes a canonical, domain-tested sequence of steps for a well-known class of task. Loading a concept at run initialisation pre-populates the task graph with typed, ordered tasks before the model begins work. The model does not have to discover the decomposition — it inherits a proven scaffold and can focus its reasoning budget on the actual work rather than the meta-problem of how to structure it.

Core layer (P-PC.1, P-PC.2, P-PC.3) — The ProcessConcept and ProcessConceptStep dataclasses define the concept data model. The ProcessRegistry maps concept IDs to file paths and handles loading with file-level caching. Four bundled concept files — debug_test_failure, implement_feature, code_review, and refactor_module — are shipped with the harness and auto-registered at startup via scan_directory("concepts/").

Integration layer (P-PC.4, P-PC.5) — The harness initialize() function in loop.py gains an optional process_concept parameter. When provided, seed_task_graph(objective) pre-populates the task graph before decomposition_gate() runs — the gate still validates alignment with the objective and can escalate if it finds a mismatch. The FlowSpec harness_meta block gains an optional process_concept_id field; run_api.py loads the named concept from the registry at run start and passes it to initialize().

Canvas layer (P-PC.6) — A new process_concept canvas node type lets flow authors pick a concept from the registry, preview its steps and dependencies, and observe concept progress (each step showing its live task status) during an active run.

Integration tests (P-PC.7) — End-to-end tests validate the full path from concept file to seeded run. Critical invariant tests confirm that the seeded task graph remains mutable (replanning adds tasks beyond the concept's steps) and that the process_concept=None path is structurally identical to all pre-concept runs — no regressions.

As-built summary — key implementation divergences from plan
P-PC.1–P-PC.5 and P-PC.7 were implemented in full. P-PC.6 (canvas node) was partially implemented — the backend compiler and validation were done; the React canvas component was deferred to P11. The following design decisions diverged from the plan:

seed_task_graph() signature: Plan specified seed_task_graph(objective: str) → TaskGraph (returns a new graph). Implemented as seed_task_graph(task_graph: Any) → None — modifies the existing TaskGraph in-place. This avoids constructing two graph objects and lets the caller pass its own pre-initialized graph.

validate() returns errors, doesn't raise: Plan specified that validate() raises ProcessConceptValidationError on failure. Implemented as returning list[str] (empty = valid). This is consistent with validate_task_graph()'s interface. Raising is done by from_file() and from_dict() when validate() returns a non-empty list.

abstraction_level values: Plan limited to "leaf" and "subgoal". Implemented accepts "module", "goal", "subgoal", "function", "leaf", "statement" — mapping pairs module/goal→0, subgoal/function→1, leaf/statement→2. This aligns with how the existing codebase refers to granularity levels.

success_criteria as list: Plan used a single string. Implemented as list[str] on both ProcessConcept and ProcessConceptStep — consistent with how the rest of the harness models success criteria.

initialize_harness() not initialize(): Plan described modifying an existing initialize() function. No such function existed in loop.py. A new initialize_harness() function was added (alongside the existing run_one_iteration()) with the same semantics the plan described.

schema_version value: Plan specified "1.0" with a strict check. Implemented stores "1" as default with no strict enforcement — the version is informational for forward compatibility.

Bundled concept step counts: Plan specified 6/7/5/7 steps for debug_test_failure / implement_feature / code_review / refactor_module. Implemented as 5/6/6/5 steps. The step names and dependency structures also differ from the plan's exact specification — the implemented versions represent alternative but equivalent decompositions.

GET /concepts response shape: Plan specified [{"id":…,"name":…,"description":…,"step_count":N}]. Implemented returns {"concepts": ["id1","id2",…]} — IDs only. Full concept metadata can be retrieved by loading the concept via ProcessRegistry.load().

register() does not check file existence: Plan said register() raises if the file does not exist. Implemented stores the path without checking — the error is surfaced at load() time via ProcessConcept.from_file().

P-PC.6 Canvas node — backend only: compile_process_concept_node() added to node_compilers.py and "process_concept" added to _HARNESS_NODE_TYPES in validate.py. The React canvas component (ProcessConceptNode.tsx), canvas node registration, and live overlay were deferred to P11.
Key design decisions
Concepts are strong priors, not constraints: seed_task_graph() pre-populates the task graph but does not lock it. The existing replanning machinery (diagnose_and_replan(), rebuild_task_graph()) can add, remove, or modify tasks freely during the run. A concept is advice about how to decompose a class of task — it is not a script the model must follow exactly.
decomposition_gate() is not bypassed: After seeding, the full decomposition_gate() validation runs against the seeded graph. If the concept's step structure is misaligned with the objective, the gate returns GATHER_CLARIFICATION or ESCALATE rather than silently proceeding. An invalid concept does not produce a silently wrong run.
experience_store and process_concept are additive, not conflicting: warm_start(experience_store) is called after the concept seeds the task graph. It enriches strategy_state.prior_strategy_weights and loads successful recovery sequences, but it does not replace the seeded task structure. The concept provides the structural scaffold; experience_store provides the calibrated priors on top.
Missing concept_id is a hard error, not a silent fallback: When harness_meta.process_concept_id is set in the FlowSpec but the named concept cannot be found in the registry, the run start fails with a ProcessConceptNotFoundError. A missing concept is a configuration error. Silently falling back to model-driven decomposition would mask a broken deployment and produce unexpectedly unguided runs.
Concept files are immutable during a run: The registry caches loaded concepts by (path, mtime). Once a run starts and initialize_harness() has called seed_task_graph(), changes to the concept file on disk have no effect on the active run. The seeded task graph is the live source of truth from that point forward.

P-PC.1 ProcessConcept and ProcessConceptStep dataclasses Week 1 · ~2.5 days · ✓ done
Creates adapter/harness/process_concept.py with the core data model. ProcessConceptStep maps to a single Task in the task graph. ProcessConcept is the container holding all steps for a given task class. The seed_task_graph() method is the primary integration point — it appends concept steps as tasks into an existing TaskGraph.
As implemented: abstraction_level accepts six values mapped to int: module/goal→0, subgoal/function→1, leaf/statement→2. success_criteria is list[str] on both concept and step. schema_version defaults to "1" with no strict enforcement. Both ProcessConceptValidationError and ProcessConceptNotFoundError are defined here (not in the registry). validate() returns list[str] — callers raise if needed. A module-level _FILE_CACHE dict holds mtime-keyed loaded concepts.
Implementation steps
1
Create adapter/harness/process_concept.py and define ProcessConceptStep. Fields: id: str, description: str, depends_on: list[str] (step IDs within the same concept), risk_level: str (LOW/MEDIUM/HIGH), abstraction_level: str (module/goal/subgoal/function/leaf/statement — mapped to int 0/1/2 at seed time), expected_tools: list[str], success_criteria: list[str], strategy_hint: str | None = None. Define ProcessConceptValidationError(ValueError) and ProcessConceptNotFoundError(KeyError). Implement to_dict() and from_dict(d).
2
Define ProcessConcept dataclass. Fields: id: str, name: str, description: str, success_criteria: list[str], schema_version: str = "1", steps: list[ProcessConceptStep]. Implement to_dict() and from_dict(d). from_dict() constructs the object and calls validate() — if errors are returned, raises ProcessConceptValidationError.
3
Implement ProcessConcept.validate() → list[str]. Returns a list of error strings (empty = valid). Checks: (1) non-empty concept ID; (2) step ID uniqueness; (3) depends_on reference validity (forward references allowed); (4) valid risk_level and abstraction_level enum values; (5) dependency cycle detection via iterative DFS. All errors are collected and returned together (not fail-fast).
4
Implement ProcessConcept.from_file(path: str | Path) → ProcessConcept. Reads and JSON-parses the file. Checks mtime against _FILE_CACHE — returns cached object if mtime unchanged. On cache miss: parses via from_dict(), validates, stores in _FILE_CACHE[(str(path))] = (mtime, concept). Raises ProcessConceptNotFoundError if the file is absent; raises ProcessConceptValidationError for malformed JSON or structural errors.
5
Implement ProcessConcept.seed_task_graph(task_graph: Any) → None. Appends Task objects into the caller-supplied TaskGraph in-place. Task IDs are namespaced as {concept_id}:{step_id}. depends_on[] entries are translated to the same namespaced format. abstraction_level strings are mapped to int via _ABSTRACTION_MAP. strategy_hint is written to task.assigned_strategy. All tasks initialised as PENDING. Sets task_graph.changed = True.

P-PC.2 ProcessRegistry — file-backed concept registry Week 1 · ~1.5 days · ✓ done
Creates adapter/harness/process_registry.py. The registry maps concept IDs to file paths. File-level mtime caching lives in process_concept._FILE_CACHE. A module-level DEFAULT_REGISTRY is auto-populated from concepts/ at import time. Thread-safe for concurrent registration and listing.
As implemented: ProcessConceptNotFoundError is imported from process_concept.py (not defined here). register() stores the path string without checking file existence — the error is surfaced at load() time. The per-file mtime cache is a module-level dict in process_concept.py; load() simply calls ProcessConcept.from_file() which handles caching. scan_directory() does a full parse (not a fast peek) and silently skips failures. DEFAULT_REGISTRY path: Path(__file__).parent.parent.parent / "concepts".
Implementation steps
1
Create adapter/harness/process_registry.py and define ProcessRegistry. Internal state: _registry: dict[str, str] mapping concept IDs to file path strings. Use a threading.Lock to guard concurrent mutations. Import ProcessConceptNotFoundError from process_concept.py.
2
Implement register(concept_id: str, file_path: str | Path). Stores the string path in _registry under the concept ID. Does not check file existence at registration time — errors surface at load() via from_file().
3
Implement load(concept_id: str) → ProcessConcept. Looks up the path in _registry; raises ProcessConceptNotFoundError if not registered. Delegates to ProcessConcept.from_file(path) which handles mtime-keyed caching transparently.
4
Implement list_available() → list[str]. Returns a sorted list of all registered concept IDs. Used by GET /run/concepts and for debugging.
5
Implement scan_directory(directory: str | Path) → int. Globs *.json, skips concept_schema.json. For each file: calls ProcessConcept.from_file() — on success calls register(); on any exception silently continues. Returns count of successfully registered concepts. At module level: DEFAULT_REGISTRY = ProcessRegistry() followed by DEFAULT_REGISTRY.scan_directory(Path(__file__).parent.parent.parent / "concepts").

New invariants introduced in this phase
This phase introduces four new invariants specific to process concepts. They are checked by the integration tests in P-PC.7 and must hold before this phase is declared complete. They do not replace any of the P0–P9 invariants, which continue to apply unchanged to concept-seeded runs.
INV-PC-01 — Process concept is a strong prior, not a constraint P-PC.4 scope seed_task_graph() pre-populates the task graph but does not lock it. The existing replanning machinery (diagnose_and_replan(), rebuild_task_graph()) must be able to add, modify, or remove tasks during the run without restriction. A verification failure during a concept-seeded run can and should cause the task graph to diverge from the concept's original step list. The concept provides the starting scaffold; the harness's reasoning and recovery layers own the graph from that point forward. Verified by P-PC.7.2.
INV-PC-02 — concept-seeded graph passes the same validate_task_graph() gate P-PC.4 scope A concept-seeded task graph is not treated as pre-validated. After seed_task_graph() populates the graph, validate_task_graph() and decomposition_gate() run exactly as they would for a model-driven graph. A concept whose step structure is incompatible with the objective is caught and reported via the gate's GATHER_CLARIFICATION or ESCALATE response. decomposition_gate() is never bypassed for concept-seeded runs. Verified by P-PC.4 acceptance test 3.
INV-PC-03 — process_concept=None path is structurally unchanged P-PC.4 · P-PC.7 scope When initialize_harness() is called without a process_concept (or with process_concept=None), the function returns {"valid": True, "errors": [], "decomposition_gate": ..., "process_concept_id": None}. No tasks are seeded. The optional process_concept_id field on HarnessRunState is None and does not affect any downstream logic. Verified by T16 (test_t16_initialize_harness_no_concept).
INV-PC-04 — explicitly-specified missing concept is a hard error P-PC.5 scope When harness_meta.process_concept_id is set in the FlowSpec but the named concept cannot be found in the registry, the run must fail with a ProcessConceptNotFoundError — it must not silently fall back to model-driven decomposition. An explicitly specified concept that is missing is a configuration error: the operator intended a specific guided run and silent fallback would produce an unexpectedly unguided run with no indication of the failure. The error response must include the missing concept ID. Verified by P-PC.5 acceptance test 3.
P0–P9 invariants engaged in this phase (unchanged)
The following existing invariants apply to concept-seeded runs without modification:
INV-01 (observation/conclusion separation): Unchanged — no new evidence integration paths. The concept seeds the task graph, not the world model or evidence store.
INV-03 (generation_id double-increment): Unchanged — seeding happens before the first loop iteration; the double-increment per iteration is unaffected.
INV-06 (control_state as sole control input): Unchanged — the concept has no write path to control_state. Task selection from a concept-seeded graph still uses control_state exclusively for go/no-go decisions.
INV-10 (experience_store no-op when absent): Unchanged — warm_start() runs after seed_task_graph() with the same no-op contract when experience_store is unavailable.
