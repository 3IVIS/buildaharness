What Phase 4 delivers
Phase 4 gives the harness the ability to plan, schedule, and coordinate work across multiple tasks — including parallel tasks that may write to overlapping state domains.

The task graph (P4.1) introduces the six-state task lifecycle (PENDING → ACTIVE → VERIFYING → COMPLETE / FAILED / BLOCKED) with a directed acyclic dependency structure. The validator catches cycles and inconsistent dependency status at decomposition time. The conflict probability cache (P4.2) computes the likelihood of write-domain conflicts between pairs of parallel tasks and selects the appropriate concurrency mode — pessimistic blocking when probability exceeds a threshold, optimistic concurrency otherwise (with merge-time detection). Parallel branch merge (P4.3) handles the optimistic case: when parallel branches complete, it reconciles their world models by taking the max generation_id, merging observations and beliefs, running contradiction detection on the merged model, and updating the conflict probability cache from actual overlap observed. Abstraction fit checking (P4.4) ensures that task granularity matches world model depth — misalignment reduces the verification_health.feasibility diagnostic, which feeds Tier 4 of the control state resolver.

The task graph state structure is persisted via HarnessRunState and is the primary artifact that P5's execution, P6's recovery, and P9's reviewer pass all operate on.

Key constraints respected in implementation
Task status transitions are strictly defined: PENDING → ACTIVE is the only forward transition from PENDING. BLOCKED is reachable from any non-COMPLETE state when a HIGH contradiction flags a dependency. FAILED is set only by the execution layer (P5) — Phase 4 must not set FAILED from planning logic. COMPLETE is terminal — no task transitions out of COMPLETE except via the reviewer pass (P9) which explicitly resets to PENDING.
Conflict probability cache is informational, not authoritative: The pessimistic/optimistic decision is based on the cached probability, but the cache is built from heuristics (same-domain → 1.0) and historical data. The cache is updated after merge to reflect actual overlap — so over time, the estimates improve. But the initial estimates may be wrong. This is acceptable: the optimistic path has merge-time detection as a safety net.
validate_task_graph() must always run after global replan (from P6): Phase 4 implements the validator; Phase 6 calls it. Phase 4 must ensure the validator is correct and fast — it runs on every global replan and on every decomposition gate passage.

P4.2 conflict_probability_cache COMPLETE Week 2 · ~2.5 days
Implement the conflict probability cache that drives the pessimistic vs optimistic concurrency decision for parallel tasks. The cache stores per-write-domain-pair conflict probabilities, initialised from structural analysis of the task graph and updated from experience store historical data and actual merge-time observations.
Implementation steps
✓ 1
Defined ConflictProbabilityCache dataclass in task_graph.py: probabilities: dict[str, float], pessimistic_threshold: float = 0.5, observation_counts: dict[str, int]. to_dict() / from_dict() included.
Dict keys use string format "da::db" (sorted pair joined by ::) instead of tuple[str, str] — tuple keys cannot be JSON-serialised as dict keys, which would break HarnessRunState.save(). The _key(domain_a, domain_b) helper sorts and joins the pair deterministically.
✓ 2
Implemented compute_initial_conflict_probabilities(task_graph) → ConflictProbabilityCache. For each pair of PENDING tasks with no dependency between them: compute domain set overlap. Identical domain sets → 1.0; disjoint → 0.0; partial overlap → overlap_count / max(len_a, len_b). Stored per (da, db) domain-pair key; takes max across multiple contributing task pairs.
Tasks with no parallel_write_domains are skipped entirely — no cache entries are created for them (they can't conflict on write domains).
✓ 3
Implemented update_from_experience_store(cache, experience_store) → None. Reads experience_store.write_conflict_history (dict mapping domain-pair key → {"conflicts": int, "total_runs": int}). Bayesian blend: new_prob = (structural_prob × N_prior + empirical_rate × N_obs) / (N_prior + N_obs) where N_prior=5.
✓ 4
Implemented should_use_pessimistic_blocking(cache, domain_a, domain_b) → bool. Looks up _key(domain_a, domain_b). Returns True if probability > pessimistic_threshold. Returns False (optimistic) when key is absent — unknown pairs fail open.
✓ 5
Implemented record_actual_overlap(cache, domain_a, domain_b, conflict_observed: bool) → None. Rolling Bayesian update: new_prob = (current_prob × (N_prior + n) + observation) / (N_prior + n + 1) where N_prior=5 and n = existing observation count. Increments observation_counts.
The denominator grows with each observation, so the prior's influence fades as empirical data accumulates. After 10 conflicts starting from 0.0, the probability reaches ≈0.67 (> 0.5 threshold), satisfying T06.

P4.4 Abstraction fit checking COMPLETE Week 4 · ~2 days
Implement check_abstraction_alignment() — the function that verifies task granularity aligns with world model depth. Tasks defined at finer granularity than the world model's current understanding reduce the verification_health.feasibility diagnostic. The check runs conditionally (only when task_graph.changed) during normal execution, and unconditionally in the reviewer pass.
Implementation steps
✓ 1
Implemented estimate_world_model_granularity(world_model) → int in task_graph.py. Returns 0 (module), 1 (function), or 2 (statement) based on keyword density in belief statements. Statement markers: ("line ", "line:", "statement", "expression", "lineno"). Function markers: ("function", "method", "def ", "procedure", "()"). Majority rule: if >50% of beliefs contain marker type, that granularity level is returned; otherwise defaults to 0.
Empty belief lists default to granularity 0 (module level) — an empty world model is treated as coarse-grained.
✓ 2
Implemented check_abstraction_alignment(task_graph, world_model, force=False) → float. Short-circuits to 1.0 when not force and not task_graph.changed. Otherwise: counts tasks where task.abstraction_level > wm_granularity + 1 (much finer than world model). Score = 1.0 - (mismatched / total), clamped to [0, 1].
The force flag was placed directly on check_abstraction_alignment() (not only on update_diagnostics()) so tests and the P9 reviewer pass can call it directly without going through diagnostics.
✓ 3
Wired into update_diagnostics() in diagnostics.py. When task_graph is not None and (force or task_graph.changed): calls check_abstraction_alignment(task_graph, world_model, force=True) and updates verification_health.feasibility with a 0.3-weighted blend: feasibility = 0.3 × alignment_score + 0.7 × current_feasibility.
Blend weight 0.3 leaves headroom for P5.2 (tool availability) and P5.5 (verification layer strength) to contribute their portions of the feasibility composite without the abstraction check dominating.

Implementation decisions and deviations from plan
Recorded for future phases that depend on P4 internals. Each note explains what was decided and why.
Cache key format: string instead of tuple
The plan specified dict[tuple[str, str], float] as the probabilities type. Changed to dict[str, float] with keys in the form "da::db" (sorted pair joined by ::) because Python tuples cannot be used as JSON object keys — they serialize as arrays and round-trip as lists, breaking HarnessRunState.save(). The _key(domain_a, domain_b) helper sorts the pair before joining so lookup is always deterministic regardless of argument order.
force parameter on check_abstraction_alignment directly
The plan placed the force flag only on update_diagnostics(). In the implementation check_abstraction_alignment(task_graph, world_model, force=False) also accepts the flag directly. This lets the P9 reviewer pass call the function standalone without wiring through diagnostics, and keeps T13 simple — T13 calls the function directly with force=True.
T03 test uses ACTIVE (not PENDING) for the "None" case
The original plan description for T03 said "B depends on A which is PENDING". But a PENDING task with no dependencies of its own IS a valid unblocked leaf — select_unblocked_leaf would correctly return it. The test was corrected to set A's status to ACTIVE (a completed-but-not-done state that is not COMPLETE), so B's dependency is unresolved and select_unblocked_leaf correctly returns None.
cross-domain pairs also recorded in reconcile_parallel_branches
The plan described recording overlap only for conflicting domain pairs. The implementation also records conflict_observed=False for cross-domain pairs (da ≠ db) within the same branch task pair. This refines the cache estimate that those specific domains do not conflict, improving future pessimistic/optimistic decisions even in the no-conflict case.
merge_world_models deduplicates assumptions
The plan listed observations, beliefs, contradictions, and environment_change_log as merge targets. Assumptions (a list[str] on WorldModel) were also deduplicated in the merge, preserving insertion order from left to right across branches. This was a small addition beyond spec to avoid duplicated string assumptions in the merged model.
Abstraction fit feasibility blend: 0.3 weight
feasibility = 0.3 × alignment_score + 0.7 × current_feasibility. The 0.3 weight leaves 0.7 of headroom for P5.2 (tool availability VOI) and P5.5 (verification layer strength) to contribute their portions. A T11 task with abstraction_level=2 against a module-level world model produces alignment_score=0.0, which reduces feasibility from 1.0 to exactly 0.7 — clearly below the 1.0 baseline without being a hard block on its own.
estimate_world_model_granularity keyword sets
Statement markers: ("line ", "line:", "statement", "expression", "lineno"). Function markers: ("function", "method", "def ", "procedure", "()"). A majority-vote rule (>50% of beliefs matching a marker type) determines the returned level. Empty belief lists default to 0. This is a heuristic and will miss edge cases, but it is sufficient for the feasibility signal — the intent is to catch gross misalignment (statement-level tasks against module-level beliefs), not precise classification.
