What Phase 5 delivers
Phase 5 implements the execution pipeline — the sequence of checks and actions that occur each time the agent is about to perform a mutation. The pipeline flows left to right:

Risk estimation (P5.1) classifies the current task as LOW/MEDIUM/HIGH risk based on file centrality, change scope, and module type. HIGH risk tasks trigger additional verification depth and an adversarial pass. VOI estimation (P5.2) decides whether gathering more evidence would meaningfully reduce decision uncertainty before proceeding. The verification adequacy critic checks which of the nine verification layers are available and enforces a minimum adequacy bar — unresolvable inadequacy degrades verification_health.strength without looping forever. Pre-execution review gate (P5.3) validates five dimensions of the proposed change before any mutation occurs — two consecutive failures on the same task escalate to human rather than retrying a third time. Execute (P5.4) performs the mutation, selects a reversibility strategy, records in environment_change_log, and creates SYSTEM_ERROR evidence on tool failure. 9-layer verification (P5.5) validates the result across syntax through output contract layers, pruning layers whose tools are unavailable. Contract shadow check (P5.6) replaces the P0.5 stub with a real interface-field checker running at the post_exec_gate. action_dep_overlap (P5.7) checks whether a proposed action's required state overlaps with compressed or pruned memory regions.

Key constraints to respect in Phase 5
Tool error path always produces SYSTEM_ERROR evidence (not an exception): When a tool call fails in the execute node, the error must be captured as Evidence(reliability=HIGH, evidence_type=SYSTEM_ERROR) and fed into update_world_model(). The execution does not raise — it returns a failed ExecutionResult. Raising instead would break the invariant that world model state is always up-to-date.
Verification layer pruning must not double-penalise feasibility: When a layer is pruned because its tool is unavailable (from the tool availability manifest), the pruning itself must not reduce verification_health.strength further — the tool availability penalty was already applied during the adequacy critic check. Layers pruned by the adequacy critic are marked SKIPPED, not FAILED.
Two consecutive pre-execution review failures escalate — no third attempt: The pre-execution review gate tracks a per-task consecutive failure count. On the first failure: block and re-invoke select_best_action(). On the second consecutive failure for the same task: escalate to human. Never attempt a third review on the same task/action pair without human input.

P5.1 Risk estimation Week 1 · ~2 days
Implement estimate_risk() — a three-factor classification function that determines whether the current task is LOW, MEDIUM, or HIGH risk. HIGH risk tasks receive smaller edit sizes, deeper verification, and an adversarial verification pass.
Implementation steps
1
Create adapter/harness/risk.py. Define RiskLevel Literal: "LOW" | "MEDIUM" | "HIGH". Define RiskFactors dataclass: file_centrality: float (normalised [0,1]), change_scope: float (normalised [0,1]), module_type_score: float (0.0 = test-only, 0.5 = utility, 1.0 = core business logic or infrastructure).
2
Implement compute_file_centrality(file_path, world_model) → float. Counts import references to the file in the world model's environment context. Normalises by dividing by an expected-maximum constant (default 50). Returns a value in [0,1]: 0.0 = leaf file, 1.0 = heavily imported core module.
3
Implement compute_change_scope(task) → float. Estimates the scope of the proposed change from the task description: count mentions of function/method names (each adds 0.1), file references (each adds 0.2), line ranges (each adds 0.05). Normalise to [0,1] with max 1.0 at scope=10.
4
Implement classify_module_type(file_path) → float. Returns: 0.0 if path contains "test" or "spec"; 0.5 if path contains "util", "helper", "common"; 1.0 otherwise (core business logic or infrastructure). File path classification is heuristic — accepts an override dict for custom project structures.
5
Implement estimate_risk(current_task, world_model) → RiskLevel. Computes all three factors. Risk score = 0.4 × file_centrality + 0.3 × change_scope + 0.3 × module_type_score. HIGH if score ≥ 0.7; MEDIUM if score ≥ 0.4; LOW otherwise. Also updates current_task.risk_level with the computed result.
Acceptance tests
A core business logic file with 40 imports in the world model → HIGH risk; a test-only file with 2 imports → LOW risk
estimate_risk() updates task.risk_level as a side effect — the task retains the classification for downstream use
HIGH risk task triggers adversarial verification pass in P5.5 (integration test with P5.5)
P5.2 VOI estimation & verification adequacy critic Week 1 · ~2.5 days
Implement the value-of-information estimator and the verification adequacy critic. The VOI estimator decides whether gathering more evidence before acting would meaningfully reduce uncertainty. The adequacy critic uses the tool availability manifest to prune unresolvable verification layers and enforce a minimum adequacy bar — unresolvable inadequacy updates verification_health.strength without looping.
Implementation steps
1
Create adapter/harness/voi.py. Define VOIResult dataclass: voi_score: float, expected_uncertainty_reduction: float, decision_impact: float, should_gather: bool.
2
Implement estimate_value_of_information(evidence_store, world_model, current_task) → VOIResult. expected_uncertainty_reduction = fraction of open hypotheses that a new evidence item would discriminate (proxy: 1 - explanation_coverage from diagnostics). decision_impact = task.risk_level mapped to float (LOW=0.3, MEDIUM=0.6, HIGH=1.0). VOI = product of both. should_gather = voi_score > VOI_THRESHOLD (default 0.3). HIGH VOI → caller should invoke gather_additional_evidence() before proceeding.
3
Define AdequacyResult dataclass: adequate: bool, available_layers: list[str], pruned_layers: list[str], requires_adversarial_pass: bool, resolution: Literal["proceed","gather_evidence","escalate"].
4
Implement verification_adequacy_critic(tool_manifest, task_risk, evidence_store) → AdequacyResult. Checks each of the 9 verification layer names against tool_manifest.check_tool_availability(). Unavailable layers → added to pruned_layers (not penalised further in strength). If fewer than 3 layers are available: adequate=False, resolution="gather_evidence". If task_risk==HIGH: requires_adversarial_pass=True. Unresolvable (fewer than 2 layers + no fallbacks): adequate=False, resolution="escalate", and call update_verification_strength() to reduce the diagnostic.
5
Implement update_verification_strength(diagnostics, available_layer_count) → None. Sets diagnostics.verification_health.strength = normalise(available_layer_count / 9, "ratio"). Called only when the adequacy critic determines that unavailability is unresolvable — not when tools are simply missing but have fallbacks.
Acceptance tests
High VOI (low explanation_coverage + HIGH risk task) triggers should_gather=True; Low VOI (high coverage + LOW risk) triggers should_gather=False
LOW adequacy (fewer than 3 available layers) with available tools sets resolution="gather_evidence" — not "escalate"
Unresolvable inadequacy (fewer than 2 layers, no fallbacks) updates verification_health.strength and does not loop — sets resolution="escalate" and returns
P5.3 Pre-execution review gate — 5 dimensions Week 2 · ~3 days
Implement the five-dimension pre-execution review gate that validates a proposed change before any mutation occurs. A failed review blocks the action and triggers re-selection. Two consecutive failures on the same task escalate to human rather than attempting a third review — this prevents endless loops on fundamentally incompatible proposals.
Implementation steps
1
Create adapter/harness/review_gate.py. Define ReviewDimension Literal: "task_alignment" | "world_model_consistency" | "output_contract_precheck" | "code_quality" | "hypothesis_compatibility". Define DimensionResult dataclass: dimension: ReviewDimension, passed: bool, reason: str. Define ReviewResult dataclass: passed: bool, failed_dimensions: list[DimensionResult], consecutive_failures: int.
2
Implement dimension 1 — task requirements alignment. check_task_alignment(proposed_change, current_task) → DimensionResult. Verifies that the proposed change's scope (files/functions modified) overlaps with the task description's stated targets. Fails if the change touches files not mentioned in the task description and the task has LOW risk (HIGH risk tasks allow broader scope by design).
3
Implement dimension 2 — world model consistency. check_world_model_consistency(proposed_change, world_model) → DimensionResult. For each HIGH-reliability belief in world_model.beliefs[]: check if the proposed change would contradict the belief (e.g., proposed to add a function that a belief asserts "does not exist"). Fails if any HIGH-reliability belief is directly contradicted by the proposed change.
4
Implement dimension 3 — output contract pre-check. check_output_contract(proposed_change, output_contract) → DimensionResult. Verifies that the proposed change does not remove any field listed in output_contract.required_interface_fields or change the type signature of any constrained interface. Lightweight static check — not a full contract validation.
5
Implement dimension 4 — code quality heuristics. check_code_quality(proposed_change, tool_manifest) → DimensionResult. If a linter is available in tool_manifest: run it on the proposed change (via subprocess). Fails if the linter reports errors (not warnings). If no linter available: returns DimensionResult(passed=True, reason="linter unavailable — skipped").
6
Implement dimension 5 — open hypothesis compatibility. check_hypothesis_compatibility(proposed_change, hypothesis_set) → DimensionResult. For each active hypothesis with high confidence (≥ 0.6): check if the proposed change would invalidate the hypothesis's predicted_observations. Fails if any high-confidence hypothesis predicts an outcome that the proposed change would make impossible.
7
Implement review_proposed_change(...) → ReviewResult. Calls all 5 dimension checks. Collects results. If any dimension fails: return ReviewResult(passed=False, failed_dimensions=[...]). Track consecutive_failures per task in a dict stored on the harness run state (persisted). If consecutive_failures[task_id] >= 2: set resolution="escalate" on the ReviewResult.
Acceptance tests
A proposed change that contradicts a HIGH-reliability belief fails dimension 2 (world_model_consistency) with a specific reason message
A proposed change that removes a required_interface_field fails dimension 3 (output_contract_precheck)
Two consecutive review failures on the same task_id set consecutive_failures >= 2 and trigger escalation — no third review is attempted
P5.4 Reversibility strategy + execute node Week 3 · ~3 days
Implement the reversibility strategy selection and the execute node — the core mutation point of the harness. The execute node records the action in environment_change_log, creates SYSTEM_ERROR evidence on tool failure, and updates task status. The loop.py skeleton is populated with the full execution flow.
Implementation steps
1
Create adapter/harness/execution.py. Define ReversibilityStrategy Literal: "snapshot" | "git-revert" | "patch-rollback" | "ephemeral". Define ExecutionResult dataclass: success: bool, output: str | None, error: str | None, strategy: ReversibilityStrategy, rollback_ref: str | None.
2
Implement select_reversibility_strategy(proposed_change, task_risk) → ReversibilityStrategy. Rules: read-only actions → "ephemeral" (no snapshot needed); file mutations + LOW risk → "patch-rollback"; file mutations + MEDIUM/HIGH risk → "git-revert" if in a git repo, "snapshot" otherwise; schema/infrastructure changes → "snapshot" always.
3
Implement execute(proposed_change, tool_workflow, world_model, task_graph, current_task) → ExecutionResult. Steps: (1) select reversibility strategy; (2) create snapshot/rollback ref if needed; (3) run the tool workflow; (4) record result in world_model.environment_change_log as a new entry with affected_source, timestamp, and outcome; (5) update current_task.status to ACTIVE during execution. Returns ExecutionResult regardless of success/failure.
4
Implement tool error path. If tool_workflow raises or returns an error: create Evidence(obs=str(error), reliability="HIGH", source=tool_name, evidence_type="SYSTEM_ERROR", freshness=1.0). Call evidence_store.append(error_evidence). Call update_world_model(evidence_store, world_model). Set ExecutionResult.success=False and error=str(error). This ensures the SYSTEM_ERROR evidence enters the world model for contradiction detection on the next loop.
5

Replace the skeleton stub with: check_external_updates → staleness_sweep → update_diagnostics → resolve_control_state → decomposition_gate → select_task → estimate_risk → estimate_voi → [gather_evidence if needed] → review_proposed_change → action_gate → execute → verify → post_exec_gate → update_experience_store. Each step reads from and writes to HarnessRunState.

Successful execution records the result in environment_change_log with the correct affected_source and timestamp

P5.5 9-layer verification Week 4 · ~3 days
Implement the full nine-layer verification system. Layers are pruned based on tool availability (not double-penalised — already accounted for in adequacy critic). HIGH risk tasks trigger an additional adversarial verification pass. Evidence-sufficiency checking is claim-scoped — global claims require more supporting evidence than local claims.
Implementation steps
1
Create adapter/harness/verification.py. Define VerificationLayer Literal: "syntax" | "unit" | "integration" | "consistency" | "requirements" | "assumptions" | "goal_correctness" | "evidence_sufficiency" | "output_contract_partial". Define LayerResult dataclass: layer: VerificationLayer, status: Literal["PASS","FAIL","SKIPPED"], detail: str. Define VerificationResult dataclass: layer_results: list[LayerResult], has_critical_failure: bool, adversarial_passed: bool | None = None.
2
Implement layer functions. One function per layer: verify_syntax(result, tool_manifest) runs available linter/parser; verify_unit(result, tool_manifest) runs pytest on affected test files; verify_integration(result, tool_manifest) runs integration test subset; verify_consistency(result, world_model) checks result against world model beliefs; verify_requirements(result, success_criteria) checks against success criteria; verify_assumptions(result, world_model) checks result does not invalidate active assumptions; verify_goal_correctness(result, current_task) checks alignment with task description; verify_evidence_sufficiency(result, evidence_store, scope) claim-scoped threshold check; verify_output_contract_partial(result, output_contract) partial contract check.
3
Implement claim-scoped evidence_sufficiency. scope parameter: "global" requires ≥5 HIGH or MEDIUM reliability evidence items; "local" (default) requires ≥2. A global claim (e.g., "this function is never called") requires more evidence than a local claim (e.g., "this specific path fails").
4
Implement verify(result, success_criteria, assumptions, tool_manifest, task_risk) → VerificationResult. For each of 9 layers: if tool unavailable (per manifest) → SKIPPED (no strength penalty beyond what adequacy critic already applied). Else run the layer function. has_critical_failure = any(r.status == "FAIL" for r in layer_results). If task_risk == "HIGH": run adversarial verification pass (simplified — checks if the result could be valid under the negation of the top-confidence hypothesis).

P5.7 action_dep_overlap() compression check Week 5 · ~1 day
Implement the compression-aware dependency overlap check. Before executing an action, verify that the action's required state structures have not been pruned or dropped by the compression system (P6). Escalation fires only when the overlap is non-empty AND risk is HIGH. Low-risk overlap produces a warning only.
Implementation steps
1
Implement action_dep_overlap(action, memory_state) → list[str] in execution.py. The action has a required_state_structures: list[str] attribute (structure names the action reads). Checks against both memory_state.compression_risk.compressed_structures[] (fully dropped) and memory_state.compression_risk.pruned_regions[] (partially truncated). Returns the list of overlapping structure names.
2
Implement overlap-based escalation logic in loop.py. After calling action_dep_overlap(): if overlap is non-empty AND task_risk == HIGH → call escalate(surface_blocker(reason="action_requires_compressed_state", missing_info=overlap)). If overlap is non-empty AND task_risk != HIGH → log a warning to execution_journal and continue. If overlap is empty → continue unconditionally.

Architecture invariants engaged in Phase 5
INV-01 — Observation vs conclusion separation (SYSTEM_ERROR path) P5.4 enforcement The tool error path creates Evidence with evidence_type="SYSTEM_ERROR". This evidence enters evidence_store and then world_model.observations[] via integrate_evidence(). It must never auto-create a Belief. The P5.4 test asserts that after a tool failure, the SYSTEM_ERROR evidence is in world_model.observations[] but not in world_model.beliefs[].
INV-06 — control_state as sole control input (execute node) P5.4 enforcement The execute node reads control_state.risk_state to decide whether to proceed (NORMAL/CAUTIOUS) or refuse (BLOCKED). It never reads world_model fields or hypothesis_set to make this decision. The action_gate blocks execution when BLOCKED — the execute function itself does not check world_model state for go/no-go decisions.
