What Phase 7 delivers
Phase 7 completes the caller interaction loop. Prior phases treat caller_state as a static input to the run — they read constraints and success criteria but have no mechanism to respond to mid-run updates. Phase 7 closes that gap.

check_external_updates() (P7.1) is a non-blocking poll inserted at the top of every main loop iteration. If the caller has posted a constraint update since the last iteration, the update is ingested immediately via inject_clarification() and the constraint change propagation path fires. If there is no update, the poll completes in under 10ms and the loop continues without any state mutation — no unnecessary staleness increments. Constraint change propagation path (P7.2) is the shared code path triggered by both proactive updates (from the poll) and reactive updates (from human responses after escalation). It runs: detect_contradictions(updated_success_criteria) → update_success_criteria → update_output_contract → revalidate_task_graph → generation_id increment → resolve_control_state. Having a single shared path ensures that both triggers produce identical state changes. Escalation with surface_blocker (P7.3) defines the structured escalation surface — the minimum information the harness surfaces to a human when it cannot continue autonomously. The surface_blocker includes a reason, missing_info, and current task context. After human response, the same constraint change propagation path from P7.2 handles the incoming clarification, ensuring consistent state transitions regardless of how the constraint change arrived.

Key constraints to respect in Phase 7
check_external_updates() must be non-blocking: The poll must complete in under 10ms regardless of whether an update is available. It must not block the main loop waiting for a response. If the underlying transport (e.g., a Postgres notification channel, Redis pubsub, or webhook inbox) is unavailable, the poll returns no-update silently — it never raises. Test the 10ms bound explicitly.
Both update paths must call the same propagation function: The constraint change triggered by check_external_updates() and the constraint change triggered by a human response after escalation must both call the same apply_constraint_change_propagation() function. Two separate implementations will drift. The propagation function is tested once — not twice for two code paths.
surface_blocker presents minimum required information: escalate() must surface the reason (why the harness stopped), missing_info (what the human needs to provide), and current task context (what was being attempted). It must not dump raw world model state or hypothesis sets — those are for debugging, not for the caller. The structured format ensures the harness can be resumed without a restart.

P7.1 check_external_updates() — non-blocking poll Week 1 · ~2.5 days
Implement the non-blocking external update poll inserted at the top of each main loop iteration. When no update is available, the function is a near-zero-cost no-op. When an update is available, it triggers the constraint change propagation path (P7.2) and refreshes control_state before the iteration proceeds.
Implementation steps
1
Create adapter/harness/external_updates.py. Define PendingUpdate dataclass: update_type: Literal["constraint","clarification","success_criteria"], payload: dict, received_at: datetime. Define UpdateChannel abstract interface with one method: poll() → PendingUpdate | None. Implement NoOpUpdateChannel that always returns None — used as the default when no external channel is configured.
2
Implement PostgresNotifyChannel as a concrete UpdateChannel. Uses a non-blocking LISTEN / NOTIFY approach via psycopg2.connection.poll(). Sets connection to non-blocking mode; if no notification is pending within 0ms, returns None immediately. Wraps all poll operations in try/except so a DB connectivity issue returns None rather than raising.
3
Implement check_external_updates(channel, caller_state, world_model, task_graph, diagnostics) → bool. Calls channel.poll(). If None: returns False immediately (no state mutation). If an update is returned: calls inject_clarification(caller_state, update.payload), then calls apply_constraint_change_propagation() (from P7.2), then increments world_model.generation_id. Returns True to signal the loop to re-resolve control_state before proceeding.
4
Wire into loop.py. At the very top of each main loop iteration — before staleness_sweep() and before update_diagnostics() — call check_external_updates(). If it returns True: refresh control_state via resolve_control_state() before continuing to the decomposition_gate. If False: continue normally.
5
Add configuration support. Accept an update_channel keyword argument in the HarnessRunState constructor. Default to NoOpUpdateChannel(). Document that any class implementing the UpdateChannel interface can be used — enables webhook channels, Redis pubsub, or mock channels for testing.
Acceptance tests
check_external_updates() with a NoOpUpdateChannel completes in under 10ms and returns False — no state mutation occurs, world_model.generation_id unchanged
When a mock channel returns a PendingUpdate with a new constraint: caller_state.constraints_changed is True, world_model.generation_id is incremented, and the propagation path fires — verified by checking task_graph for re-validation
No-update case: loop continues without state mutation — caller_state.clarification_history length is unchanged, control_state generation_id is not invalidated
P7.2 Constraint change propagation path Week 2 · ~2.5 days
Implement the shared constraint change propagation function. This is the canonical path called by both check_external_updates() when a proactive update arrives and by the escalation response handler when the human responds. Both callers use the same function — divergent implementations are not permitted.
Implementation steps
1
Implement update_success_criteria(caller_state, world_model) → None in caller_state.py. Re-evaluates which current beliefs and observations are relevant given the new success_criteria. Marks beliefs that reference objectives outside the updated success_criteria scope as stale (adds them to world_model's stale_flags). Does not delete beliefs — only flags them.
2
Implement update_output_contract(caller_state, output_contract) → OutputContract in output_contract.py. Replaces output_contract.caller_specific_constraints with the new constraints from caller_state.current_constraints. Re-derives required_interface_fields where the constraint specifies mandatory output fields. Returns a new OutputContract (immutable update).
3
Implement revalidate_task_graph(task_graph, caller_state, world_model) → TaskGraph. For each task in task_graph: check if the task's scope is still within the updated success_criteria. If a task's target falls outside the narrowed scope: set status to BLOCKED with reason "scope_eliminated". If the scope expanded and new work is needed: add new PENDING tasks derived from the new criteria.
4
Implement apply_constraint_change_propagation(caller_state, world_model, task_graph, output_contract, diagnostics) → tuple[TaskGraph, OutputContract]. Calls in sequence: (1) update_success_criteria(), (2) detect_contradictions() on updated belief set, (3) update_output_contract(), (4) revalidate_task_graph(). Returns updated (task_graph, output_contract). The caller (check_external_updates or escalation handler) then increments generation_id and calls resolve_control_state().
5
Verify single code path. In external_updates.py (P7.1) and in the escalation handler (P7.3), both call apply_constraint_change_propagation() directly. Write a parametrised test that runs both trigger paths with the same input and asserts identical output state — this is the conformance test that prevents path drift.
Acceptance tests
A scope-narrowing constraint update (removing a target from success_criteria) sets tasks targeting that scope to BLOCKED with reason "scope_eliminated" — tasks outside the narrowed scope are not affected
A scope-expanding constraint update (adding a new success criterion) adds new PENDING tasks to the task_graph derived from the new criterion — existing tasks are not disturbed
Both the check_external_updates() path and the escalation response path call apply_constraint_change_propagation() and produce identical task_graph and output_contract state for the same input — verified by the parametrised conformance test
P7.3 Escalation with surface_blocker Week 3 · ~2 days
Implement the escalation surface — the structured halt point at which the harness surfaces minimum required information to the human and awaits a response. After a response arrives, the same constraint change propagation path from P7.2 handles the update, ensuring a consistent resume path.
Implementation steps
1
Create adapter/harness/escalation.py. Define SurfaceBlocker dataclass: reason: str (structured reason from a defined set: "blocked_state" | "cannot_make_progress" | "budget_exhausted" | "review_failure" | "action_requires_compressed_state"), missing_info: list[str] (what the human must provide), current_task_summary: str (brief description of what was being attempted), escalated_at: datetime. Design the type to carry only minimum required information — no raw world model dumps.
2
Implement escalate(surface_blocker, harness_run_state, run_id) → None. Steps: (1) set harness_run_state.escalation_pending = True; (2) persist the surface_blocker to HarnessRunState as pending_escalation: SurfaceBlocker; (3) emit a structured log entry to the execution_journal with action_class="escalation" and the blocker reason; (4) save HarnessRunState to Postgres; (5) halt the loop by raising EscalationHalt(blocker) — a non-error exception the loop runner catches and converts to a paused run state.
3
Implement await_clarification(run_id, db) → PendingUpdate | None. Checks if a human response has been posted to harness_run_state.pending_clarification for the given run_id. Returns None if no response yet — the caller (typically a resume endpoint) calls this function and exits if None. Returns PendingUpdate if a response is present, clears the pending_clarification field, and clears escalation_pending flag.
4
Add resume endpoint to run_api.py. POST /runs/{run_id}/escalation/respond: accepts a JSON body with {"clarification": {...}}. Writes the payload to harness_run_state.pending_clarification. Returns 200 with the updated run state. The main loop on next execution calls await_clarification(), receives the update, and calls apply_constraint_change_propagation() to resume.
5
Wire escalation triggers in loop.py. Two trigger points: (1) after resolve_control_state() when risk_state == BLOCKED — call escalate(SurfaceBlocker(reason="blocked_state", ...)); (2) after cannot_make_progress() returns True and all strategies are exhausted (current_strategy == ESCALATE) — call escalate with reason="cannot_make_progress". Both use the same escalate() function.
Acceptance tests
BLOCKED risk_state triggers escalate(): harness_run_state.escalation_pending is True, pending_escalation.reason is "blocked_state", EscalationHalt is raised, loop halts cleanly
cannot_make_progress()==True with current_strategy=ESCALATE triggers escalation with reason="cannot_make_progress"; the surface_blocker.missing_info list is non-empty
surface_blocker contains reason and missing_info but does not contain raw world_model, hypothesis_set, or evidence_store data — the escalation is human-readable, not a debug dump

Architecture invariants engaged in Phase 7
INV-03 — generation_id double-increment (constraint change path) P7.1 + P7.3 enforcement When a constraint update arrives (either via the external poll or after escalation response), world_model.generation_id must be incremented after the constraint propagation completes and before any gate reads control_state. This is the same increment mechanism as the standard loop sub-steps — not a special-case increment. The generation_id staleness check (P0.3) will catch any gate that attempts to read stale control_state after a constraint change.
INV-06 — control_state as sole control input (escalation trigger) P7.3 enforcement The escalation in P7.3 is triggered by control_state.risk_state == BLOCKED — not by reading world_model.contradictions[] directly or hypothesis_set state. The same invariant that governs select_best_action() applies here: control_state is the sole source of control decisions. The world_model state that caused BLOCKED is accessible only as context for surface_blocker content, not as the decision input.
