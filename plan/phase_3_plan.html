What Phase 3 delivers
Phase 3 is the control intelligence of the harness. It takes everything built in P0–P2 — the world model, evidence, hypotheses, belief graph, and contradictions — and synthesises them into a single risk_state signal (NORMAL / CAUTIOUS / BLOCKED) that drives all subsequent action selection.

The diagnostic health vectors provide the ten normalised sub-dimensions that feed the control state resolver. The normalisation contract ensures all inputs are commensurable before Tier 4 arithmetic — each dimension has its own normalisation method. The five-tier resolver applies rules in strict order: Tier 1 handles SYSTEM_BREAKING contradictions, Tier 2 detects critical dimension failures and deadlocks, Tier 3 identifies coverage gaps, Tier 4 applies proportional caution, Tier 5 returns NORMAL. Deadlock detection in Tier 2 uses directed cycle detection on the recovery-action graph to catch cases where two blocked dimensions each require the other to be resolved first.

The full generation_id gate implementations replace the P0.3 stubs — each gate now performs its real staleness assertion plus a re-resolution of control_state when stale, plus its phase-specific logic. The ControlStateStub placeholder is retired and replaced by the full ControlState dataclass.

Key constraints to respect in Phase 3
Normalisation contract is non-negotiable (INV-02): Every diagnostic sub-dimension value entering resolve_control_state() must be normalised to [0,1] via the dimension-specific normalise() method before any tier uses it. Raw values entering Tier 4 arithmetic are a protocol violation. All tier code must assert normalised input — never assume it.
control_state is the sole control input (INV-06): select_best_action() (stubbed in P3, wired in P5) reads control_state exclusively for control decisions. World model, hypothesis set, and task graph are read-only informational context. Phase 3 must not create any code path where world model state or belief values directly suppress or permit actions without going through resolve_control_state().
dep_class_gap_annotation never enters arithmetic (INV-07): The annotation is attached to control_state.notes[] as a string. The dep_class_gap_annotation field must not appear as a parameter in any tier 1–4 computation inside resolve_control_state(). Tests in P3.3 assert this by inspecting the function signature and call sites.

P3.1 Diagnostic health vectors ✓ Done
Implement all four health vectors as typed dataclasses with their ten sub-dimensions. The vectors accumulate proxy values from world model state, hypothesis set, verification results, and execution history. update_diagnostics() recomputes all ten sub-dimensions from current system state in a single pass. The dep_class_gap_annotation is a string field, not a numeric sub-dimension.
Implementation steps
1
Create adapter/harness/diagnostics.py. Define BeliefHealth dataclass: freshness: float = 1.0, consistency: float = 1.0, support: float = 1.0. All fields represent pre-normalised proxy values — they will be normalised by P3.2 before entering the resolver.
2
Define CoverageHealth dataclass: symptom_coverage: float = 1.0 (fraction of known symptoms that have at least one hypothesis to explain them), explanation_coverage: float = 1.0 (fraction of active hypotheses with at least one discriminating evidence entry).
3
Define VerificationHealth dataclass: strength: float = 1.0 (fraction of 9 verification layers available and passing — updated by P5.5), feasibility: float = 1.0 (composite of abstraction alignment from P4.4, tool availability from P1.3, and VOI adequacy from P5.2 — stub value 1.0 until those phases are implemented).
4
Define ExecutionHealth dataclass: progress_rate: float = 1.0 (ratio of tasks completed this iteration / total tasks), failure_recurrence: float = 0.0 (fraction of recent iterations ending in the same failure mode — 0.0 is healthy), oscillation_score: float = 0.0 (fraction of recent risk_state transitions that were reversals — 0.0 is healthy).
5
Define Diagnostics dataclass: belief_health: BeliefHealth = field(default_factory=BeliefHealth), coverage_health: CoverageHealth = field(default_factory=CoverageHealth), verification_health: VerificationHealth = field(default_factory=VerificationHealth), execution_health: ExecutionHealth = field(default_factory=ExecutionHealth), dep_class_gap_annotation: str | None = None. The annotation is a string, never a numeric field (INV-07).

Recomputes all sub-dimensions in a single pass: belief_health from world_model.stale_flag_ratio, contradiction density, and mean belief reliability; coverage_health from symptom/hypothesis counts; verification_health uses stub values (updated in P5); execution_health from task completion velocity and failure recurrence in execution_journal.
7
Add to_dict() / from_dict() to all four health vector dataclasses and Diagnostics. Extend HarnessRunState with diagnostics: Diagnostics and update save() / load().

P3.2 Normalisation contract ✓ Done
Implement the normalisation contract — the requirement that all diagnostic sub-dimension values entering resolve_control_state() are normalised to [0,1] via dimension-specific methods. This is INV-02: Tier 4 arithmetic is only valid when inputs are commensurable. The normalise() function dispatches to the correct method based on the dimension_type argument.
Implementation steps
1
Add DimensionType Literal to diagnostics.py: Literal["ratio","composite","entropy","match_confidence"]. These four types cover all ten sub-dimensions: freshness/consistency/support/strength/feasibility/progress_rate are ratio; failure_recurrence/oscillation_score are composite; symptom_coverage/explanation_coverage use entropy; matched_pattern.confidence uses match_confidence.
2
Implement normalise_ratio(raw: float) → float: clamps to [0,1] using max(0.0, min(1.0, raw)). Used for freshness, consistency, support, strength, feasibility, progress_rate.
3
Implement normalise_composite(raw: float, weights: list[float], components: list[float]) → float: computes weighted sum of components, divides by sum of weights, clamps to [0,1]. Used for failure_recurrence and oscillation_score which are blended from multiple sub-signals.
4
Implement normalise_entropy(source_counts: dict[str, int]) → float: computes Shannon entropy over the source frequency distribution, divides by log2(num_unique_sources) (handling division by zero when fewer than 2 sources: returns 0.0). Used for symptom_coverage and explanation_coverage. Clamps to [0,1].
5
Implement normalise_match_confidence(raw: float) → float: clamps to [0,1]. The failure mode library returns a raw confidence in [0,1] by design; this function provides the explicit clamping contract so callers cannot accidentally pass un-normalised library outputs into Tier 4.
6
Implement normalise(raw_value, dimension_type: DimensionType, **kwargs) → float: dispatches to the correct method. Raises ValueError for unknown dimension types. All calls to Tier 4 in resolve_control_state() must pass through normalise() — not call the sub-methods directly.
7
Add assertion wrapper. Implement assert_normalised(value: float, label: str) → float: if value < 0.0 or value > 1.0, raise NormalisationError(f"{label} value {value} is outside [0,1]"). All tier computations inside resolve_control_state() call this before using values in comparisons or arithmetic.
Acceptance tests
normalise(1.5, "ratio") returns 1.0 (upper clamp)
normalise(-0.1, "ratio") returns 0.0 (lower clamp)
normalise_entropy({"A": 1, "B": 1, "C": 1, "D": 1}) returns 1.0 (uniform distribution over 4 sources = max entropy)
assert_normalised(1.5, "test_dim") raises NormalisationError

The full ControlState dataclass replaces the ControlStateStub from P0.3. The dep_class_gap_annotation is attached to control_state.notes[] — never used in tier arithmetic.

Create adapter/harness/control_state.py. Define RiskState Literal: "NORMAL" | "CAUTIOUS" | "BLOCKED". Define BlockEntry dataclass: dimension: str, value: float, recovery_action_class: str. Define full ControlState dataclass: generation_id: int = 0, risk_state: RiskState = "NORMAL", escalation_reason: str | None = None, block_mask: list[BlockEntry] = field(default_factory=list), notes: list[str] = field(default_factory=list). Add to_dict() / from_dict().
2
Update staleness.py to import the real ControlState from control_state.py instead of the ControlStateStub. Remove the stub definition. Update all existing P0 tests to use the real dataclass — they should pass without changes since ControlState has the same generation_id field.
⚠ Deviation ControlStateStub was kept in staleness.py rather than removed. control_state.py imports StalenessError from staleness.py; if staleness.py also imported ControlState from control_state.py this would create a circular import that Python cannot resolve at module load time. Instead, ControlStateStub remains as a backward-compat class (same generation_id field) and all P0 tests continue to import and use it unchanged. New P3 code imports ControlState directly from control_state.py.
3
Implement PRE-check and Tier 1 in resolve_control_state(diagnostics, world_model, failure_diagnostics, step) → ControlState: PRE: assert world_model.generation_id == step (raises StalenessError if mismatch). Tier 1: scan world_model.contradictions[] for any entry with severity == "SYSTEM_BREAKING". If found: set risk_state = "BLOCKED", escalation_reason = "SYSTEM_BREAKING_CONTRADICTION", append to block_mask. Return immediately — do not evaluate lower tiers.
4
Implement Tier 2: for each of the 10 sub-dimensions: normalise its current value; if normalised_value < CRITICAL_THRESHOLD (default 0.2): add a BlockEntry to block_mask. After collecting all blocked dimensions, call detect_deadlock(block_mask). If deadlock detected: set escalation_reason = "HUMAN_REQUIRED" and return with BLOCKED. If no deadlock but block_mask non-empty: return BLOCKED with each dimension's recovery_action_class in block_mask.
5
Implement Tier 3: if coverage dimensions are below CAUTION_THRESHOLD (default 0.4) but above CRITICAL_THRESHOLD: set risk_state = "CAUTIOUS" and add a note indicating that exploration actions (those not requiring blocked dimensions) are allowed. Tier 3 does not prevent all action — it biases selection toward evidence-gathering.
6
Implement Tier 4: proportional caution elevation. Compute the mean distance from CAUTION_THRESHOLD across all sub-dimensions that are between CAUTION_THRESHOLD and 1.0. Scale this distance to a [0,1] elevation factor. If the factor exceeds a small constant (0.05): set risk_state = "CAUTIOUS" if not already set. Also incorporate failure_diagnostics.matched_pattern.confidence (normalised via normalise_match_confidence()) as an advisory input to the elevation factor — blended at 0.2 weight.
7
Implement Tier 5: if none of the above tiers changed the risk_state from "NORMAL": return NORMAL with an empty block_mask and no escalation_reason. Set control_state.generation_id = world_model.generation_id to mark the control state as fresh.
8
Attach dep_class_gap_annotation: if diagnostics.dep_class_gap_annotation is not None: append it to control_state.notes[] after tier resolution completes. The annotation must not appear as a parameter or value in any tier 1–4 computation (INV-07 assertion point).
Acceptance tests
Tier 1 fires before any other tier when a SYSTEM_BREAKING contradiction exists in world_model.contradictions[] — lower tiers are not evaluated
Tier 3 conflict rule: when risk_state = CAUTIOUS due to coverage, actions not requiring any blocked dimension are allowed (block_mask does not suppress them)
Tier 4 caution elevation is proportional — a dimension at 0.35 (just below CAUTION_THRESHOLD 0.4) produces a lower elevation factor than a dimension at 0.05
Implementation note T11 tests the compute_elevation_factor() helper directly rather than through resolve_control_state(). A dim at 0.05 is below CRITICAL_THRESHOLD (0.2) so passing it to the full resolver would trigger Tier 2 BLOCKED before Tier 4 is ever reached. compute_elevation_factor() is therefore exposed as a public module-level function from control_state.py so this proportionality can be verified in isolation.
Tier 5 returns NORMAL with empty block_mask and no escalation_reason when all sub-dimensions are above their thresholds
P3.4 Deadlock detection ✓ Done
Implement deadlock detection as a sub-routine of Tier 2. A deadlock occurs when two (or more) entries in the block_mask have mutually blocking recovery actions — the recovery of each requires the other's blocked dimension to be resolved first. Implemented as directed cycle detection on a recovery-action dependency graph.
Implementation steps
1
Define RECOVERY_ACTION_DEPENDENCIES dict in control_state.py: maps each recovery_action_class to the set of dimension names it requires to be unblocked. Example: "dep_graph_refresh" requires {"dep_graph_quality"}; "verification_pass" requires {"verification_strength"}. This dict defines the recovery action graph.
⚠ Deviation — self-reference constraint The plan's example entries (e.g. dep_graph_refresh → {"dep_graph_quality"}) would make every single-dimension block appear as a deadlock: the blocked dimension's own recovery action would depend on itself, creating a self-loop that has_cycle() detects as a cycle. The actual implementation requires each recovery action to depend on a different dimension than the one it recovers. Final wiring: dep_graph_refresh → {"verification_strength"}, verification_pass → {"dep_graph_quality"} (mutual block), belief_refresh → {"verification_feasibility"}, etc. T14 (single block = no deadlock) passes only with this constraint enforced.
2
Implement build_recovery_action_graph(block_mask) → dict[str, set[str]]. For each BlockEntry in block_mask: find its recovery_action_class; look up the required dimensions in RECOVERY_ACTION_DEPENDENCIES; for each required dimension that is also blocked (i.e., in block_mask): add a directed edge from this block_entry's dimension to the required dimension. Returns the adjacency dict.
3
Implement has_cycle(graph: dict[str, set[str]]) → bool using iterative DFS with a visited set and a recursion stack set. Returns True if any cycle exists in the directed graph. This is the standard DFS-based cycle detection algorithm applied to the recovery-action dependency graph.
4
Implement detect_deadlock(block_mask) → bool: calls build_recovery_action_graph(block_mask) then has_cycle(graph). Returns True when a mutual block is detected. Called from Tier 2 after populating block_mask — if True, sets escalation_reason = "HUMAN_REQUIRED".
Acceptance tests
A block_mask with two entries whose recovery actions mutually require each other's dimension returns detect_deadlock() == True
A block_mask with a single blocked dimension returns detect_deadlock() == False — single block is not a deadlock
A three-way mutual block (A requires B, B requires C, C requires A) is correctly detected as a deadlock
P3.5 Generation-ID gates (full implementation) ✓ Done
Replace the P0.3 stubs in gates.py with full implementations. Each gate performs the staleness assertion (from P0.3), re-resolves control_state if stale, then executes its phase-specific logic. The action_gate reads the sub-step A generation_id value; the post_exec_gate reads the sub-step B value. This wires the double-increment model (INV-03) into the main loop for the first time.
Implementation steps
1
Implement decomposition_gate(task_graph, control_state, world_model, diagnostics) → bool in gates.py. Logic: (1) call @assert_generation_fresh decorator check — raises StalenessError if stale, which caller must handle by re-resolving; (2) if control_state.risk_state == "BLOCKED": return False (decomposition not allowed); (3) if CAUTIOUS: allow decomposition but add a note to the result; (4) if NORMAL: return True.
⚠ Deviation — staleness handled internally, not via decorator The @assert_generation_fresh decorator is not applied to the gate functions in the final implementation. Because the decorator raises before the function body runs, the gate cannot catch the error and re-resolve inside itself. Instead, a shared _maybe_resolve() helper is called at the top of each gate: it checks staleness, calls resolve_control_state() once if stale (when diagnostics is provided), and raises StalenessError if still stale or if diagnostics is absent. The decorator remains importable from gates.py and is tested independently in P0.
2
Implement stale recovery in all gates. When the @assert_generation_fresh decorator would raise StalenessError: instead of propagating, the gate catches it, calls resolve_control_state(diagnostics, world_model, failure_diagnostics, step=world_model.generation_id) to get a fresh control_state, updates the caller's control_state reference, and retries the freshness check. Only re-resolves once — if still stale after one resolution, raise.
3
Implement action_gate(action, control_state, world_model, diagnostics) → bool. Logic: (1) freshness check against sub-step A generation_id (before execution); (2) check control_state.block_mask — if action's required resources match any blocked dimension, return False; (3) if risk_state == CAUTIOUS and action is NOT an exploration-type: also return False; (4) otherwise return True.
4
Implement post_exec_gate(result, verification_result, control_state, world_model) → bool. Logic: (1) freshness check against sub-step B generation_id (after execution, incremented second time); (2) run contract_shadow_check(result, output_contract) — if fails, return False; (3) if verification_result.has_critical_failure: return False; (4) otherwise return True.
5
Wire the double-increment model. Add the two-substep increment sequence to a new adapter/harness/loop.py stub. Sub-step A: increment_generation_id(world_model) → resolve control_state → run action_gate. Sub-step B: execute action → increment_generation_id(world_model) → run post_exec_gate. This stub is the main loop skeleton that P5 will populate with full execution logic.
Acceptance tests

post_exec_gate checks the sub-step B generation_id (after second increment) — a control_state with the sub-step A value (one increment behind) is correctly identified as stale

Implementation summary
Phase 3 was implemented in a single pass against the plan. All 29 spec steps are complete. Three design decisions diverged from the literal spec text — each is recorded below with the reason and the chosen resolution. 20/20 acceptance tests pass. 73/73 P0–P2 tests remain green.
Deviation 1 — ControlStateStub not removed from staleness.py (P3.3 step 2)
What the plan said Remove ControlStateStub from staleness.py and replace it with a from .control_state import ControlState import. Existing P0 tests should pass because both classes share the generation_id field.

What was done instead ControlStateStub was kept in staleness.py as a separate backward-compat dataclass. P0 tests import it unchanged and still pass.

Why control_state.py must import StalenessError from staleness.py at module load time. If staleness.py also imported ControlState from control_state.py, Python would hit a circular import that cannot be resolved with standard from __future__ import annotations deferred evaluation alone (because the names need to be bound as runtime values, not just type annotations). Keeping ControlStateStub as a structurally-compatible class avoids the cycle entirely.

Impact on P4+ None. All new code imports ControlState directly from control_state.py. ControlStateStub is effectively retired for production use.
Deviation 2 — RECOVERY_ACTION_DEPENDENCIES uses cross-dimension dependencies (P3.4 step 1)
What the plan said Example entries: "dep_graph_refresh" → {"dep_graph_quality"}, "verification_pass" → {"verification_strength"}. Each action depends on the dimension it is recovering.

What was done instead Each recovery action was wired to depend on a different dimension: dep_graph_refresh → {"verification_strength"}, verification_pass → {"dep_graph_quality"}, belief_refresh → {"verification_feasibility"}, etc.

Why If an action depends on the same dimension it recovers, then any single blocked dimension creates a self-loop in the dependency graph: dim_A → action_A → requires dim_A. has_cycle() correctly detects this as a cycle, making detect_deadlock() return True for every single-dim block — breaking T14 ("single block is not a deadlock"). The fix is to ensure recovery actions require other dimensions, so a cycle only appears when two or more dimensions genuinely block each other's recovery path.

Impact on P4+ P4 or later phases that add new recovery actions must follow the same constraint: a recovery action for dimension X must not list X in its dependency set.
Deviation 3 — Gates use internal _maybe_resolve() instead of @assert_generation_fresh (P3.5 step 1–2)
What the plan said Apply @assert_generation_fresh to the gate functions. The decorator raises StalenessError when stale; the gate catches it, re-resolves, and retries.

What was done instead @assert_generation_fresh is not applied to the gate functions. Each gate calls a shared _maybe_resolve(control_state, world_model, diagnostics, failure_diagnostics) helper at the top of its body. The helper checks staleness, re-resolves once via resolve_control_state() when diagnostics is available, and raises StalenessError if still stale or if diagnostics is None.

Why The decorator wraps the outer function call and raises before the function body executes. A gate cannot catch an exception raised by its own wrapper — Python evaluates the decorator first and the body never runs. To re-resolve inside the gate body, the staleness check must happen inside the body, not in an outer decorator layer.

Impact on P4+ @assert_generation_fresh remains usable as a standalone decorator for custom functions (P0 tests verify this). Gate signatures now include optional diagnostics and failure_diagnostics kwargs — callers should supply diagnostics when automatic stale re-resolution is desired.
Additional design decision — compute_elevation_factor() exposed publicly
Context T11 requires verifying that a dimension at 0.05 produces a higher Tier 4 elevation factor than a dimension at 0.35. A dimension at 0.05 is below CRITICAL_THRESHOLD (0.2), so feeding it into resolve_control_state() triggers Tier 2 BLOCKED — Tier 4 is never reached. The test cannot verify Tier 4 proportionality through the full resolver in this scenario.

Decision compute_elevation_factor(sub_dims) is exposed as a public module-level function from control_state.py and exported from __init__.py. T11 calls it directly with synthetic dim values. resolve_control_state() calls it internally for its Tier 4 logic.
