itsharness Deploy — Client-Side Runtime

Phased implementation plan for two complementary TypeScript packages: (1) a FlowSpec runtime that executes a compiled canvas flow directly in the browser, paired with a thin authenticated LLM proxy; and (2) a Harness reasoning engine implementing the full 22-node agent reasoning/control architecture from the harness architecture spec. Together these give any React app both flow execution and the full agent loop — world model, hypothesis tracking, control-state resolution, verification, and reviewer pass.

What this plan delivers
The itsharness canvas authors a FlowSpec — a neutral JSON intermediate representation. Today that spec is compiled by a Python adapter into LangGraph, CrewAI, Mastra, or MAF, then executed server-side. This plan builds a second execution path: a pure TypeScript runtime that interprets the FlowSpec directly in the browser.

Track 1 (P0–P6) — FlowSpec Runtime: The client-side runtime handles the complete flow execution lifecycle — graph traversal, state management, all 14 FlowSpec node types, streaming, memory, tools, HITL pausing, and agent loops. The only server component is a ~100-line LLM proxy that holds API keys and forwards requests. No job queue, no Postgres, no Redis, no Python adapter at runtime. A useHarness(flowSpec) React hook wraps the runtime.

Track 2 (P7–P11) — Harness Reasoning Engine: A full TypeScript port of the 22-node agent reasoning/control architecture. This includes all state objects (WorldModel, CallerState, HypothesisSet, ControlState, Diagnostics, TaskGraph, EvidenceStore, MemoryState, StrategyState, FailureDiagnostics, ExperienceStore), all node implementations (gather_evidence, detect_contradictions, resolve_control_state, verify, reviewer_pass, etc.), and a HarnessRuntime class that runs the full agent loop. The FlowSpec runtime (Track 1) is usable as the execution layer inside harness tasks via the subgraph node pattern.
MVP scope (Phases 0–2 + 5, ~4 weeks): Proxy working on day 1. Graph engine + core node types (input, output, llm_call, transform, condition) by end of P1. Parallel fork/join + full flow control by end of P2. useHarness hook in P5. This covers ~80% of real-world flows built in the canvas — including the RAG Agent, Content Moderation+HITL (except HITL pause), and Parallel Risk Assessment examples in the spec.

Harness engine scope (Phases 7–11, ~6 weeks): Full TypeScript port of the 22-node reasoning architecture. P7 lays all state types. P8 adds world-model + evidence nodes. P9 adds control + planning nodes. P10 adds policy, execution, and recovery nodes. P11 closes the lifecycle (Initialize, WarmStart, ContextCompression, ReviewerPass, OutputValidation) and wires HarnessRuntime. ExperienceStore ships as a stub interface — production adapters are out of scope.

design invariants — must hold throughout all phases
FlowSpec is consumed read-onlyThe runtime never mutates the FlowSpec it receives. All mutable state lives in the FlowState object created at run time. This keeps the spec reusable across multiple parallel runs in the same app.
No server state at runtimeThe proxy is stateless. It holds no session, no run history, no flow metadata. All run state lives in the client. A proxy restart has zero effect on in-flight runs.
Node executors are pure functions of (node, state, context)Each executor takes a node definition, the current FlowState, and an ExecutionContext. It returns a new partial state update. No executor writes to global or module-level state.
Streaming is opt-in, not requiredEvery LLM call works in both streaming and non-streaming modes. If the host app doesn't subscribe to the event bus, execution proceeds normally and the final state is returned. Streaming adds no new failure modes.
HITL pause is a Promise, not a callbackThe HITLBreakpointExecutor suspends by holding an unresolved Promise. resume(payload) resolves it. This makes the execution model straightforward: the async/await chain is never broken, it simply waits.
Memory adapters are swappable without changing node executorsMemoryReadExecutor and MemoryWriteExecutor call through a MemoryAdapter interface. The in-memory, IndexedDB, and any future external adapters are interchangeable. The FlowSpec memory_stores config selects the adapter at init time.
The proxy is the only place API keys appearLLMClient sends requests to the proxy URL — never directly to Anthropic or OpenAI. API keys are env vars on the proxy only. A host app cannot exfiltrate keys even with full access to the runtime source.
Harness: generation_id must match before any gate reads control_stateAll three harness policy gates (decomposition, action, post_exec) assert control_state.generation_id === world_model.generation_id before reading. If stale, the gate re-invokes resolveControlState() via _maybeResolve() before proceeding. A stale control state is a structural bug, not a runtime condition — gates must never silently proceed with a stale value.
Harness: SYSTEM_BREAKING contradictions do not inline-haltdetectContradictions() never throws on a SYSTEM_BREAKING contradiction. It pushes the contradiction into world_model.contradictions[] and returns. The next resolveControlState() call picks it up via TIER 1. This ensures all system-breaking conditions produce the same response through the same code path regardless of where in the loop they are detected.
Process concepts: loadProcess() is idempotent (INV-PC-05)Calling loadProcess(conceptId, taskGraph, registry) a second time with the same conceptId must not add duplicate tasks. The function detects existing tasks whose IDs carry the concept namespace prefix ({conceptId}:{stepId}) and skips seeding, returning the current first step instead. Allows the agent to safely call loadProcess() again after a pause or context reload without corrupting the task graph.
Process concepts: getCurrentStep() returns null when the process is complete (INV-PC-06)getCurrentStep(taskGraph) delegates to selectUnblockedLeaf(). When all concept tasks are COMPLETE (or no PENDING task has all its dependencies COMPLETE), the function returns null. The agent treats a null return as the process-complete signal — no separate "isDone" predicate is needed.
Process concepts: completeStep() only touches the named task (INV-PC-07)completeStep(stepId, taskGraph) transitions exactly one task to COMPLETE. It does not automatically advance any other task's status. Dependency unlocking is emergent: on the next getCurrentStep() call, selectUnblockedLeaf() naturally returns any task whose dependsOn list is now fully COMPLETE. Status transitions are explicit and auditable.

Browser constraints on FlowSpec node types
transform fn_ref: Cannot use Node.js require() or file paths. fn_ref values must be registered in a FunctionRegistry passed to ExecutionContext at init. The proxy can optionally serve a fn_ref bundle endpoint, but this is optional — the common case is that transforms are registered by the host app.

memory_read semantic mode: Vector search cannot run in-browser (no WASM embedding model in MVP). Semantic memory_read in browser delegates to a /memory/search endpoint on the proxy, or falls back to a full in-memory linear scan. IndexedDB adapter supports key-value reads natively.

condition expr evaluation: Cannot use eval(). Conditions use a restricted expression parser (jsonpath-plus for $.state.* paths + comparison operators). Complex conditions requiring arbitrary code are fn_ref transforms instead.
All tests run without a live proxy, without a running LLM, and without a browser. LLM calls are mocked via an injectable LLMClient interface. Memory adapters use the in-memory implementation. Browser APIs (IndexedDB) are shimmed via vitest's jsdom environment.

HIGH
CORS on the proxy
Browser fetch to the proxy requires CORS headers. If the proxy is misconfigured, every LLM call fails silently in production. Mitigation: CORS headers are set in P0 and tested explicitly. Default config allows the configured host app origin only — not *. Document required env var (ALLOWED_ORIGIN) prominently.
HIGH
Streaming across CF Worker limits
Cloudflare Workers have a 30-second CPU limit and a response streaming buffer. Long LLM generations may hit this. Mitigation: use streaming pass-through (ReadableStream, not buffered) from P0. Proxy never accumulates the full response. Document CF Workers as the recommended deployment target specifically because it has no memory limit on streaming.
HIGH
fn_ref transform security
The transform node uses fn_ref to call registered functions. If the host app registers unsafe functions (e.g., eval wrappers), the runtime can be weaponised. Mitigation: FunctionRegistry has an explicit allowlist — functions must be registered by the host app at init time, not resolved dynamically from the FlowSpec string. The FlowSpec fn_ref value is a key into the registry, not a code path.
MEDIUM
HITL session persistence
If the user closes the browser while a flow is paused at a HITL breakpoint, the in-memory Promise is lost. Mitigation: P4 persists HITL pause state to IndexedDB before emitting the 'paused' event. On next mount, useHarness detects persisted HITL state and restores the paused status. Full recovery of in-flight execution is deferred to a future phase — for now, the run must be restarted.
MEDIUM
Parallel branch state merge conflicts
Two parallel branches writing to the same state key produce a conflict at join. The merge strategy (last-write-wins by default, append for array reducer) is documented but may surprise flow authors. Mitigation: P2 logs a console.warn for each conflict. P3 adds a per-key conflict_resolver option in FlowSpec. The canvas can surface conflict annotations visually in a future phase.
MEDIUM
agent_role max_iter not respected by LLM
If the LLM keeps emitting tool_calls, the agent loop could run significantly longer than expected before max_iter fires. Mitigation: max_iter is enforced as a hard counter before calling the LLM, not after. If max_iter is reached, the agent returns the best partial answer accumulated so far and emits a warning event.
LOW
IndexedDB unavailability
IndexedDB is unavailable in some environments (private browsing on Safari, WebWorkers without storage access). Mitigation: IndexedDBAdapter constructor checks availability and falls back to InMemoryAdapter with a console.warn. The fallback is transparent to the executor layer.
LOW
FlowSpec version drift
If the canvas upgrades FlowSpec to v1.1 and the runtime still targets v1.0, new node types or field shapes will cause silent failures or runtime errors. Mitigation: the runtime validates FlowSpec via the same Zod schema from @itsharness/canvas at init time. An unsupported spec_version throws immediately with a clear error message rather than failing mid-execution.
HIGH
generation_id staleness propagation bugs
The harness has two generation_id increments per iteration (sub-step A and sub-step B). If any gate reads the wrong sub-step value, control decisions are made on a stale world model — silently wrong, not a thrown error. Mitigation: GateContext type carries both sub-step A and sub-step B values explicitly; gates declare which value they assert against. _maybeResolve() re-resolves when stale (confirmed P3 pattern from Python implementation). Integration test asserts sub-step A and B generation_ids are distinct every iteration.
HIGH
Self-referential RECOVERY_ACTION_DEPENDENCIES causing false deadlock
If RECOVERY_ACTION_DEPENDENCIES are wired self-referentially (a dimension's recovery action depends on itself), every single-dimension block appears as a deadlock and HUMAN_REQUIRED fires for every minor diagnostic dip. The architecture requires cross-dimension dependencies — recovery action for dimension D must depend on a different dimension than D. Mitigation: _validate_recovery_action_dependencies() at HarnessRuntime init asserts no self-referential dependencies. A failing dependency validation throws at init time with a clear message identifying the conflicting pair.
MEDIUM
Observation vs conclusion invariant erosion
The most important invariant in the architecture is that a HIGH-reliability observation does not automatically become a HIGH-reliability conclusion. Under time pressure, node implementations may add beliefs to worldModel.beliefs[] without a derived_from[] chain, silently promoting observations to conclusions. Mitigation: addBelief() in WorldModel class validates that derived_from[] is non-empty before insertion, throwing if missing. Integration tests assert no belief is missing its derivation chain after a full loop run.
MEDIUM
dep_class_gap_annotation leaking into numeric control
dep_class_gap_annotation is advisory only — it must never become a numeric input to any resolve_control_state() tier. If a future change passes it to TIER 3 or TIER 4 arithmetic, uncontrolled noise is injected into risk_state. Mitigation: resolveControlState() accepts a Diagnostics object that intentionally does not expose dep_class_gap_annotation as a numeric field — it is typed as string (the annotation text) and assertNormalised() will throw if it is ever passed to the tier arithmetic accidentally.
