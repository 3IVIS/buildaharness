What Phase 11 delivers
Phases 0–10 built all the harness logic and visual tooling in framework-agnostic Python and TypeScript modules. Phase 11 wires everything together: it makes the four existing framework adapters harness-aware, ensures all harness behaviour is observable in Langfuse, validates the full system against eight end-to-end scenario tests and all ten architectural invariants, and benchmarks the overhead to verify it stays within the defined budget.

Framework adapter updates (P11.1) — The harness control layer (generation_id gates, diagnostics, resolve_control_state, verification, recovery, reviewer pass) runs in the adapter middleware layer, not in per-framework code. Each framework adapter needs to be updated to compile harness node types from FlowSpec and route their execution through the middleware. The key architectural constraint: no harness logic is duplicated in per-framework code. All four adapters call the same Python harness functions from adapter/harness/.

Langfuse harness tracing (P11.2) — The existing Langfuse integration records per-node token counts and HITL trace links. Phase 11 extends it with harness-specific spans: generation_id per iteration, control_state risk level per iteration, diagnostic health vector values, and recovery strategy changes. These spans are queryable from the Langfuse dashboard and provide the primary debugging surface for calibrating threshold constants in production.

E2E scenario tests (P11.3) — Implements the eight end-to-end scenarios defined in the Testing Strategy of the full implementation plan (E2E-01 through E2E-08). Each scenario is run on all four frameworks. The test gate for this phase is that all eight scenarios pass on all four frameworks before the phase is marked complete.

Invariant tests (P11.4) — Implements the ten architectural invariant tests (INV-01 through INV-10) as a standalone test file that runs against the full integrated system. These tests become a permanent CI gate: any PR that regresses an invariant is blocked. The test file uses black-box assertions where possible — testing observable behaviour rather than internal implementation details.

Performance benchmarks (P11.5) — Benchmarks harness loop overhead against a framework-only baseline. Target: <500ms added overhead per iteration. Profiles the top three contributors (expected: hypothesis generation, belief propagation, contradiction detection) and documents accepted trade-offs in a benchmark report committed to the repository.

Phase completion gates — hard requirements before Phase 11 closes
Gate 1 — All 4 adapters compile harness flows without error: Each of LangGraph, CrewAI, Mastra, and MAF must be able to compile a FlowSpec v1.0.0 flow with harness nodes enabled and execute it end-to-end. A framework that fails to compile a harness flow is a blocker.
Gate 2 — All 8 E2E scenarios pass on all 4 frameworks: E2E-01 through E2E-08 must each pass on LangGraph, CrewAI, Mastra, and MAF. A scenario that passes on 3 frameworks but not the 4th is a blocker. No partial credit.
Gate 3 — All 10 invariants pass on all 4 frameworks: INV-01 through INV-10 must pass when run against each framework. Invariant failures produce actionable error messages identifying which invariant was violated and at which step. An invariant failure is a blocker.
Performance target is a warning, not a blocker: The <500ms overhead target from P11.5 is aspirational. If the target is missed, the phase may still close — but the benchmark report must document which operations exceed budget, the measured overhead, and the accepted trade-off rationale. Closing Phase 11 with an unacknowledged performance regression is not acceptable.
End-to-end scenario reference
Eight scenarios are required for the E2E gate. Each must pass on all four frameworks.
E2E-01Happy path: Full 22-node harness loop completes, output passes validate_output_contract(), experience_store populated when available.
E2E-02BLOCKED escalation: Inject SYSTEM_BREAKING contradiction. Assert Tier 1 fires, BLOCKED state set, escalation surfaces to HITL, caller response restarts loop.
E2E-03Recovery cycle: Force verification failure three times on same task. Assert DIRECT_EDIT → TRACE_EXEC strategy progression. Assert global replan fires when replan_scope=GLOBAL.
E2E-04Warm start benefit: Run harness twice on same task class. Assert second run's strategy_weights differ from flat priors. Assert warm_start loaded prior decomposition patterns.
E2E-05Parallel branch merge: Run two parallel tasks with write-domain overlap. Assert optimistic path fires when conflict_prob ≤ PESSIMISTIC_THRESHOLD. Assert contradiction detection runs on merged world model at join.
E2E-06Context compression: Force context_pressure.critical. Assert compress_memory() populates both compressed_structures[] and pruned_regions[]. Assert action_dep_overlap() fires correctly when HIGH-risk action depends on a pruned region.
E2E-07Reviewer pass re-entry: Seed reviewer with findings that reopen a completed task. Assert drain_propagation_queue() triggers loop re-entry and task moves from COMPLETE to ACTIVE.
E2E-08max_steps budget: Set max_steps=5, run a task requiring 7 steps. Assert warning fires at step 4 (0.8×5), escalation fires at step 5 with "budget exhausted" reason.

P11.5 (benchmarks) runs last, after all tests pass, to avoid benchmarking broken code paths.
P11.1 Framework adapter updates for harness features Weeks 1–2 · ~4 days
Updates all four framework adapters to compile and execute harness-capable FlowSpec v1.0.0 flows. The harness control layer (all harness modules from adapter/harness/) runs in the adapter middleware — not duplicated per framework. Each adapter adds a harness execution path that routes harness node types through the shared Python middleware while leaving the existing non-harness execution path unchanged.
Implementation steps
1
Audit each framework adapter for harness integration points. For each of adapter/langgraph_adapter.py, adapter/crewai_adapter.py, adapter/maf_adapter.py, and the Mastra runner: identify where nodes are compiled and where the execution loop runs. Map each harness node type to its compiler function in adapter/harness/node_compilers.py. Document which parts of each adapter need a harness-aware code path and which can remain unchanged. This audit drives the implementation order for steps 2–4.
2
Update adapter/langgraph_adapter.py and adapter/crewai_adapter.py. For each adapter: add a harness compilation path that checks flow.harness_meta.enabled. If true: compile harness node types via the dispatch table in node_compilers.py; wrap the execution loop with the harness middleware (initialise HarnessRunState, wire generation_id gates, route control_state checks through resolve_control_state()). If false: use the existing compilation path unchanged. The non-harness path must not be modified in any way that changes its existing behaviour.
3
Update adapter/maf_adapter.py and the Mastra runner. Apply the same harness compilation path from step 2. For the Mastra runner (TypeScript): the harness middleware calls are made via the Python adapter bridge already in place — Mastra nodes compile to TypeScript execution stubs that call back into the Python harness layer for all control decisions. Confirm the bridge correctly passes HarnessRunState across the language boundary via the existing JSON serialisation.
4
Add per-framework harness integration test files. Create adapter/tests/test_harness_integration_LG.py, _CR.py, _MA.py, _MAF.py. Each file runs a minimal harness flow (world_model + gather_evidence + update_world_model + control_state) on its framework and asserts: (a) the flow compiles without error, (b) HarnessRunState is populated after the run, (c) the generation_id has incremented at least twice (double-increment invariant), (d) no harness logic is executed from within per-framework code (verified by checking that harness module imports appear only in the adapter middleware, not in framework-specific files).
5
Confirm the non-harness execution path is unaffected. Run the full existing test suite (pytest adapter/tests/ -v -k "not harness") after each adapter update. All 470+ existing tests must continue to pass. A regression in a non-harness test caused by a harness code path change is a blocker — the harness and non-harness paths must be isolated from each other at the adapter level.
Acceptance tests
All four framework adapters compile a FlowSpec v1.0.0 flow with harness_meta.enabled=true and three harness node types without error; the compiled output references only adapter/harness/ functions, not framework-specific harness implementations
After executing a minimal harness flow on each framework: HarnessRunState is populated with a non-zero generation_id and at least one observation in world_model — confirming the middleware wiring is active end-to-end
All 470+ pre-existing non-harness adapter tests pass without modification after the adapter updates — the non-harness execution path is unaffected by the harness code path additions
P11.2 Langfuse harness tracing Week 2 · ~2 days
Extends the existing Langfuse integration (which already records per-run traces, per-node token counts, and HITL trace links for all four frameworks) with harness-specific spans. After Phase 11, every harness run produces a queryable trace showing generation_id progression, control_state risk level, diagnostic health vectors, and any recovery strategy changes — the primary observability surface for tuning harness behaviour in production.
Implementation steps
1
Create adapter/harness/langfuse_tracing.py. Define a HarnessTraceContext dataclass: run_id: str, langfuse_trace_id: str, iteration: int. Implement start_harness_span(ctx, span_name) → LangfuseSpan and end_harness_span(span, attributes: dict) → None. These wrap the existing Langfuse span API with harness-specific attribute conventions. If Langfuse is not configured: all functions are no-ops — harness tracing is optional instrumentation, not a required dependency.
2
Instrument the main harness loop in adapter/harness/loop.py. At the start of each iteration: open a span named "harness_iteration_{n}" with attribute generation_id: world_model.generation_id. At the end of each iteration: close the span with attributes control_state_risk: control_state.risk_state and iteration_duration_ms: elapsed. The span wraps the entire iteration from generation_id pre-increment to post-exec gate.
3
Add diagnostic health vector attributes to each iteration span. Before closing the iteration span: attach the four health vector summaries as span attributes — belief_health_freshness, belief_health_consistency, belief_health_support, coverage_symptom, coverage_explanation, verification_strength, verification_feasibility, execution_progress_rate, execution_failure_recurrence, execution_oscillation. These attributes are queryable from the Langfuse dashboard to build health trend charts across a run.
4
Emit recovery strategy change events. In adapter/harness/recovery.py: whenever switch_strategy() is called, emit a Langfuse event (not a span) with attributes from_strategy, to_strategy, trigger_reason, failure_class, iteration. These events are distinct from iteration spans — they appear as point events on the trace timeline, making strategy progressions visible at a glance. Also emit an event when escalate() is called, with the escalation reason and surface_blocker summary.
Acceptance tests
After running a minimal harness flow on LangGraph with Langfuse configured: each loop iteration produces a Langfuse span named "harness_iteration_N" with the correct generation_id attribute — verified by querying the Langfuse test instance
A run that triggers a strategy switch from DIRECT_EDIT to TRACE_EXEC emits a Langfuse event with from_strategy="DIRECT_EDIT", to_strategy="TRACE_EXEC", and the trigger reason — event appears on the trace timeline at the correct iteration
When Langfuse is not configured (no LANGFUSE_SECRET_KEY in environment): all harness tracing calls are no-ops and the harness run completes successfully — no exception raised, no missing-attribute error
P11.3 End-to-end harness scenario tests (E2E-01 through E2E-08) Weeks 3–4 · ~6 days
Implements all eight E2E scenario tests from the Testing Strategy in the full implementation plan. Each scenario must pass on all four frameworks (LangGraph, CrewAI, Mastra, MAF). The test suite is parameterised by framework so that adding a new framework in the future requires only a single registration. These tests run against a real Postgres instance and real adapters — they are not unit tests.
Implementation steps
1
Create adapter/tests/test_harness_e2e.py with a parameterised framework fixture. The fixture yields a running adapter client for each of the four frameworks. All eight E2E test functions are decorated with @pytest.mark.parametrize("framework", ["langgraph","crewai","mastra","maf"]). A shared helper run_harness_scenario(framework, spec, initial_state) → HarnessRunState executes a given FlowSpec on the specified framework and returns the final HarnessRunState.
2
Implement E2E-01 (happy path) and E2E-02 (BLOCKED escalation). E2E-01: build a minimal 22-node harness flow spec, run it, assert validate_output_contract() passes and experience_store has at least one entry (when available). E2E-02: build a flow that injects a SYSTEM_BREAKING contradiction at iteration 2. Assert: contradiction enters world_model.contradictions[] without raising inline; Tier 1 of resolve_control_state() fires on the next call; control_state.risk_state == "BLOCKED"; escalation surfaces with a non-empty surface_blocker reason.
3
Implement E2E-03 (recovery cycle), E2E-04 (warm start), and E2E-05 (parallel branch merge). E2E-03: force verification failure on the same task three times; assert strategy transitions DIRECT_EDIT → TRACE_EXEC and that a GLOBAL replan fires when scope is GLOBAL. E2E-04: run the harness twice on the same task class; assert that on the second run, strategy_state.prior_strategy_weights differ from flat priors. E2E-05: configure two parallel tasks with overlapping write domains; assert that when conflict_prob ≤ PESSIMISTIC_THRESHOLD the optimistic path is taken; assert detect_contradictions() runs on the merged world model at the parallel join.
4
Implement E2E-06 (context compression), E2E-07 (reviewer re-entry), and E2E-08 (max_steps budget). E2E-06: force context_pressure.critical; assert compress_memory() produces non-empty compressed_structures[] and pruned_regions[]; assert action_dep_overlap() returns non-empty when a HIGH-risk action depends on a pruned region. E2E-07: seed the reviewer with a finding that invalidates a COMPLETE task's evidence; assert drain_propagation_queue() transitions the task to PENDING and the loop re-enters. E2E-08: set max_steps=5; assert a warning fires at step 4 and escalation fires at step 5 with "budget exhausted" as the escalation_reason.
5
Run the full E2E suite on all four frameworks and resolve blockers. Run pytest adapter/tests/test_harness_e2e.py -v and address any failures. Framework-specific failures that reveal a missing harness wiring in an adapter must be fixed in the adapter before the suite can pass. Document any framework-specific behaviour differences (e.g. timing-related differences in parallel execution) as test comments. All 32 parameterised test cases (8 scenarios × 4 frameworks) must pass for the E2E gate to clear.
Acceptance tests
E2E-01 passes on all four frameworks: harness loop completes, output satisfies validate_output_contract(), and the run does not raise any exception — confirmed by 4 parameterised test runs each returning a passing HarnessRunState
E2E-02 passes on all four frameworks: SYSTEM_BREAKING contradiction enters contradictions[] without raising inline, BLOCKED state is set on the subsequent resolve_control_state() call (not the current one), and escalation fires with a non-empty surface_blocker reason
E2E-03 through E2E-08 pass on all four frameworks: all 24 remaining parameterised test cases return green — the full E2E gate clears and Phase 11 may proceed to the invariant tests
P11.4 All 10 invariant tests pass Weeks 4–5 · ~3 days
Implements the ten architectural invariant tests as a standalone test file that runs against the full integrated system. Each invariant test uses the real harness modules — not mocks. The tests become a permanent CI gate: merging a PR that regresses any invariant is blocked. Invariant failures produce error messages that identify the specific invariant and which condition was violated.
Implementation steps
1
Create adapter/tests/test_harness_invariants.py. Structure: one test function per invariant, named test_inv_01_observation_conclusion_separation through test_inv_10_experience_store_noop. Each test constructs the minimal harness state required to exercise the invariant, runs the relevant harness operation, and asserts the invariant condition. Tests use in-memory fixtures where possible; tests that require integration (INV-03 double-increment, INV-04 deadlock) use the full loop infrastructure. Parameterise by framework where relevant (INV-03 and INV-04 run on all four).
2
Implement INV-01 through INV-05. INV-01: inject a HIGH-reliability tool observation; assert it appears in observations[] not beliefs[]; assert no belief is derived from it without an explicit derived_from[] chain. INV-02: inject raw values at boundary conditions (0, 1, beyond-range) into each of the 10 sub-dimensions; assert all outputs are clamped [0,1]; assert all TIER 4 arithmetic inputs are normalised. INV-03: run one full loop iteration; assert generation_id incremented exactly twice. INV-04: construct a block_mask where two entries mutually block recovery; assert escalation_reason == "HUMAN_REQUIRED". INV-05: inject a SYSTEM_BREAKING contradiction at three different loop positions; assert it enters contradictions[] each time and BLOCKED is set only on the next resolve_control_state() call.
3
Implement INV-06 through INV-10. INV-06: instrument select_best_action(); assert no write to control_state occurs from world_model, hypothesis_set, or task_graph; assert block_mask is the only suppression mechanism. INV-07: assert dep_class_gap_annotation does not appear as a parameter in any tier 1–4 computation inside resolve_control_state(); assert it is attached to control_state.notes[] only. INV-08: assert failure_mode_library.match() return value enters only Tier 4 and hypothesis generation sources — no write path to block_mask or escalation_reason. INV-09: run reviewer_pass(); assert no field of HarnessRunState, world_model, or hypothesis_set holds an AdversarialPrior instance after the call returns. INV-10: run the full harness loop with experience_store.available=False; assert the outcome is structurally identical to a run with it present; assert warm_start() returns immediately.
4
Add invariant tests to the CI gate. In the CI configuration: add pytest adapter/tests/test_harness_invariants.py -v --tb=short as a required check on every PR. Configure the check to emit a structured failure message that names the violated invariant: "INV-03 FAILED: generation_id incremented {actual} times, expected exactly 2". This actionable output ensures developers can identify the root cause without needing to read the invariant specification from scratch.
Acceptance tests
All ten invariant tests (INV-01 through INV-10) pass when run against the full integrated system: pytest adapter/tests/test_harness_invariants.py -v exits 0 with no failures or errors
Deliberately breaking INV-03 (changing the loop to increment generation_id only once) causes the invariant test to fail with an error message containing "INV-03" and the actual vs expected increment count — confirming the tests detect regressions
The invariant test CI check runs on a sample PR; a PR that would pass all existing unit tests but regress INV-05 (SYSTEM_BREAKING via Tier 1) is correctly blocked by the CI gate
P11.5 Performance benchmarks Week 5 · ~2 days
Benchmarks the full harness loop overhead against a framework-only baseline. Target: <500ms added overhead per iteration above the baseline (framework-only, no harness). Profiles the top three contributors and documents accepted trade-offs. A benchmark report is committed to the repository as docs/harness_benchmark_report.md.
Implementation steps
1
Create adapter/tests/benchmark_harness.py. Implements two benchmark scenarios: (a) framework-only baseline — run a FlowSpec v1.0.0 flow with harness_meta.enabled=false and measure iterations per second; (b) harness-enabled — run the same flow structure with harness_meta.enabled=true and all harness nodes active. The difference is the "harness overhead." Run each scenario 50 times and report mean, median, p95, and p99 latencies. Use pytest-benchmark or Python's timeit module (whichever is already in the project).
2
Benchmark individual harness operations. Profile: (a) generate_hypotheses() — four-source generation for a representative world model with 10 beliefs; (b) propagate_beliefs() — full propagation pass for a belief dependency graph with 100 beliefs and 150 edges; (c) detect_contradictions() — pairwise detection pass for 20 beliefs; (d) resolve_control_state() — full five-tier resolution for a representative diagnostics state. Report mean latency for each. These isolate bottlenecks from the full-loop benchmark.
3
Identify and optimise the top-3 bottlenecks. Sort the individual operation benchmarks by mean latency. For the top three: (a) identify the specific code path causing the overhead using Python's cProfile or line_profiler; (b) implement the optimization if it does not increase code complexity significantly; (c) if the optimization is not straightforward, document the bottleneck and accepted trade-off rather than introducing a complex fix. The bar for optimization is: a simple change that reduces latency by ≥ 30% without adding new abstractions.
4
Write and commit the benchmark report. Create docs/harness_benchmark_report.md. Sections: (1) Benchmark methodology and environment (CPU, RAM, Python version, framework versions); (2) Full-loop overhead table (baseline vs harness-enabled, all four frameworks); (3) Individual operation latencies table; (4) Top-3 bottleneck analysis with profiler output excerpts; (5) Optimizations applied and their impact; (6) Accepted trade-offs for operations that miss the target with rationale. The report is a permanent repository artefact — it becomes the baseline for future performance regression tracking.
Acceptance tests
Full harness loop overhead is <500ms per iteration above the framework-only baseline on the benchmark task (reported as the mean across 50 runs on LangGraph); if the target is missed, the benchmark report documents the actual overhead and the accepted trade-off rationale
Individual operation benchmarks: generate_hypotheses() completes in <200ms for a 10-belief world model; propagate_beliefs() completes in <100ms for 100 beliefs / 150 edges — both targets verified on the benchmark hardware
docs/harness_benchmark_report.md exists in the repository and contains all four required sections (methodology, full-loop table, operation latencies, bottleneck analysis with trade-offs); the report is committed and reviewable before Phase 11 is closed

All 10 architectural invariants verified in Phase 11
Phase 11 is the first phase where all ten invariants are tested against the full integrated system simultaneously. Earlier phases tested each invariant in isolation as each piece was built. Phase 11 verifies they all hold together in a running system, on each framework, with real data flowing through the full pipeline.
INV-01 — Observation vs conclusion separation P11.4 full integrated test Injecting a HIGH-reliability tool observation into the running system must not produce a belief without an explicit derived_from[] chain. Tested end-to-end: the observation goes through the adapter, through the evidence store, through update_world_model(), and the resulting world_model.beliefs[] must not contain the observation directly — only beliefs derived from it via the explicit integration step.
INV-02 — Normalisation contract P11.4 full integrated test Every diagnostic sub-dimension entering resolve_control_state() must be normalised [0,1]. The integrated test injects raw values outside [0,1] into each sub-dimension source and asserts the normalised value at the Tier 4 arithmetic boundary is clamped. Also asserts that all 10 sub-dimensions produce valid [0,1] outputs when the harness runs on a representative task.
INV-03 — generation_id double-increment per iteration P11.4 all four frameworks One full harness loop iteration must increment world_model.generation_id exactly twice: once after sub-step A (pre-execution) and once after sub-step B (post-execution). This invariant is tested on all four frameworks because the double-increment depends on the middleware wiring introduced in P11.1. A framework that increments only once is a bug in the adapter wiring.
