What Phase 2 delivers
Phase 2 completes the World Model layer by adding everything that makes the world model dynamic and self-correcting. The P0 data structures defined the shape of the world model; Phase 2 implements the processes that operate on it over time.

The belief dependency graph tracks causal links between beliefs — each edge carries its own confidence score that decays independently of content changes via dep_graph_budget. propagate_beliefs() forwards updates through these edges with confidence weighting, widening the invalidation frontier when the budget is breached. The contradiction detection system operates at four structural levels — pairwise, set-level, temporal, and abstraction — and assigns severity levels from LOW through SYSTEM_BREAKING. The resolution policy routes each contradiction to the appropriate response without ever raising an inline exception for SYSTEM_BREAKING contradictions (INV-05). The staleness sweep periodically invalidates beliefs using two independent mechanisms: TTL-based expiry and environment-change-based invalidation from environment_change_log.

Phase 2 also introduces the update_world_model canvas node type — the mechanism by which flow authors integrate Evidence from the evidence store into the world model in a reliability-weighted way.
Implementation notes & deviations
Belief.source not in data model: Belief has no source field (by P0 design). Temporal contradiction detection and the staleness sweep both match environment_change_log.affected_source against belief.derived_from[] and belief.supporting_evidence[] instead. This is fully consistent with the derivation-chain approach and passes T12 and T17.
T16–T18 run without Postgres: The plan stated T16–T18 required Postgres for HarnessRunState round-trip. In practice all three test staleness and edge decay in-memory only — no DB needed. All 18 tests pass with pytest adapter/tests/test_harness_p2.py -v and zero infrastructure.
state_store.py upgraded: HarnessRunState.belief_dep_graph was a raw dict[str, Any] placeholder. It is now a typed BeliefDepGraph with full to_dict()/from_dict() round-trip wired into save()/load().
Resolution policy idempotency mechanism: Each Belief gains a transient applied_contradiction_ids: set[str] attribute at resolution time. Before modifying confidence, the resolver checks this set and skips if the contradiction ID is already present — satisfying T15 without modifying the core Belief dataclass.

Key constraints to respect in Phase 2
SYSTEM_BREAKING contradictions must never cause inline halts (INV-05): detect_contradictions() calls world_model.add_contradiction() — it never raises. The BLOCKED state that results from a SYSTEM_BREAKING contradiction is set only on the next resolve_control_state() call (in P3). Phase 2 must enforce this at the type level: contradiction detection has no return path that triggers execution termination.
Observation / belief separation remains inviolate (INV-01): The update_world_model node (P2.1) integrates Evidence into observations[] using add_observation(). Beliefs are only created via explicit inference with non-empty derived_from[] chains. The node compiler must never call add_belief() with raw evidence as the sole derived_from entry.
dep_class_gap_annotation is advisory only (INV-07): The annotation may be computed and stored in Phase 2 as a string note on the world model, but it must not be represented as a numeric field, must not feed into any arithmetic, and must not influence any control decision. It surfaces only to the adversarial reviewer lens (P9) and human escalation.

P2.2 Belief dependency graph ✓ DONE Week 1 · ~2 days
Implement the belief dependency graph data structures — the directed graph that tracks which beliefs are derived from which other beliefs, with confidence-weighted edges. Also implement the DepGraphBudget that controls when the graph widens its invalidation frontier and how quickly edge confidence decays.
Implementation steps
1
Create adapter/harness/belief_graph.py. Define BeliefEdge dataclass: from_id: str, to_id: str, confidence: float (starts at the source belief's confidence; decays over time), last_verified: datetime. Edges have their own confidence independent of the beliefs they connect — a belief can be high-confidence while the link to it is uncertain.
2
Define DepGraphBudget dataclass: max_unverified_edge_ratio: float = 0.3 (if more than 30% of edges are unverified, widen the frontier); confidence_decay_rate: float = 0.02 (edge confidence decreases by this amount per iteration); refresh_policy: Literal["lazy","eager"] = "lazy" (lazy = only decay on access; eager = decay all edges each iteration).
3
Define BeliefDepGraph dataclass: belief_nodes: dict[str, str] (belief_id → statement summary), edges: list[BeliefEdge], invalidation_frontier: set[str] (belief IDs currently flagged for re-verification), propagation_queue: list[str] (belief IDs awaiting propagation). Implement add_edge(from_id, to_id, confidence), get_downstream(belief_id) → list[str] (returns all belief IDs reachable from this one), and compute_unverified_edge_ratio() → float.
4
Implement apply_decay(graph, budget) → None in belief_graph.py. Iterates over all edges and subtracts budget.confidence_decay_rate from each edge's confidence, clamping at 0.0. An edge with confidence 0.0 is flagged as unverified (added to invalidation_frontier). This decay runs independently of content changes — an edge can become unreliable even if both connected beliefs are still valid.
5
Add to_dict() / from_dict() to BeliefDepGraph and DepGraphBudget. Extend HarnessRunState in state_store.py with belief_dep_graph: BeliefDepGraph and update save() / load() accordingly.
Acceptance tests
Calling add_edge("A", "B", confidence=0.8) followed by get_downstream("A") returns ["B"]
apply_decay() called 5 times with confidence_decay_rate=0.1 on an edge starting at confidence 0.5 produces an edge with confidence 0.0 (clamped, not negative)
compute_unverified_edge_ratio() returns 0.5 when 1 of 2 edges has confidence 0.0
P2.3 propagate_beliefs() ✓ DONE Week 2 · ~3 days
Implement belief propagation — the process that forwards belief updates through the dependency graph with confidence weighting. Low-confidence edges propagate updates with proportionally lower weight. When the budget is breached, the invalidation frontier widens to include all downstream beliefs. The dep_graph_quality diagnostic is recomputed after each propagation pass.
Implementation steps
1
Implement propagate_single_update(graph, belief_id, updated_confidence, budget) → list[str] in belief_graph.py. For the given belief_id: find all outgoing edges. For each downstream belief: new_confidence = edge.confidence × updated_confidence. If new_confidence differs from the downstream belief's current confidence by more than 0.05 (threshold to avoid infinite micro-updates): add the downstream belief_id to propagation_queue. Return the list of queued belief IDs.
2
Implement propagate_beliefs(graph, budget, world_model) → None. Process graph.propagation_queue in FIFO order until empty or a safety limit (100 iterations) is reached. For each dequeued belief_id: look up the belief in world_model.beliefs and call propagate_single_update() with its current confidence. Remove processed entries from the queue after propagation.
3
Implement budget-breach handling. After each propagation pass, call compute_unverified_edge_ratio(). If ratio exceeds budget.max_unverified_edge_ratio: widen the invalidation frontier — add all beliefs reachable from any belief already in the frontier to graph.invalidation_frontier. This signals to the diagnostics layer (P3) that a large portion of the belief graph is uncertain.
4
Implement compute_dep_graph_quality(graph, rolling_prediction_accuracy) → float. Quality = weighted average of two factors: (1 - unverified_edge_ratio) × 0.6 + rolling_prediction_accuracy × 0.4. Result is normalised to [0,1]. Store on graph as graph.dep_graph_quality. This scalar feeds into Tier 2 of resolve_control_state() in P3 — it is normalised here so no further normalisation is needed upstream.
5
Update __init__.py to export BeliefDepGraph, DepGraphBudget, propagate_beliefs, and compute_dep_graph_quality from the harness package.
Acceptance tests
A 0.3-confidence edge propagates a belief update at 0.3× weight — if the source belief has confidence 0.9, the downstream belief receives propagated confidence of 0.27
Budget breach (unverified_edge_ratio > max) widens the invalidation_frontier to include all downstream beliefs of all frontier members
compute_dep_graph_quality() returns a value in [0,1] and decreases when unverified_edge_ratio increases
P2.4 Typed contradiction detection ✓ DONE Week 3 · ~3 days
Implement detect_contradictions() — the function that scans the world model, evidence store, and hypothesis set for inconsistencies. Four contradiction types are detected independently. SYSTEM_BREAKING contradictions are always appended to world_model.contradictions[] and never cause inline execution halts — this is the central invariant (INV-05).
Implementation steps
1
Create adapter/harness/contradiction.py. Import WorldModel, Contradiction from world_model.py; EvidenceStore from evidence.py; HypothesisSet from hypothesis.py.
2
Implement detect_pairwise_contradictions(beliefs) → list[Contradiction]. For each pair of beliefs: check if the statements are semantically opposed using keyword negation (simple: "X is present" vs "X is absent", "X is true" vs "X is false"). If opposed and both have confidence ≥ 0.5: create a Contradiction with type="pairwise". Severity assignment: both HIGH confidence → HIGH; at least one MEDIUM → MEDIUM; otherwise LOW.
3
Implement detect_set_level_contradictions(beliefs) → list[Contradiction]. Check if any group of three or more beliefs are jointly inconsistent (if A implies ¬B, and B implies ¬C, but C is asserted alongside A). Uses a simple inference chain check. Creates Contradictions with type="set-level" and severity="HIGH" when all involved beliefs have confidence ≥ 0.6.
4
Implement detect_temporal_contradictions(beliefs, environment_change_log) → list[Contradiction]. For each belief: check if a newer entry in environment_change_log invalidates the belief's statement (e.g., a file the belief was derived from has since been deleted or modified). Creates Contradictions with type="temporal". Severity: MEDIUM if the change is recent; HIGH if the change pre-dates the belief's recorded_at.
5
Implement detect_abstraction_contradictions(beliefs, task_graph) → list[Contradiction]. Checks for beliefs stated at a finer granularity than the current task's abstraction level (e.g., a belief about line 47 of a file when the current task is at module level). Creates Contradictions with type="abstraction" and severity LOW (these are typically style issues, not logical errors).
6
Implement assign_system_breaking_severity(contradictions, hypothesis_set) → list[Contradiction]. Upgrades a pairwise or set-level contradiction to SYSTEM_BREAKING if: (a) both beliefs have HIGH confidence AND HIGH reliability supporting evidence, AND (b) the contradiction is also present in an active hypothesis as a predicted observation conflict. SYSTEM_BREAKING contradictions invalidate the entire current execution path.
7
Implement detect_contradictions(world_model, evidence_store, hypothesis_set, task_graph=None) → list[Contradiction]. Orchestrator: calls all four detection functions, collects results, upgrades severities via assign_system_breaking_severity(), then calls world_model.add_contradiction(c) for each. Returns the full list. Does not raise or halt — SYSTEM_BREAKING contradictions simply enter the list for Tier 1 pickup (INV-05).
Acceptance tests
A SYSTEM_BREAKING contradiction is appended to world_model.contradictions[] via add_contradiction() without any exception being raised anywhere in the call stack
Pairwise and set-level contradictions are detected independently — a set-level contradiction is not reported as a pairwise one
Contradiction scope correctly maps to replan breadth: a contradiction with scope="global" is flagged for global replan (this scope field is set by the resolution policy in P2.5)
P2.5 Resolution policy ✓ DONE Week 3 · ~1.5 days
Implement apply_resolution_policy() — the routing function that decides what to do with each detected contradiction based on its severity and scope. Resolution actions range from confidence updates (LOW) through task graph flagging (HIGH) to entering the contradiction queue for Tier 1 pickup (SYSTEM_BREAKING). The function is idempotent: applying it twice has the same effect as applying it once.
Implementation steps
1
Implement apply_resolution_policy(contradiction, world_model, task_graph=None) → None in contradiction.py. Route by severity: LOW → call _resolve_low(contradiction, world_model); MEDIUM → call _resolve_medium(contradiction, world_model); HIGH → call _resolve_high(contradiction, world_model, task_graph); SYSTEM_BREAKING → call _resolve_system_breaking(contradiction, world_model).
2
Implement _resolve_low(): reduce the confidence of each involved belief by 10% (multiply by 0.9, clamp at 0.0). Mark the belief with a pending_sweep: True attribute so the next staleness sweep includes it in its re-evaluation pass. Does not touch the task graph.
3
Implement _resolve_medium(): reduce involved belief confidence by 25% (multiply by 0.75). Add the belief IDs to belief_dep_graph.propagation_queue so downstream beliefs are also updated via propagation. Mark beliefs for staleness sweep.
4
Implement _resolve_high(): for each belief ID in contradiction.involved_belief_ids, find all tasks in task_graph that list these beliefs as dependencies. Set those tasks' status to BLOCKED with block_reason="high_contradiction". Also add the belief IDs to the invalidation frontier.
5
Implement _resolve_system_breaking(): ensure the contradiction is in world_model.contradictions[] (it should already be there from detection — this is a safety re-check). Set contradiction.scope = "global" if not already set. No other action — Tier 1 of resolve_control_state() (P3) handles the rest.
6
Ensure idempotency: before modifying a belief's confidence, check if the same contradiction ID has already been applied (store applied IDs in a set on each belief as applied_contradiction_ids). Skip if already applied. This prevents double-penalising beliefs when apply_resolution_policy() is called more than once per loop iteration.
Acceptance tests
Resolution policy correctly routes each severity level — LOW reduces confidence by 10%, HIGH blocks dependent tasks, SYSTEM_BREAKING does not touch the task graph
A HIGH-severity contradiction flags all downstream task_graph tasks that depend on the involved beliefs as BLOCKED
Applying apply_resolution_policy() twice with the same contradiction produces identical final state to applying it once (idempotency)
P2.6 Staleness sweep ✓ DONE Week 4 · ~2 days
Implement staleness_sweep() — the periodic process that invalidates beliefs using two independent mechanisms: TTL-based expiry and environment-change-based invalidation. Also integrate the edge confidence decay (apply_decay()) which runs on the same schedule but operates on the belief dependency graph independently of belief content.
Implementation steps
1
Extend adapter/harness/staleness.py (extending the P0.3 file). Add a stale_flags: dict[str, bool] field to WorldModel (or as a separate tracked dict, if modifying the dataclass is cleaner). Each key is a belief_id; value is True when the belief is considered stale.
2
Implement TTL-based staleness check. Add belief_ttl: timedelta = timedelta(minutes=30) as a configurable constant. In staleness_sweep(world_model, environment_change_log, belief_ttl): for each belief in world_model.beliefs, check if datetime.utcnow() - belief.recorded_at > belief_ttl. If so, set stale_flags[belief.id] = True.
3
Implement environment-change-based invalidation. For each entry in environment_change_log: extract the affected_source field. For each belief where belief.source == affected_source AND the change log entry's timestamp is newer than belief.recorded_at: set stale_flags[belief.id] = True.
4
Compute and store stale_flag_ratio. After processing: stale_flag_ratio = len([k for k,v in stale_flags.items() if v]) / max(1, len(world_model.beliefs)). Store on the world model as a transient proxy field. This value feeds into belief_health.freshness recomputation in P2.1 and ultimately into Tier 4 of resolve_control_state() in P3.
5
Wire edge decay into the sweep. At the end of staleness_sweep(), call apply_decay(belief_dep_graph, dep_graph_budget). The edge decay runs on the same schedule as the staleness sweep but operates independently — a stale belief does not necessarily mean its outgoing edges are decayed, and vice versa.
Acceptance tests
A belief with recorded_at older than belief_ttl is correctly flagged as stale in stale_flags
A belief whose source file appears in environment_change_log with a newer timestamp than belief.recorded_at is invalidated
stale_flag_ratio correctly reflects the ratio of stale beliefs after the sweep completes — 0.0 when none are stale, 1.0 when all are stale

Architecture invariants engaged in Phase 2
Phase 2 delivers the first full test of INV-05 (SYSTEM_BREAKING routing) and enforces INV-01 at the integration level. INV-07 is seeded by the dep_class_gap_annotation design.
INV-05 — SYSTEM_BREAKING via Tier 1 only P2.4 full enforcement · P3 Tier 1 test Phase 2 establishes the non-negotiable rule: detect_contradictions() and apply_resolution_policy() never raise an exception or halt execution for any contradiction severity. SYSTEM_BREAKING contradictions enter world_model.contradictions[] via the same add_contradiction() path as LOW contradictions. The BLOCKED control state is set exclusively by Tier 1 of resolve_control_state() in P3. Phase 2 tests inject SYSTEM_BREAKING contradictions at three different positions in the flow and assert no exception is raised.
INV-01 — Observation vs conclusion separation P2.1 integration test The update_world_model node (P2.1) is the first place where Evidence objects flow into the world model at runtime. The compiler must generate code that calls add_observation() for all OBSERVATION and SYSTEM_ERROR evidence — never add_belief(). This is tested in P2: run the compiled node with a HIGH-reliability OBSERVATION evidence entry and assert the result appears in world_model.observations[], not in world_model.beliefs[].
INV-07 — dep_class_gap advisory only P2.1 seeds The dep_class_gap_annotation — when detected during world model integration — is stored as a string in world_model.assumptions[] or as a text note. It has no numeric representation. Phase 2 must not create a dep_class_gap numeric field on any dataclass, must not include it in any arithmetic, and must not pass it as an argument to any control-path function. The annotation surfaces only to the adversarial reviewer in P9.
Invariants that Phase 2 must not break
INV-03 — generation_id double-increment Phase 2 does not touch the generation_id increment path. None of the world model operations or contradiction detection functions call increment_generation_id(). The generation_id advances only in the main loop — not in sub-operations.
INV-06 — control_state as sole control input Contradiction detection and resolution policy produce no control signals. They update world model state and task graph BLOCKED flags — but no function in Phase 2 writes to a control_state structure or block_mask directly. Those writes are exclusively in P3's resolve_control_state().
