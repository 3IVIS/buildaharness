What Phase 1 delivers
Phase 1 builds the evidence and reasoning infrastructure that turns raw tool outputs into typed, reliability-graded evidence and structured hypotheses. On the evidence side it defines the Evidence dataclass, the EvidenceStore, tool reliability envelopes (which cap the maximum conclusion reliability a given tool can support), and the tool availability manifest (which records which tools are present at runtime). On the reasoning side it implements all four hypothesis generation sources — symptom_inference, counterfactual_reasoning, failure_mode_library contribution, and analogy_based_generation — together with the elimination policy (three elimination conditions, K-retention of the eliminated set) and diversity enforcement via Shannon entropy over generation sources.

Two new canvas node types are introduced: gather_evidence (which a flow author places to direct the agent to gather evidence from a specific tool) and apply_tool_reliability (which applies the registered envelopes to cap inference reliability in the evidence store).

Phase 1 completes the Reasoning layer of the architecture coverage table and seeds the EvidenceStore, HypothesisSet, and ToolAvailabilityManifest state structures into HarnessRunState.

Key constraints to respect in Phase 1
Observation / belief separation is still the primary invariant (INV-01): Evidence of type OBSERVATION is raw sensor data. It must never be auto-promoted to a Belief. The guard introduced in add_belief() in P0.2 enforces this at the WorldModel level. Phase 1 tests this invariant end-to-end: an OBSERVATION in the evidence store cannot appear in world_model.beliefs[] without an explicit derived_from[] chain.
experience_store absence must be a silent no-op: analogy_based_generation() in P1.6 is the first function that conditionally uses the experience store. It must return [] — not raise, not log an error — when experience_store is None or experience_store.available == False. This seeds INV-10, which is fully tested in P11.
Tool reliability envelopes are fail-open for unknown tools: If a tool is not registered in TOOL_RELIABILITY_ENVELOPES, the envelope lookup returns None and no capping is applied. Fail-open is intentional — it prevents overly-aggressive degradation on custom tools. Fail-closed (blocking unknown tools) is reserved for the verification adequacy critic in P5.
failure_mode_library is advisory only: Its contribution to hypothesis generation is one of four sources. The library is a stub in P1 (full implementation in P6.3). Its confidence values must be treated as advisory and never as blocking — this enforces INV-08 from the start.

P1.1 Evidence data model Week 1 · ~2 days ✓ DONE
Implement the Evidence dataclass, the EvidenceStore, and their serialisation helpers. This is the foundational structure for the entire reasoning layer — every subsequent P1 task depends on these types. Extend HarnessRunState to hold an EvidenceStore so evidence survives pause/resume.
Implementation steps
1
Create adapter/harness/evidence.py. Define EvidenceType as a Literal["OBSERVATION","INFERENCE","SYSTEM_ERROR"] type alias (not an enum — keeps JSON serialisation trivial). Define ReliabilityClass as a Literal["HIGH","MEDIUM","LOW"] type alias for the same reason.
2
Define Evidence dataclass: id: str, obs: str (the raw observation text — the only field that carries content), reliability: ReliabilityClass, source: str (the tool name or agent identifier that produced this evidence), evidence_type: EvidenceType, freshness: float (normalised [0,1] — 1.0 = just recorded; decays over time in later phases), recorded_at: datetime. Add a __post_init__ guard: if evidence_type == "SYSTEM_ERROR" and reliability != "HIGH", raise ValueError("SYSTEM_ERROR evidence must have reliability=HIGH"). SYSTEM_ERROR always denotes a definite tool-level failure — its reliability is HIGH by architectural contract.
3
Define EvidenceStore dataclass: entries: list[Evidence] = field(default_factory=list). Implement three methods: append(evidence: Evidence) → None — simply appends; query(reliability: str | None = None, evidence_type: str | None = None) → list[Evidence] — returns entries matching all provided filters; clear() → None — empties the store (used during compression in P6). Store maintains insertion order throughout its lifetime.
4
Add to_dict() / from_dict() to Evidence and EvidenceStore. Evidence.to_dict() converts recorded_at to ISO-8601 string. Evidence.from_dict() parses it back to datetime. EvidenceStore.to_dict() serialises as {"entries": [e.to_dict() for e in self.entries]}. EvidenceStore.from_dict() reconstructs correctly — including an empty-list default when "entries" key is absent.
5
Extend adapter/harness/state_store.py. Add evidence_store: EvidenceStore = field(default_factory=EvidenceStore) to HarnessRunState. Update save() to call state.evidence_store.to_dict() when writing the evidence_store JSONB column. Update load() to call EvidenceStore.from_dict() on the column value — handling a missing or null column gracefully by returning an empty EvidenceStore().
6
Update adapter/harness/__init__.py to export Evidence, EvidenceStore, EvidenceType, and ReliabilityClass so downstream modules can import from the package root.

P1.2 Tool reliability envelopes Week 1 · ~1.5 days ✓ DONE
Implement per-tool mappings that cap the maximum conclusion reliability an inference drawn from that tool's output can claim. The cap applies to INFERENCE evidence only — raw OBSERVATION evidence and SYSTEM_ERROR evidence are not capped. Unknown tools fail open (no cap applied).
Implementation steps
1
Create adapter/harness/tool_reliability.py. Define ToolEnvelope dataclass: tool_name: str, description: str, scope_limits: list[str] (human-readable list of what the tool cannot conclude), max_conclusion_reliability: ReliabilityClass. Import ReliabilityClass and Evidence from adapter.harness.evidence.
2
Populate the built-in TOOL_RELIABILITY_ENVELOPES: dict[str, ToolEnvelope] with the following entries: grep (max=LOW — confirms text presence, cannot confirm semantic meaning or absence), linter (max=MEDIUM — confirms syntax only, not behaviour), type_checker (max=MEDIUM — confirms type signatures, not runtime correctness), unit_test_runner (max=HIGH — authoritative for the exact paths under test), integration_test_runner (max=HIGH — authoritative for the tested integration path). These defaults can be overridden by adding entries to the dict at adapter init time.
3
Implement apply_tool_reliability_envelope(evidence: Evidence, envelope: ToolEnvelope) → Evidence. If evidence.evidence_type != "INFERENCE", return the evidence unchanged — raw observations and system errors are not subject to scope capping. Otherwise, compute capped = min(evidence.reliability, envelope.max_conclusion_reliability) using the ordering LOW < MEDIUM < HIGH. Return a new Evidence with reliability=capped and all other fields unchanged. Evidence objects are immutable by convention — never mutate in place.
4
Implement get_envelope(tool_name: str) → ToolEnvelope | None. Looks up TOOL_RELIABILITY_ENVELOPES.get(tool_name). Returns None for unknown tools — the caller must handle None as "no capping". This is intentionally fail-open: unregistered tools are not penalised, they are just not granted a reliability upgrade either. Add a note in the module docstring explaining the fail-open design.
Acceptance tests
apply_tool_reliability_envelope() on a HIGH-reliability INFERENCE evidence produced by grep returns a new Evidence with reliability="LOW"
apply_tool_reliability_envelope() on an OBSERVATION evidence is a no-op — the original Evidence is returned unchanged regardless of the envelope
get_envelope("grep") returns the built-in grep envelope with max_conclusion_reliability="LOW"
get_envelope("some_unknown_tool") returns None without raising a KeyError
P1.3 Tool availability manifest Week 1 · ~1.5 days ✓ DONE
Implement the tool availability manifest — a frozen record, built once at harness init, of which tools are present in the runtime environment and what fallbacks exist. The verification adequacy critic in P5.2 consults this manifest to prune unavailable 9-layer checks. The manifest is read-only after construction — runtime mutation raises a FrozenManifestError.
Implementation steps
1
Create adapter/harness/tool_manifest.py. Define FrozenManifestError(Exception). Define ToolEntry dataclass: tool_name: str, available: bool, fallback_tool: str | None.
2
Define ToolAvailabilityManifest class. Internal state: _entries: dict[str, ToolEntry] (private), _frozen: bool = False. Public methods: check_tool_availability(tool_name: str) → bool (returns _entries.get(tool_name, ToolEntry(tool_name, False, None)).available), get_fallback(tool_name: str) → str | None (returns fallback_tool if available, else None), all_tools() → list[str] (returns sorted list of registered tool names). Any write attempt after freeze raises FrozenManifestError.
3
Implement build_manifest(runtime_checks: dict[str, callable] | None = None) → ToolAvailabilityManifest. Runs availability probes for a default set of tools: grep, pylint, mypy, pytest, ruff. Default probe: shutil.which(tool_name) is not None. The runtime_checks parameter allows passing custom probe functions (used in tests to inject deterministic results). Fallback mappings: pylint → ruff, mypy → pyright (also probed). After all probes complete, calls manifest._freeze() which sets _frozen=True. Returns the frozen manifest.
4
Add to_dict() / from_dict() to ToolAvailabilityManifest for HarnessRunState persistence. The frozen flag is not persisted — a loaded manifest is always re-built (not restored from store) on harness init. The JSONB column is kept for diagnostic inspection only.
5
Extend adapter/harness/state_store.py to include a tool_manifest: ToolAvailabilityManifest | None = None field on HarnessRunState. The field holds the manifest built at harness init. load() sets this field to None — callers are expected to call build_manifest() on resume and re-assign. This design ensures the manifest always reflects the current runtime environment, not the environment when the run was first started.
Acceptance tests
build_manifest(runtime_checks={"grep": lambda: True, "mypy": lambda: False}) marks grep as available and mypy as unavailable without raising
get_fallback("mypy") returns "pyright" when mypy is unavailable and pyright is available in the manifest
Calling any write method on a frozen manifest raises FrozenManifestError
A probe that raises an exception during build_manifest() results in available=False for that tool — the build does not propagate the exception
P1.6 Hypothesis generation — 4 sources Week 2 · ~3 days ✓ DONE
Implement the Hypothesis and HypothesisSet dataclasses and all four structurally distinct generation sources. The four sources are independent by design: each one approaches the problem from a different angle, which is what gives the diversity enforcement (P1.7) its meaning. analogy_based_generation is the first function to conditionally use the experience store — it must be a clean no-op when the store is absent.
Implementation steps
1
Create adapter/harness/hypothesis.py. Define Hypothesis dataclass: id: str, explanation: str, confidence: float (normalised [0,1]), predicted_observations: list[str] (what the agent would expect to observe if this hypothesis is correct), discriminating_evidence: list[str] (evidence IDs that would confirm or refute this hypothesis), generation_sources: list[str] (names of the sources that contributed this hypothesis — must be non-empty).
2
Define HypothesisSet dataclass: active: list[Hypothesis] = field(default_factory=list), eliminated: list[Hypothesis] = field(default_factory=list), diversity_score: float = 0.0. Add to_dict() / from_dict(). Extend HarnessRunState with hypothesis_set: HypothesisSet = field(default_factory=HypothesisSet). Update save() and load() in state_store.py accordingly.
3
Implement symptom_inference(world_model: WorldModel, evidence_store: EvidenceStore) → list[Hypothesis]. Scans evidence_store.query(evidence_type="OBSERVATION") and world_model.observations[]. Groups raw observations by shared keywords or patterns (simple word-overlap clustering at this stage — full semantic clustering is a later refinement). For each cluster, synthesises one Hypothesis with generation_sources=["symptom_inference"], confidence=0.5 (uniform prior — the diagnostics layer will refine this), and predicted_observations set to the cluster members' obs fields. Returns the list of generated hypotheses.
4
Implement counterfactual_reasoning(world_model: WorldModel) → list[Hypothesis]. For each belief in world_model.beliefs[] with confidence >= 0.6, asks: "what alternative explanation would produce the same supporting evidence?" Generates one Hypothesis per high-confidence belief with explanation set to the negation/alternative of the belief's statement, confidence=0.3 (counterfactuals start lower), generation_sources=["counterfactual_reasoning"]. If world_model.beliefs is empty, returns [] without error.
5
Implement failure_mode_library_contribution(fml_stub: dict | None) → list[Hypothesis]. Accepts a stub dictionary of pattern name → description (this is populated fully in P6.3). For each entry in the stub, creates a Hypothesis with generation_sources=["failure_mode_library"], confidence=0.2 (advisory — failure mode patterns are suggestive, not authoritative), and explanation set to the pattern description. If fml_stub is None or empty dict, returns [] cleanly. This function must never write to any control structure — it is read-only advisory (INV-08 enforcement point).
6
Implement analogy_based_generation(experience_store) → list[Hypothesis]. Guard on line 1: if experience_store is None or not getattr(experience_store, "available", False): return []. This is the canonical no-op guard for INV-10. When the store is available: queries experience_store.successful_decompositions for entries structurally similar to the current world model (cosine similarity on observation vectors — stub returns empty until P8 implements the real store). Returns hypotheses tagged generation_sources=["analogy_based"].
7
Implement generate_hypotheses(world_model, evidence_store, fml_stub=None, experience_store=None) → list[Hypothesis]. Calls all four source functions, collects results. Deduplicates by checking if two hypotheses have >80% word overlap in their explanation fields (simple Jaccard similarity — merges sources by combining their generation_sources lists). Assigns unique IDs via uuid.uuid4(). Returns the deduplicated list. The caller (in P1.7) is responsible for updating the HypothesisSet.
Acceptance tests
Each of the four generation functions independently produces ≥1 hypothesis when given a WorldModel fixture containing at least one observation and one belief
The generation_sources[] list on each returned hypothesis correctly names its source (e.g., ["symptom_inference"])
analogy_based_generation(None) returns [] without raising any exception (INV-10 seed test)
generate_hypotheses() with all four sources active returns a non-empty list and no hypothesis has an empty generation_sources[]
P1.7 Hypothesis elimination policy + diversity enforcement Week 3 · ~2.5 days ✓ DONE
Implement the three-condition elimination policy, K-retention of the eliminated set, the Shannon entropy diversity score, and the enforce_diversity() function that triggers additional generation passes when diversity falls below 0.7. All constants (posterior_floor, prediction_failure_threshold, k_retention, diversity_threshold) are explicit fields on EliminationPolicy — never hardcoded in the logic functions.
Implementation steps
1
Add EliminationReason type alias to hypothesis.py: Literal["CONTRADICTING_EVIDENCE","PREDICTION_FAILURE","POSTERIOR_BELOW_FLOOR"]. Define EliminationRecord dataclass: hypothesis_id: str, reason: EliminationReason, eliminated_at: datetime. The eliminated set stores EliminationRecord objects alongside the Hypothesis objects — update HypothesisSet.eliminated type to list[tuple[Hypothesis, EliminationRecord]].
2
Define EliminationPolicy dataclass: prediction_failure_threshold: int = 3, posterior_floor: float = 0.05, k_retention: int = 20, diversity_threshold: float = 0.7, max_diversity_passes: int = 3. All defaults are documented in the module docstring with the rationale for each value. Add to_dict() / from_dict().
3
Implement check_contradicting_evidence(hypothesis: Hypothesis, evidence_store: EvidenceStore) → bool. For each entry in evidence_store.entries: if the entry's obs text directly contradicts any string in hypothesis.predicted_observations (contradiction = a HIGH-reliability OBSERVATION or INFERENCE that says the opposite of a predicted observation), return True. Use simple keyword negation at this stage ("no X found" contradicts "X present"). Returns False if no contradiction found.
4
Implement check_prediction_failure(hypothesis: Hypothesis, failure_counts: dict[str, int], policy: EliminationPolicy) → bool. Returns True if failure_counts.get(hypothesis.id, 0) >= policy.prediction_failure_threshold. The failure_counts dict is maintained by the main loop (incremented each iteration when a predicted_observation was expected but not observed). This function only reads it.
5
Implement check_posterior_floor(hypothesis: Hypothesis, policy: EliminationPolicy) → bool. Returns True if hypothesis.confidence < policy.posterior_floor. Straightforward threshold check.
6
Implement eliminate(hypothesis_set: HypothesisSet, evidence_store: EvidenceStore, failure_counts: dict, policy: EliminationPolicy) → HypothesisSet. For each hypothesis in hypothesis_set.active: check all three conditions. If any condition is True, move it to hypothesis_set.eliminated with the appropriate EliminationReason. After all eliminations: if len(hypothesis_set.eliminated) > policy.k_retention, drop the oldest entries (by eliminated_at) until the list has exactly k_retention entries. Returns the updated HypothesisSet. Does not raise if active list becomes empty — an empty active set is a valid signal that feeds into coverage_health in P3.
7
Implement compute_diversity_score(hypothesis_set: HypothesisSet) → float. Collects all generation_sources values across hypothesis_set.active hypotheses. Computes frequency of each unique source. Applies Shannon entropy: H = -sum(p * log2(p) for p in probs). Normalises by log2(num_unique_sources) (avoids division by zero: if fewer than 2 unique sources, score = 0.0). Clamps result to [0,1]. Updates hypothesis_set.diversity_score = score in place and returns the score.
8
Implement enforce_diversity(hypothesis_set, world_model, evidence_store, fml_stub, experience_store, policy) → HypothesisSet. Calls compute_diversity_score(). While hypothesis_set.diversity_score < policy.diversity_threshold and pass_count < policy.max_diversity_passes: calls generate_hypotheses() to generate additional hypotheses, appends them to hypothesis_set.active, recomputes diversity score. Returns the updated set. The max_diversity_passes cap prevents infinite looping when the generation sources genuinely cannot produce diverse output.
Acceptance tests
A hypothesis with confidence=0.02 (below posterior_floor=0.05) is moved to hypothesis_set.eliminated with reason="POSTERIOR_BELOW_FLOOR"
After eliminating 25 hypotheses with k_retention=20, len(hypothesis_set.eliminated) is exactly 20 — the 5 oldest are dropped
enforce_diversity() triggers additional generation passes when all active hypotheses share the same single source (diversity_score = 0.0 < 0.7)
Each of the three elimination conditions independently triggers elimination when its threshold is met: contradicting evidence eliminates, prediction_failure_threshold=1 eliminates after 1 failure, posterior_floor=0.99 eliminates all but the highest-confidence hypothesis

Architecture invariants engaged in Phase 1
Phase 1 delivers the first full test of INV-01 (seeded in P0.2) and seeds INV-10 through the analogy_based_generation() no-op guard. It also establishes the enforcement points for INV-08 through the failure mode library contribution design.
INV-01 — Observation vs conclusion separation P0.2 seeds · P1 full test The Evidence type introduced in P1.1 is the first place where the separation is tested end-to-end: an Evidence object with evidence_type="OBSERVATION" can only enter world_model.observations[] via add_observation() — it can never appear as the sole basis for a Belief without an explicit derived_from[] chain. The P1 test suite includes a test that creates an OBSERVATION evidence entry and asserts that calling add_belief() with only that entry's ID in derived_from[] still requires the caller to explicitly provide the derivation — there is no auto-promotion shortcut.
INV-08 — Failure mode library scope P1.6 seeds · P6 full test The failure_mode_library_contribution() function in P1.6 is the first code that bridges the failure mode library into the reasoning pipeline. The function's return type is list[Hypothesis] — it has no write path to block_mask, escalation_reason, or any control structure. This is enforced by the type signature. Phase 6.3 will replace the stub dict parameter with the full library, but the scope constraint is set here: the library contributes hypotheses only.
INV-10 — experience_store is a no-op when absent P1.6 seeds · P8/P11 full test analogy_based_generation() in P1.6 is the first function that conditionally uses the experience store. The guard on its first line — if experience_store is None or not getattr(experience_store, "available", False): return [] — is the canonical pattern for all experience-store-conditional code in later phases. Phase 1 tests this guard directly: calling analogy_based_generation(None) must return [] without side effects.
Invariants that Phase 1 must not break
All invariants established in P0 must continue to hold. Specifically:
INV-03 — generation_id double-increment Phase 1 does not touch the generation_id increment path. None of the new functions call increment_generation_id(). Evidence and hypothesis updates do not advance the generation_id — that is strictly the responsibility of the main loop wired in P3.
INV-06 — control_state as sole control input The hypothesis set and evidence store introduced in P1 are read-only context for action selection. Neither HypothesisSet nor EvidenceStore has any write path to a control_state structure. Diagnostic proxies updated by P1.5 (the feasibility stub) do not constitute a control signal — they are inputs to the normalisation contract that feeds resolve_control_state() in P3.
