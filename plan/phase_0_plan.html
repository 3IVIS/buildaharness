What Phase 0 delivers
Phase 0 creates the skeleton that all other phases hang off. It does not implement any intelligent behaviour — no diagnostics, no hypothesis generation, no control state resolution. Instead it establishes the data contracts (FlowSpec v1.0.0, all 13 state structure dataclasses, the HarnessRunState persistence container) and the staleness tracking infrastructure (generation_id increment helper, staleness_check(), gate-assertion stubs) so that Phase 1 can begin writing real logic against stable interfaces.

Every subsequent phase depends on at least one artifact from Phase 0. Specifically: P1 needs the evidence_store and world_model structures; P2 needs world_model with its contradiction list; P3 needs the generation_id infrastructure and caller_state; P4 needs the task_graph schema; P5 needs the output_contract model; P6 needs the backend state store for pause/resume. Skipping or shortcutting Phase 0 tasks causes cascading rework.

Key constraints to respect in Phase 0
Backwards compatibility: FlowSpec v1.0.0 must validate all 6 existing v0.2.0 reference flows in flows/ without modification. All new harness fields are optional with sensible defaults.
Observation / belief separation is enforced at struct level: The world_model dataclass must make it structurally impossible to auto-promote an observation to a belief — they are separate lists with distinct types. This invariant (INV-01) must be enshrined in the types, not just in documentation.
Stubs must fail loudly, not silently: Gate stubs (decomposition_gate, action_gate, post_exec_gate) should raise NotImplementedError in Phase 0, not return a default. Silent pass-throughs will mask integration failures in P3.

Implementation steps

4
Implement the gate-assertion decorator @assert_generation_fresh. Decorator wraps a gate function. On call: runs staleness_check(control_state, world_model); if stale, raises StalenessError(f"control_state.generation_id={cs.generation_id} is behind world_model.generation_id={wm.generation_id}"). Define StalenessError(Exception) in the same file.
5
Create adapter/harness/gates.py. Implement three stub gate functions decorated with @assert_generation_fresh: decomposition_gate(task_graph, control_state, world_model), action_gate(action, control_state, world_model), post_exec_gate(result, verification, control_state, world_model). All three stubs raise NotImplementedError("gate not implemented until P3") after passing the staleness assertion. This ensures the decorator is exercised and that callers don't accidentally treat a stub as a real gate.
Note on double-increment
The architecture specifies that world_model.generation_id increments twice per main loop iteration: once after sub-step A (pre-execution) and once after sub-step B (post-execution). Phase 0 provides the increment helper; the two-substep wiring is done in P3 when the main loop structure is established. Tests below verify single increments now; the double-increment invariant test (INV-03) is added in P3.

Architecture invariants seeded in Phase 0
Phase 0 lays the groundwork for two of the ten architectural invariants. The others are tested in their respective phases but depend on the structures defined here.
INV-01 — Observation vs conclusion separation P0.2 seeds · P1 full test observations[] and beliefs[] are separate typed structures in WorldModel. A HIGH-reliability tool output stored as an Observation cannot automatically produce a Belief — the add_belief() helper enforces that derived_from[] is non-empty. This is enforced at the struct level in P0.2 so that no amount of careless code in later phases can silently bypass it.
INV-03 — generation_id double-increment per iteration P0.3 seeds · P3 full test increment_generation_id() is the single canonical place to advance the counter. Phase 0 provides the helper and the staleness assertion decorator. The double-increment wiring (sub-step A before execution, sub-step B after execution) is established in P3 when the main loop structure exists. The P0.3 tests verify single increments and staleness detection; the double-increment invariant test (INV-03) is added in P3.
Invariants that Phase 0 must not break
All invariants from full_harness_architecture.html that are not explicitly seeded in P0 must still hold in the structures defined here. Specifically:
INV-06 — control_state as sole control input The ControlStateStub defined in P0.3 must not be mixed into world_model or hypothesis_set. It is a separate object. Future phases may not pull control signals from world_model fields — control_state is the only gate.
INV-10 — experience_store is a no-op when absent HarnessRunState (P0.6) stores an experience_store_ref JSONB column that defaults to {}. The design must ensure that an absent (null/empty) ref does not cause load() to raise — it returns None for that field, and all downstream consumers guard with if experience_store is not None.
