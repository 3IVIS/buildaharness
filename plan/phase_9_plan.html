What Phase 9 delivers
Phase 9 adds the final quality layer before the harness returns a result: the three-lens reviewer pass and the full output contract validator. Both were stubbed in earlier phases — Phase 9 replaces those stubs with real implementations.

Adversarial prior — causal proximity filter (P9.1) implements the seeding mechanism for the adversarial lens. The prior is built by selecting beliefs from the world model that are most causally close to the success_criteria dependency chain — not by selecting the highest-confidence beliefs. This means the adversarial lens challenges the beliefs the harness is most relying on to conclude success, not just the beliefs it is most certain about. The prior is ephemeral: created at the start of reviewer_pass(), consumed by adversarial_lens(), and discarded before the function returns. It never writes to world_model or hypothesis_set (INV-09).

Three-lens reviewer pass (P9.2) runs three structurally independent review lenses. The implementer lens asks "did I do what I intended?" — checking that executed actions match task descriptions and that no task has been silently skipped. The reviewer lens asks "what would a PR reviewer criticise?" — checking output contract pre-conditions, interface consistency, and known code quality heuristics. The adversarial lens asks "what would fail if the adversarial prior were true?" — deliberately trying to break the harness's conclusions from a hostile starting point. Each lens produces ReviewFinding objects. After all three run, their findings are integrated into world_model, hypotheses are updated and eliminated, contradictions are re-detected, and abstraction_fit is recomputed unconditionally regardless of whether task_graph.changed is set.

drain_propagation_queue() + loop re-entry (P9.3) handles the case where reviewer findings invalidate work that was already marked COMPLETE. When a finding propagates through the belief dependency graph and reaches a belief that a completed task depended on, that task is re-opened. The main loop re-enters for these tasks before the run can conclude.

Full validate_output_contract() (P9.4) replaces the P0.5 stub with a complete implementation. It checks every dimension of the output contract — format, required sections, interface fields, caller-specific constraints — using the current caller_state, which may have been updated mid-run by the P7 constraint propagation path. It is the authoritative final gate before the harness returns.

P9.1 Adversarial prior — causal proximity filter Week 1 · ~2 days
Implement the seeding mechanism for the adversarial lens. Beliefs are selected for negation based on their causal proximity to the success_criteria dependency chain — not based on raw confidence. This ensures the adversarial lens attacks the beliefs the harness is most relying on to declare success, creating a genuine blind-spot break rather than a challenge to peripheral beliefs.
Implementation steps
1
Create adapter/harness/reviewer.py. Define ReviewFinding dataclass: lens: Literal["implementer","reviewer","adversarial"], finding_type: Literal["contradiction","gap","regression","assumption_violation","contract_miss"], description: str, affected_belief_ids: list[str], affected_task_ids: list[str], severity: Literal["LOW","MEDIUM","HIGH"]. Define AdversarialPrior dataclass: negated_beliefs: list[dict] (each entry has belief_id: str, negated_statement: str, causal_proximity: float), seeded_from_failure_history: bool, dep_class_gap_included: bool.
2
Implement compute_causal_proximity(belief_id, success_criteria, belief_dep_graph) → float. Performs a reverse BFS from the success_criteria leaf nodes in the belief dependency graph back towards the root beliefs. Proximity score = 1.0 / (shortest_path_length + 1), normalised so direct dependencies of success_criteria beliefs score highest. Beliefs with no path to any success_criteria belief score 0.0. Returns a float in [0,1].
3
Implement seed_adversarial_prior(world_model, success_criteria, failure_history, dep_class_gap_annotation=None) → AdversarialPrior. Steps: (1) compute causal_proximity for each belief in world_model.beliefs[]; (2) select top-K beliefs by proximity (K defaults to 5, configurable); (3) negate each selected belief's statement to form adversarial hypotheses; (4) supplement with beliefs implied by failure_history class priors (any failure class with base_rate > 0.3 seeds additional negations); (5) if dep_class_gap_annotation is present, include the beliefs it references as additional seeds. Return AdversarialPrior with the assembled negated_beliefs list.
4
Enforce ephemeral scope. The AdversarialPrior returned by seed_adversarial_prior() must only ever be stored as a local variable in the calling scope. Add a type-level guard: AdversarialPrior does not implement to_dict() or from_dict() — making it structurally impossible to serialise to HarnessRunState. Add a test assertion (in P9.2) that after reviewer_pass() returns, no field of HarnessRunState holds an AdversarialPrior instance.
Acceptance tests
Given two beliefs — one with direct causal path to a success criterion (proximity 1.0) and one with no path (proximity 0.0) — seed_adversarial_prior() selects the high-proximity belief for negation, not the high-confidence low-proximity one
After reviewer_pass() completes: no field of HarnessRunState holds an AdversarialPrior instance — the prior has been discarded as required by INV-09
When dep_class_gap_annotation is provided: the beliefs it references appear in adversarial_prior.negated_beliefs and dep_class_gap_included is True
Implementation notes — reviewer.py lines 1–175. compute_causal_proximity() uses undirected BFS returning 1.0/(dist+1); success-criteria nodes matched by exact belief-ID only (substring matching discarded to avoid false positives, e.g. "sc" in "description"). seed_adversarial_prior() takes an additional belief_dep_graph kwarg so reviewer_pass() can pass it down. AdversarialPrior has no to_dict()/from_dict() — structural persistence guard for INV-09. T01 ✅ T02 ✅ T03 ✅
P9.2 Three-lens reviewer pass Weeks 1–2 · ~3 days
Implement the three reviewer lenses and the reviewer_pass() orchestrator that runs them in sequence. Each lens is structurally independent — implementer and reviewer lenses have no knowledge of what the adversarial lens will check. After all three lenses complete, their findings are integrated into world_model via the standard update path, and a full contradiction detection + belief propagation cycle runs. Abstraction fit is recomputed unconditionally regardless of task_graph.changed.
Implementation steps
1
Implement implementer_lens(world_model, task_graph, success_criteria) → list[ReviewFinding]. Checks: (a) every task in COMPLETE status has at least one observation in world_model.observations[] that references it by task_id — silent task completion without evidence is a HIGH finding; (b) all success_criteria are covered by at least one COMPLETE task — uncovered criterion is a MEDIUM finding; (c) no task has contradicting COMPLETE status and an open contradiction in world_model.contradictions[] referencing its output — if found, finding_type is "contradiction" with HIGH severity.
2
Implement reviewer_lens(world_model, output_contract, task_graph, evidence_store) → list[ReviewFinding]. Checks: (a) any required_interface_fields from output_contract not yet produced by any COMPLETE task — finding_type "contract_miss", severity HIGH; (b) any assumption in world_model.assumptions[] that has not been validated by at least one HIGH-reliability observation — finding_type "assumption_violation", severity MEDIUM; (c) any HIGH-reliability evidence that contradicts a current belief but has not yet triggered a contradiction record — finding_type "gap", severity MEDIUM. The reviewer lens does not have access to the adversarial prior.
3
Implement adversarial_lens(world_model, adversarial_prior, hypothesis_set, task_graph) → list[ReviewFinding]. For each negated belief in adversarial_prior.negated_beliefs: check whether, if the negation were true, any COMPLETE task result would be invalidated. If yes: produce a ReviewFinding(lens="adversarial", finding_type="contradiction", severity=HIGH) listing the affected task_ids. Also check whether any active hypothesis in hypothesis_set is only supported by beliefs in the adversarial prior's negated set — if so, that hypothesis should be flagged as inadequately supported (severity MEDIUM).
4
Implement reviewer_pass(world_model, task_graph, success_criteria, output_contract, hypothesis_set, evidence_store, caller_state, belief_dep_graph, failure_history, dep_class_gap_annotation=None) → list[ReviewFinding]. Sequence: (1) call seed_adversarial_prior() to build the local prior; (2) call all three lenses, collecting findings; (3) discard adversarial_prior (it goes out of scope after this block); (4) for each HIGH-severity finding: call integrate_evidence() to add a new observation reflecting the finding, then call add_belief() to record the corresponding corrective belief with appropriate derived_from[] links; (5) call propagate_beliefs() on the updated dep graph; (6) call update_hypothesis_set() and run the elimination policy; (7) call detect_contradictions(); (8) call check_abstraction_alignment() unconditionally.
5
Return the aggregated findings list and update diagnostics. After the full post-pass update: recompute diagnostics.verification_health.strength based on unresolved HIGH-severity findings (each unresolved HIGH finding reduces strength by a configurable delta). Recompute diagnostics.coverage_health.explanation_coverage from updated hypothesis coverage. Return the full list of all findings from all three lenses — callers use this list to populate the propagation queue (P9.3) and to feed the Langfuse trace (P11).
Acceptance tests
All three lenses execute when called via reviewer_pass() — each lens contributes at least one finding to the returned list when given a non-trivial world_model with an unvalidated assumption and an incomplete task
Adversarial lens findings reference beliefs and tasks that are not flagged by the implementer or reviewer lenses — confirming the adversarial prior creates partial blind-spot coverage independent of the other two lenses
After reviewer_pass() returns: abstraction_fit has been recomputed even when task_graph.changed is False — the unconditional recompute is verified by comparing the pre-pass and post-pass scores
Implementation notes — reviewer.py lines 176–460. Each lens is a standalone function; reviewer_pass() orchestrates the 10-step sequence: seed → 3 lenses → discard prior → integrate HIGH findings (Observation then Belief, INV-01) → propagate_beliefs() → eliminate() → detect_contradictions(task_graph.to_dict()) (the detect_abstraction_contradictions helper expects a dict, not a TaskGraph) → check_abstraction_alignment(force=True) → drain → return ReviewPassResult. P9.3 drain_propagation_queue() + loop re-entry Week 2 · ~2 days
Implement the propagation drain that converts reviewer findings into task re-opens, and wire the loop re-entry signal that causes the main loop to process those tasks before the run can return. The drain runs as the final step of reviewer_pass() and is the mechanism by which a completed task can be moved back to PENDING when the reviewer discovers that its underlying beliefs have been invalidated.
Implementation steps
1
Implement drain_propagation_queue(belief_dep_graph, task_graph) → list[str] in reviewer.py. The belief_dep_graph.propagation_queue contains belief IDs that have been updated or invalidated since the last drain. For each belief_id in the queue: look up all task_ids in task_graph that list this belief_id in their completed evidence. Collect the distinct set of affected task_ids. Clear the propagation_queue. Return the collected task_ids list (may be empty).
2
Implement task re-open logic. For each task_id in the returned list from drain_propagation_queue(): if the task's current status is COMPLETE — transition it to PENDING and clear its completed_evidence list. If the task's status is FAILED, BLOCKED, or ACTIVE — leave it unchanged (it is already being processed or awaits processing). Log each re-open as a journal entry with reason "reviewer_finding_invalidated_evidence" and the relevant belief_ids.
3
Wire into reviewer_pass() as the final step. After the full post-pass update sequence from P9.2 step 4: call drain_propagation_queue(belief_dep_graph, task_graph). If the returned list is non-empty: set a tasks_reopened: bool = True field on the WarmStartResult-equivalent return object (or use a dedicated ReviewPassResult dataclass). Callers check this flag to decide whether to re-enter the main loop.
4
Wire loop re-entry in loop.py. After reviewer_pass() returns: if review_result.tasks_reopened is True — do not proceed to completion_check_final(). Instead, increment world_model.generation_id, call resolve_control_state(), and continue the main loop from the top of the next iteration. The loop will naturally pick up the reopened PENDING tasks via select_unblocked_leaf(). If tasks_reopened is False — proceed to completion_check_final() (P9.4).
Acceptance tests
When a HIGH-severity reviewer finding invalidates the evidence of a COMPLETE task: drain_propagation_queue() returns that task's ID, and the task's status is transitioned from COMPLETE to PENDING — the task will be re-executed in the next loop iteration
When the propagation_queue is empty after reviewer_pass() (no invalidated evidence): drain_propagation_queue() returns an empty list and review_result.tasks_reopened is False — the loop does not re-enter and proceeds to completion_check_final()
A task whose status is FAILED or BLOCKED at the time of the drain is not re-transitioned to PENDING — only COMPLETE tasks are subject to re-open; the drain does not interfere with tasks already in non-terminal states
Implementation notes — reviewer.py lines 461–530 + task_graph.py (added completed_evidence: list[str] to Task). drain_propagation_queue() directly sets task.status = "PENDING" on COMPLETE tasks (bypasses update_task_status() validation, which guards COMPLETE as terminal for all other callers). The _VALID_TRANSITIONS["COMPLETE"] = set() contract is preserved — only the drain has this bypass. loop.py wired with belief_dep_graph + evidence_store params; reviewer pass fires after post_exec_gate; tasks_reopened included in return dict. 2
Implement check_format_requirements(result, output_contract) → list[str]. Validates output_contract.format_requirements against the result. Checks: response format matches (e.g. JSON, markdown, plain text as specified); if max_length is set in format_requirements, result length does not exceed it; if required_fields are listed at the top level, they are all present. Returns a list of violation strings (empty list = pass).
3
Implement check_required_sections(result, output_contract) → list[str]. For each section name in output_contract.required_sections: verify the section is present and non-empty in the result. A section is "present" if a top-level key (for dict results) or a heading (for text results) matching the section name exists. Returns violation strings for each missing or empty section.
4
Implement check_interface_constraints(result, output_contract) → list[str]. For each field in output_contract.required_interface_fields: verify the field is present with the correct type as declared in output_contract.interface_constraints. Detects type regressions (field was previously a string, now returned as int). A missing required interface field is a HIGH violation. A type regression on a non-required field is a MEDIUM violation. Returns violation strings for each issue found.
5
Implement check_caller_specific_constraints(result, caller_state) → list[str]. Reads caller_state.current_constraints — the live list that may have been updated by P7 constraint propagation during the run. For each constraint: evaluate whether the result satisfies it. Constraints are expressed as simple predicate strings (e.g. "output must not reference deleted files", "response must be in English") that are pattern-matched or evaluated as heuristics. Returns violation strings for each unmet constraint.
6
Assemble the full validate_output_contract(result, output_contract, caller_state) → ContractCheckResult. Calls all four sub-checks in sequence. Collects all violation strings. Returns ContractCheckResult(passed=len(violations)==0, violations=violations, is_stub=False). Then implement completion_check_final(result, output_contract, caller_state, harness_run_state) → ContractCheckResult: calls validate_output_contract(), and if passed=False: calls escalate(SurfaceBlocker(reason="contract_violation", missing_info=violations, ...)) — the harness does not return a contract-failing result silently.

Architecture invariants engaged in Phase 9
INV-09 — Adversarial prior discarded after use P9.1 + P9.2 full enforcement adversarial_prior is seeded on causal proximity to success_criteria (P9.1) and must be discarded after adversarial_lens() completes (P9.2). The discard is structural: AdversarialPrior does not implement to_dict(), making it impossible to persist to HarnessRunState. The prior must never pollute world_model, hypothesis_set, or evidence_store. After reviewer_pass() returns, no field of the live harness state graph may hold an AdversarialPrior reference. The P9.1 acceptance test explicitly asserts this post-call state.
INV-07 — dep_class_gap advisory only (adversarial prior seeding) P9.1 scope dep_class_gap_annotation is an advisory annotation, not a numeric control input. In P9.1 it contributes to the adversarial prior's belief selection pool — its referenced beliefs are added as additional seed candidates alongside the causal-proximity-selected beliefs. This is its only numeric influence pathway. It does not modify control_state, block_mask, or any diagnostic sub-dimension as a result of being referenced in the adversarial prior.
INV-01 — Observation vs conclusion separation (reviewer findings integration) P9.2 scope When reviewer findings are integrated into world_model in P9.2 step 4: each finding produces an Observation (the finding itself is an observation of a review outcome) and, if the finding warrants a corrective action, a Belief with explicit derived_from=[finding_observation_id]. A HIGH-severity ReviewFinding never directly produces a Belief without an intermediate Observation — the same invariant that governs all other evidence integration paths applies here without exception.
INV-06 — control_state as sole control input (reviewer re-entry) P9.3 scope The loop re-entry triggered by tasks_reopened=True from P9.3 must not modify control_state directly. After re-entry, the loop increments world_model.generation_id and calls resolve_control_state() — which is the sole mechanism that may produce a new control_state value. The reviewer findings that caused the re-entry are observable in world_model but do not write to control_state or block_mask directly.
