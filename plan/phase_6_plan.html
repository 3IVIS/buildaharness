What Phase 6 delivers
Phase 6 closes the loop between execution failure and autonomous recovery. Without this phase, a failing task simply fails — there is no mechanism to detect stagnation, try alternative strategies, or compact a growing world model. With Phase 6 complete, the harness can sustain long runs gracefully.

cannot_make_progress() (P6.1) detects four distinct stall patterns via measurable proxies: completion velocity, strategy loops, failure recurrence, and risk-state oscillation — any one of which triggers recovery logic. 6 named recovery strategies (P6.2) are applied in a defined order (DIRECT_EDIT → TRACE_EXEC → BROADER_SEARCH → REIMPLEMENT → MINIMAL_FIX → ESCALATE), with the failure mode library providing advisory bias and experience_store enabling adaptive softmax ordering after runs accumulate. Failure mode library (P6.3) is the read-only diagnostic pattern matcher that contributes to both strategy selection and hypothesis generation — it never writes to block_mask or escalation_reason directly. Local vs global replanning (P6.4) scopes recovery to the minimum footprint needed: a local contradiction triggers task-level replanning; a global contradiction triggers full task_graph rebuild. Context compression (P6.5) manages world model growth by tracking which structures were fully dropped (compressed_structures[]) versus partially truncated (pruned_regions[]), and updating completeness_flags accordingly so downstream checks can reason about data completeness. Journal retention + max_steps (P6.6) bounds the execution journal to prevent unbounded growth and enforces a hard step budget that warns at 80% and escalates at 100%.

Key constraints to respect in Phase 6
Failure mode library is read-only and advisory only: The failure_mode_library contributes matched_pattern.confidence (normalised) to Tier 4 of resolve_control_state() and seeds hypotheses. It has no write path to block_mask or escalation_reason. Its MatchResult type must not include any field that can influence control decisions directly — the type system enforces this, not just convention.
Recovery strategy order is the default, not the authority: The fixed DIRECT_EDIT → TRACE_EXEC → BROADER_SEARCH → REIMPLEMENT → MINIMAL_FIX → ESCALATE order governs when experience_store is unavailable. When experience_store is available, the softmax policy replaces the fixed order entirely. Both paths must produce the same interface — the caller does not need to know which policy is active.
Structure present ≠ structure complete after compression: When compress_memory() prunes a belief region, the world model still contains that belief list, but completeness_flags{} must mark the region as incomplete. Callers that check "does the world model have beliefs about X?" must also check completeness_flags["X"] before treating the absence of a belief as meaningful. This distinction is enforced in action_dep_overlap() (P5.7).
max_steps escalation is hard — not silent halt: When step_count reaches max_steps, the harness must call escalate(surface_blocker(reason="budget_exhausted")) rather than raising an exception or returning silently. The caller must receive a structured escalation so they can decide whether to extend the budget or abandon the task.

P6.1 cannot_make_progress() — 4 proxies Week 1 · ~2 days
Implement the stall detector with four measurable proxies. Any single proxy returning True causes the function to return True and triggers recovery logic in the main loop. Threshold constants must be named, documented, and tunable via config.
Implementation steps
1
Create adapter/harness/progress.py. Define threshold constants at module level: STALL_WINDOW: int = 5 (steps with no completion advance), MAX_SWITCHES: int = 3 (strategy switches without progress), RECURRENCE_THRESHOLD: int = 3 (consecutive same-failure-mode entries), OSCILLATION_WINDOW: int = 6 (steps for risk-state oscillation detection). Add docstring explaining each constant and how to override via environment variable.
2
Implement proxy 1 — completion velocity: _stalled_completion(strategy_state, task_graph) → bool. Reads strategy_state.completion_history[] — a list of completed task counts at each step. Returns True if the last STALL_WINDOW entries all have the same count (no new tasks completed in STALL_WINDOW steps). Returns False if completion_history has fewer than STALL_WINDOW entries.
3
Implement proxy 2 — strategy loop: _strategy_looping(strategy_state) → bool. Returns True if strategy_state.switch_count > MAX_SWITCHES AND the completion count has not advanced since the first switch. The combination check prevents false positives when a switch actually helped.
4
Implement proxy 3 — failure recurrence: _failure_recurring(failure_diagnostics) → bool. Reads the last RECURRENCE_THRESHOLD entries in failure_diagnostics.failure_history[]. Returns True if all entries share the same failure_class value — the same category of failure repeating without resolution.
5
Implement proxy 4 — risk-state oscillation: _risk_oscillating(strategy_state) → bool. Reads the last OSCILLATION_WINDOW entries in strategy_state.risk_state_history[]. Returns True if the sequence alternates between values below and above CAUTIOUS (i.e., the harness cannot settle into a stable risk state). At least 2 alternations within the window are required.
6
Implement cannot_make_progress(strategy_state, failure_diagnostics, task_graph) → bool. Calls all four proxy functions. Returns True if any one returns True — short-circuits to avoid unnecessary computation. Records which proxy triggered in strategy_state.stall_reason for downstream logging and debugging.
Acceptance tests
Each proxy independently returns True when its condition is met while all other proxies would return False
All four proxies False → cannot_make_progress() returns False; the stall_reason field is empty
Threshold constants are overridable: setting STALL_WINDOW=2 causes proxy 1 to fire after 2 stalled steps instead of 5
P6.2 6 named recovery strategies Week 2 · ~3 days
Implement the recovery strategy state machine with six strategies in a defined progression. The failure mode library provides advisory bias (does not override). When experience_store is available, the fixed order is replaced by a softmax policy — both paths present the same interface to callers.
Implementation steps
1
Create adapter/harness/recovery.py. Define StrategyType as a Literal: "DIRECT_EDIT" | "TRACE_EXEC" | "BROADER_SEARCH" | "REIMPLEMENT" | "MINIMAL_FIX" | "ESCALATE". Define STRATEGY_ORDER: list[StrategyType] = the fixed progression in order. This list is the default when no experience data is available.
2
Define StrategyState dataclass: current_strategy: StrategyType, switch_count: int = 0, switch_triggers: list[str] = field(default_factory=list), prior_strategy_weights: dict = field(default_factory=dict), completion_history: list[int] = field(default_factory=list), risk_state_history: list[str] = field(default_factory=list), stall_reason: str = "". Add to_dict() / from_dict() for persistence.
3
Implement get_next_strategy(strategy_state) → StrategyType. Returns the next strategy in STRATEGY_ORDER after the current one. If current is ESCALATE (the last), returns ESCALATE again — escalation is a terminal state. Does not mutate strategy_state.
4
Implement switch_strategy(strategy_state, reason: str) → StrategyState. Advances current_strategy to the next in order, increments switch_count, appends reason to switch_triggers[]. Returns a new StrategyState (immutable update). The caller replaces the old state with the returned value.
5
Implement advisory bias: apply_failure_mode_bias(match_result, strategy_state) → StrategyType. If match_result.confidence ≥ 0.7 and the matched pattern maps to a specific strategy, return that strategy as a suggestion. Otherwise return get_next_strategy(strategy_state). The caller decides whether to follow the suggestion — advisory bias does not override the caller's choice. Document that this function returns a suggestion, not a command.
6
Implement adaptive upgrade: get_strategy_with_experience(strategy_state, failure_class, experience_store) → StrategyType. When experience_store.available: compute softmax over strategy weights keyed by (strategy, failure_class). Return the highest-probability strategy. When unavailable: fall back to get_next_strategy(strategy_state) transparently. This is the function the main loop calls — it handles both paths identically from the caller's perspective.
Acceptance tests
Fixed strategy order is respected when no experience_store and no matched_pattern: DIRECT_EDIT → TRACE_EXEC → BROADER_SEARCH → REIMPLEMENT → MINIMAL_FIX → ESCALATE
Advisory bias from matched_pattern (confidence ≥ 0.7) returns a specific strategy as a suggestion but does not override the caller's decision — the fixed order is still consulted if the suggestion is rejected
Softmax policy returns a different strategy ordering than the fixed enum after 5 runs with a consistent DIRECT_EDIT failure on the same failure_class — experience_store.available=False falls back to fixed order identically
P6.3 Failure mode library Week 1 · ~1.5 days
Implement the failure mode library as a read-only diagnostic pattern matcher. It contributes normalised confidence to Tier 4 of resolve_control_state() and seeds hypotheses as one of the four generation sources. Its MatchResult type is designed to have no write path to block_mask or escalation_reason — this is enforced by type structure, not just documentation.
Implementation steps
1
Create adapter/harness/failure_modes.py. Define FailurePattern dataclass: name: str, description: str, required_conditions: list[str] (belief/observation patterns that must be present), excluded_conditions: list[str] (patterns that must be absent), strategy_affinity: StrategyType | None (advisory only), hypothesis_template: str (for hypothesis generation).
2
Define MatchResult dataclass: matched: bool, pattern_name: str, raw_confidence: float, normalised_confidence: float. The type has no fields for block_mask entries, escalation_reason, or control decisions. Implement normalise_confidence(raw: float) → float: clamp to [0,1] via max(0.0, min(1.0, raw)).
3
Define FailureModeLibrary class: holds a patterns: list[FailurePattern] that is frozen at init (no mutations after construction). Implement match(world_model, hypothesis_set, task_graph) → MatchResult. For each pattern: check required_conditions against world_model.beliefs and hypothesis_set, exclude if excluded_conditions match. The pattern with the most matching conditions wins; confidence = matched_count / total_conditions (raw). Return a single best-match MatchResult.
4
Seed initial patterns in a build_default_library() → FailureModeLibrary factory. Include at least 4 starting patterns: (1) CIRCULAR_DEPENDENCY — belief set contains A→B and B→A, (2) TOOL_UNAVAILABLE_CASCADE — multiple SYSTEM_ERROR evidences from different tools, (3) SCOPE_CREEP — task write_domains grew across iterations, (4) STALE_BELIEF_RELIANCE — high-confidence beliefs whose sources are in stale_flags. These seed the library for real use.
5
Wire as hypothesis generation source. In adapter/harness/hypothesis.py, add generate_from_failure_library(world_model, failure_mode_library) → list[Hypothesis]. Uses the matched pattern's hypothesis_template to produce a new Hypothesis with generation_sources=["failure_mode_library"]. This is source (3) of the four structurally distinct generators from P1.6. Returns empty list if no pattern matches.
6
Wire MatchResult.normalised_confidence into Tier 4 of resolve_control_state() in diagnostics.py. Pass the normalised_confidence value as one advisory input to the Tier 4 proportional caution elevation. Assert via a type-level check (isinstance guard or dataclass field inspection) that MatchResult fields are not present in block_mask computation — enforcing INV-08.
Acceptance tests
match() returns a MatchResult with normalised_confidence in [0,1]; injecting a raw confidence of 1.5 is clamped to 1.0
Library contribution to hypothesis generation is recorded in Hypothesis.generation_sources[] as "failure_mode_library" — the generated hypothesis is structurally distinct from symptom_inference output
MatchResult fields do not appear in block_mask computation inside resolve_control_state() — assert via test that block_mask is derived solely from diagnostic sub-dimension thresholds
P6.4 Local vs global replanning Week 3 · ~2 days
Implement the replan scope assessment and both replanning paths. Local replanning touches only the current task and its direct dependents. Global replanning rebuilds the entire task_graph from the world model and caller state, then validates the result. Contradiction scope drives the routing decision.
Implementation steps
1
Create adapter/harness/replanning.py. Define ReplanScope = Literal["LOCAL", "GLOBAL"]. Implement assess_replan_scope(contradiction, task_graph) → ReplanScope. Mapping: contradiction.scope == "global" → GLOBAL; "task" or "local" → LOCAL. Also return GLOBAL if the current task_graph has no tasks (empty graph always rebuilds).
2
Implement diagnose_and_replan(current_task, task_graph, world_model) → TaskGraph. LOCAL path: finds all tasks in task_graph with depends_on containing current_task.id, sets their status to PENDING (re-queued for re-evaluation), removes any FAILED entries downstream of current_task from the task list, and returns the modified graph. Does not touch tasks outside the dependency chain.
3
Implement rebuild_task_graph(world_model, caller_state) → TaskGraph. GLOBAL path: creates a fresh TaskGraph from scratch using caller_state.success_criteria as the decomposition input and world_model.beliefs[] as context. Uses the same decomposition logic as the initial planning step. Returns a newly constructed graph with all tasks in PENDING status.
4
Implement apply_replan(scope, contradiction, current_task, task_graph, world_model, caller_state) → TaskGraph. Routes to diagnose_and_replan() for LOCAL, or rebuild_task_graph() followed by validate_task_graph() for GLOBAL. validate_task_graph() is always called after GLOBAL replan — never skipped.
5
Wire into the main loop in loop.py. After cannot_make_progress() returns True or after a contradiction is detected with scope-driving info: call apply_replan() with the appropriate scope. Update HarnessRunState with the new task_graph. Increment world_model.generation_id after replan to invalidate any cached control_state.
Acceptance tests
GLOBAL scope rebuilds entire task_graph: the returned graph has all-PENDING tasks, none carry over from the old graph
LOCAL scope replans only current_task and its direct dependents: tasks that do not depend on current_task retain their original status
validate_task_graph() is always called after GLOBAL replan — an invalid graph (cycle, bad depends_on) raises immediately rather than silently producing a broken graph
P6.5 Context compression (compressed_structures + pruned_regions) Week 3 · ~2 days
Implement context compression with a structured distinction between fully dropped structures and partially truncated regions. Both lists feed into action_dep_overlap() (P5.7). completeness_flags{} tracks which world model belief regions are incomplete after pruning — "structure present" is not the same as "structure complete".
Implementation steps
1
Create adapter/harness/memory.py. Define CompressionRisk dataclass: compressed_structures: list[str] (names of fully removed structures), pruned_regions: list[str] (names of partially truncated structures). Define MemoryState dataclass: token_budget: int, max_steps: int, journal: list[dict] = field(default_factory=list), journal_retention_policy: dict = field(default_factory=dict), compression_risk: CompressionRisk = field(default_factory=CompressionRisk), rollback_points: list[str] = field(default_factory=list). Add serialisation helpers.
2
Implement should_compress(world_model, memory_state) → bool. Returns True when the estimated token count of world_model exceeds memory_state.token_budget * 0.9 (trigger at 90% of budget). Token count estimate: sum of character lengths of all observations, beliefs, and assumptions as a proxy.
3
Implement compress_memory(world_model, memory_state) → tuple[list[str], list[str]]. Compression policy: (1) Remove observations older than belief_ttl that have no beliefs derived from them → add to compressed_structures[]. (2) Truncate belief lists for regions with more than 10 beliefs, keeping the 5 most recently updated → add region name to pruned_regions[]. Returns (dropped_names, pruned_names). Mutates both world_model and memory_state.compression_risk in place.
4
Update world_model.completeness_flags{} inside compress_memory(). For each name in pruned_names: set world_model.completeness_flags[name] = False. For each name in dropped_names: set world_model.completeness_flags[name] = False as well (dropped = maximally incomplete). This ensures any code reading completeness_flags knows which regions are partial or absent.
5
Wire compression trigger in the main loop. At the top of each iteration in loop.py, after the staleness sweep: call should_compress(). If True: call compress_memory(), then re-increment world_model.generation_id to invalidate cached beliefs downstream. Log the compression event to memory_state.journal.
6
Verify action_dep_overlap() integration (P5.7). Confirm that action_dep_overlap(action, memory_state) correctly checks both memory_state.compression_risk.compressed_structures and memory_state.compression_risk.pruned_regions. Write a combined integration test: compress a region, then create an action that depends on it, assert overlap is detected.
Acceptance tests
compress_memory() correctly separates dropped observations (→ compressed_structures[]) from truncated belief lists (→ pruned_regions[]); the two lists are mutually exclusive
world_model.completeness_flags["beliefs_about_X"] is set to False after X is added to pruned_regions[] — a belief list that still exists but was truncated is correctly marked incomplete
action_dep_overlap() detects overlap with both compressed_structures and pruned_regions — a structure in pruned_regions (partially present) still triggers overlap detection
P6.6 Journal retention policy + max_steps Week 4 · ~2 days
Implement the journal retention policy that prevents unbounded growth of the execution journal, and the max_steps hard budget cap that warns at 80% and escalates at 100%. Both enforce predictable resource usage across arbitrarily long harness runs.
Implementation steps
1
Define the retention policy structure in memory.py. journal_retention_policy dict fields: retain_failures_permanently: bool = True (all failure entries kept), max_passing_verbatim: int = 20 (last N passing entries kept in full), compress_older_passing: bool = True (older passing entries compressed to summary tuple). Default values should be explicitly documented as tunable constants.
2
Implement apply_retention_policy(journal, policy) → list[dict]. Separate entries into failures and passing. Keep all failures verbatim. Keep the last policy.max_passing_verbatim passing entries verbatim. Compress older passing entries to {"action_class": ..., "outcome": "pass", "step": ...} summary dicts. Return the compacted journal. Called at the end of each step before persisting to HarnessRunState.
3
Implement check_max_steps(step_count, memory_state, diagnostics) → Literal["ok","warn","escalate"]. Returns "ok" if step_count < 0.8 * memory_state.max_steps. Returns "warn" if 0.8 * max_steps <= step_count < max_steps — also reduces diagnostics.verification_health.feasibility by 0.1 as a signal to reduce edit scope. Returns "escalate" if step_count >= max_steps.
4
Wire into the main loop in loop.py. At the start of each iteration: call check_max_steps(). On "warn": log to journal and continue with reduced feasibility. On "escalate": call escalate(surface_blocker(reason="budget_exhausted", missing_info=["step_count", "max_steps"])) and halt the loop. On "ok": continue normally.
5
Wire journal retention in loop.py. At the end of each step, before persisting HarnessRunState: call apply_retention_policy(memory_state.journal, memory_state.journal_retention_policy) and replace memory_state.journal with the result. This ensures the journal never exceeds all_failures + max_passing_verbatim + older_summaries in size.
Acceptance tests
After 30 passing steps and 5 failure steps, with max_passing_verbatim=10: journal contains all 5 failures verbatim + 10 passing verbatim + 20 compressed passing summaries — total entries bounded
At 80% of max_steps: check_max_steps() returns "warn" and diagnostics.verification_health.feasibility is reduced by 0.1
At max_steps: check_max_steps() returns "escalate"; the main loop calls escalate(surface_blocker(reason="budget_exhausted")) rather than raising an exception or silently stopping

Architecture invariants engaged in Phase 6
INV-08 — Failure mode library as one of four — not a controller P6.3 primary The failure_mode_library contributes matched_pattern.confidence (normalised) to Tier 4 of resolve_control_state() as one advisory input, and contributes as one of four hypothesis generation sources. It does not veto, block, or redirect execution. The MatchResult type is designed so that its fields cannot be inserted into block_mask or used to set escalation_reason — this is enforced by the type definition, not just documentation. The P6.3 test asserts this by verifying that block_mask is derived solely from diagnostic sub-dimension thresholds after a high-confidence pattern match.
INV-10 — experience_store is a no-op when absent P6.2 partial The adaptive strategy upgrade (P6.2 step 6) falls back transparently to the fixed STRATEGY_ORDER when experience_store.available=False. The get_strategy_with_experience() function presents an identical interface regardless of which path is taken — the caller never needs to check experience_store.available directly. This is the first phase where the experience_store guard pattern is established; it is extended in P8 when the full experience_store is implemented.
